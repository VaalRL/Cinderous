# 去中心化即時通訊軟體 (專案代號：Nostr Buddy) 技術規格書

## 1. 系統概述

本專案旨在開發一款具備極致隱私、無中央資料庫（Serverless），且高度還原早期 MSN Messenger 經典互動體驗（如震動、音樂狀態）的跨平台即時通訊軟體。系統透過融合 Nostr 協議與 WebRTC 技術，達成「零伺服器維護成本」的營運目標。

「極致隱私」在本專案有明確的工程定義，詳見〈6. 威脅模型〉。簡言之：**訊息內容與社交圖譜（誰跟誰聊）對中繼站皆不可見**；殘餘可觀測項目（如概略上線時間）被明確界定並採取緩解。

## 2. 核心架構設計 (雙軌混合網路)

系統將網路通訊拆分為「狀態與信令」及「巨量資料傳輸」雙引擎，並根據連線狀態動態切換：

* **引擎 A：Nostr 協議 (Cloudflare Workers + D1)**
* **定位：** 負責離線留言暫存、線上狀態廣播、以及 WebRTC 的初始信令交換。
* **資料生命週期 (NIP-40)：** 所有寫入 Cloudflare D1 的加密訊息皆押上 7 天的過期時間戳（Expiration Timestamp）。中繼站定時自動銷毀資料，確保永久免費額度不被耗盡。
* **無狀態廣播 (Ephemeral Events)：** 種類 20000-29999 的事件（線上狀態、正在聆聽音樂、WebRTC SDP 交換）僅做記憶體轉發，絕不寫入資料庫。**注意：Ephemeral 不寫 D1，但仍消耗 Worker 請求數與 CPU，並非「零消耗」；容量估算見〈8. 中繼站濫用防護與容量模型〉。**


* **引擎 B：WebRTC 直連 (點對點 P2P)**
* **定位：** 負責即時互動與大流量傳輸。
* **無縫升級：** 雙方上線並透過 Nostr 完成 SDP 交換後，即刻打通 P2P 資料通道。後續的「震動 (Nudge)」、「動畫快遞」與大檔案傳輸皆繞過中繼站，達成毫秒級延遲。
* **降級保底：** 對稱式 NAT 下 P2P 有 10%~15% 失敗率。為避免核心功能在這些情境完全失效，需提供 TURN 中繼與「經 Nostr 中繼的降級資料路徑」作為保底，詳見〈9. 可靠性與降級策略〉。



## 3. 終端平台與儲存策略

為確保端到端加密的完整性與避免資料遺失，徹底捨棄缺乏持久儲存權限的網頁版（Web）。

* **桌面端 (Rust Tauri + React/TypeScript)：** 作為系統的第一優先平台。擁有作業系統底層權限，可完美執行背景 WebSocket 長連線，並將對話紀錄與金鑰永久寫入本機原生 SQLite。
* **行動端 (React Native + SQLite)：** 作為輔助平台。受限於 iOS/Android 背景休眠機制，需透過 Cloudflare Worker 存放 APNs / FCM 憑證，以「無聲推播 (Silent Push)」喚醒 App，進行背景資料拉取與同步。**此推播路徑為刻意保留的中央依賴，且 Apple/Google 可觀測「何時喚醒誰」之元資料；推播內容僅含喚醒訊號，不含任何明文或對話元資料。**
* **資料庫架構：** 以純本機 SQLite 為「唯一真相來源 (Source of Truth)」。這為未來實作本地 AI 助理（直接讀取結構化 JSON 對話紀錄進行 RAG 摘要）提供了絕對隱私的封閉環境。
* **靜態落地加密 (At-Rest)：** 本機 SQLite 與私鑰**不得以明文落地**。私鑰存入作業系統安全儲存（macOS Keychain / Windows DPAPI / Linux libsecret / 行動端 Keystore/Secure Enclave）；對話資料庫以裝置綁定金鑰加密（如 SQLCipher），可選由使用者開機密語強化。此為對抗「設備竊取」威脅的必要條件。

## 4. 帳號、身分與多設備同步機制

* **零知識身分驗證：** 捨棄傳統帳號密碼。首次開啟軟體時，於本機端隨機生成 secp256k1 橢圓曲線公私鑰對（Nostr/NIP-01 標準，簽章採 BIP-340 Schnorr）。公鑰（`npub`）即為全網唯一 ID，顯示名稱允許無限重複（Zooko 三角難題之妥協方案）。
* **金鑰外洩的後果（已知取捨）：** 身分等同私鑰，**不提供金鑰輪替或撤銷**。私鑰外洩 = 永久可被冒名，且因採靜態 ECDH，外洩亦可解密歷史訊息（無前向保密）。此取捨與緩解見〈7. 隱私與加密設計〉與對應 ADR。
* **多設備同步 (QR Code + 連線競速)：**
* 手機掃描電腦端 QR Code 時，QR Code 僅包含「一次性 AES 金鑰」、「內網 IP (LAN)」與「WebRTC 房間號 (WAN)」。
* **通道安全：** 一次性金鑰須為 **AEAD（AES-256-GCM，每訊息唯一 nonce）**，傳輸通道與該金鑰綁定；兩設備以該金鑰互相認證（雙向 challenge-response），防止 LAN 內 MITM 或搶佔 WebRTC 房間者插入。金鑰用後即焚，房間號為高熵一次性值。
* 實作 **Happy Eyeballs 競速演算法**：手機端同時發起區網 TCP 直連與外網 WebRTC 打洞請求，優先連通者負責將數十 MB 的 SQLite 資料庫與私鑰整包加密傳輸至手機。
* **同步後的持續一致性：** 首次整包同步後，各設備並非彼此孤立。詳見〈10. 多設備持續同步〉。


* **極端災難復原：** **不提供** 12 字助記詞（BIP-39）或其他備份機制。貫徹極致隱私與零責任歸屬，設備與本機資料庫全毀即代表帳號永久死亡。

## 6. 威脅模型 (Threat Model)

「極致隱私」必須是可被驗證的工程界線，而非行銷語言。本節定義對手、保護標的與明確不保護的範圍。

* **對手 (Adversaries)：**
1. **中繼站營運者 / 託管方（Cloudflare）：** 假設可讀取所有經過的事件與其元資料、可被傳喚。
2. **被動網路監聽者：** 可觀測連線時間與流量大小。
3. **惡意聯絡人：** 已是你的好友，但試圖去匿名化或騷擾。
4. **設備竊取者：** 取得已解鎖或上鎖的實體裝置。

* **必須保護 (In-Scope)：**
- 訊息**內容**：對中繼站、監聽者一律密文（NIP-44）。
- **社交圖譜**：誰與誰通訊，對中繼站不可見（NIP-17/59 Gift Wrap）。
- **靜態資料**：私鑰與資料庫在裝置上加密落地（對抗設備竊取，見 §3）。

* **已知殘餘風險與緩解 (Accepted Residual Risks)：**
- **上線狀態時序：** 心跳天然會把「某 npub 此刻在線」暴露給中繼站。緩解：心跳僅在雙向同意聯絡人間訂閱、可加入隨機抖動（jitter）、可選「隱身上線」。
- **WebRTC 真實 IP 揭露：** P2P 會將 IP 暴露給對端聯絡人。緩解：預設提供 TURN 中繼選項以遮蔽 IP；對非信任聯絡人預設不直連。
- **無前向保密 / 後妥協安全：** 靜態 ECDH 下單次私鑰外洩可解全部歷史與未來訊息。此為與 Nostr 生態相容性的取捨；高敏感場景建議搭配 WebRTC 即時通道並考慮未來導入棘輪（ratchet）機制（見 ADR）。
- **行動端推播元資料：** APNs/FCM 可見喚醒時序（見 §3）。

* **明確不在保護範圍 (Out-of-Scope)：**
- 已遭入侵（rootkit/惡意軟體）的作業系統。
- 對端聯絡人自行轉貼/截圖你的訊息。
- 流量分析等級的全球被動對手（national-scale）。

## 7. 隱私與加密設計

* **內容加密：** 一律使用 **NIP-44（版本化 AEAD 加密）**，取代已棄用且不安全的 NIP-04（舊 kind 4）。
* **元資料隱藏：** 私訊採 **NIP-17 + NIP-59 Gift Wrap**：真實事件（kind 14）先封入 kind 13 seal，再以一次性金鑰封入 kind 1059 Gift Wrap，使中繼站只看到「一則指向某收件人臨時金鑰的密文」，無法還原寄件者與社交圖譜。
* **加密原語一致性：** 加解密、簽章與金鑰衍生集中於 `packages/core`，桌面/行動/Worker 不得各自重造，避免多重實作造成不一致或漏洞（SSOT）。選型（Web Crypto vs Rust `ed25519-dalek`/`ring`）以一份 ADR 定案。
* **金鑰落地：** 見 §3 靜態落地加密。
* **前向保密取捨：** 預設沿用 Nostr 靜態金鑰模型以維持生態相容；其侷限已於威脅模型載明，未來是否導入 Double Ratchet 類機制由專屬 ADR 評估。

## 8. 中繼站濫用防護與容量模型

開放中繼站若無防護，任何人都能灌爆事件以耗盡免費額度或塞滿 7 天留言視窗，直接威脅「永久免費」目標。

* **存取控制 (NIP-42)：** 訂閱與發布前要求 AUTH，避免匿名者大量拉取他人留言或洪泛寫入。
* **抗濫用：** 對「需持久化的事件（離線留言）」要求 **NIP-13 工作量證明（PoW）**；對每 pubkey 設發布速率上限與事件大小上限。
* **過期與配額：** 沿用 NIP-40 7 天過期；對單一收件人佇列設容量上限，超量丟棄最舊。
* **容量重估（取代「零消耗」說法）：** Cloudflare Workers 免費層約 10 萬請求/日。心跳（每 30 秒）與訂閱轉發會線性放大請求數，須以「平均在線好友數 × 心跳頻率 × 訂閱者數」估算並設計批次/合併策略，必要時拉長心跳間隔或改用單一彙整事件。此估算需在 M1 完成前定案並記錄於 ADR。

## 9. 可靠性與降級策略

* **P2P 失敗保底：** 震動與檔案傳輸不得「僅」在 WebRTC 可用時運作。連線建立失敗（對稱 NAT 等）時：
- 檔案傳輸與 Nudge 走 **TURN 中繼**維持可用；
- 若 TURN 亦不可用，Nudge 降級為「經 Nostr 中繼的延遲送達」（喪失毫秒級同步感但不丟失意圖）。
* **離線留言過期的 UX 對策：** 7 天硬過期會造成超時未取的訊息永久消失。需於產品層面：對寄件者顯示「未送達/已過期」狀態、對長期離線者在重新上線時提示可能的訊息缺口；不得讓使用者誤以為訊息必達。
* **重放與時鐘信任：** 對 Ephemeral 與心跳採用 `created_at` 有效窗口與一次性 nonce/事件 id 去重，避免重放偽造上線或重放舊 SDP；跨裝置排序不可僅信任 `created_at`，需搭配本機接收序與 Lamport/向量時鐘策略。

## 10. 聯絡人同意與封鎖

* **加好友需雙向同意：** 任何人僅憑 npub 即可寄送 Gift Wrap 留言，故需「聯絡人請求 → 核准」流程；未核准者的訊息進入隔離區或直接丟棄，避免騷擾與 spam 向量。
* **封鎖名單：** 本機維護封鎖清單，封鎖後不再解封其 Gift Wrap、不回應其 SDP、不廣播心跳予對方。

## 11. 多設備持續同步

首次 QR 整包同步只是起點；多設備若各自為政必然發散。

* **持續收斂：** 同一身分的多台設備需就「新訊息、已讀位置、聯絡人/封鎖清單變更」持續對帳。實作上可透過自己加密寄給自己的同步事件（NIP-17 自封）或 P2P 設備間直連，將增量變更廣播至本人其他設備。
* **衝突解法：** 訊息以事件 id 去重；可變狀態（已讀、暱稱、封鎖）採 last-writer-wins 或 CRDT 策略，於 ADR 定案。
* **邊界更新：** 多設備同步協定一旦定版，須同步更新 `ARCHITECTURE.md` 的事件契約。

## 12. 功能路線圖（借鏡 LINE，M6–M9）

在 MSN 經典體驗（M1–M5）之外，借鏡現代即時通的熱門功能，於後續里程碑納入。**一律沿用既有隱私機制**（NIP-44 加密 + NIP-17/59 Gift Wrap 隱藏元資料；即時媒體走 WebRTC P2P、不經中繼）。決策與範疇界線見 `docs/adr/0010`。

* **M6 訊息互動**
  * **訊息回應（Reaction）：** 對訊息按 emoji（NIP-25 kind 7），`e` tag 指向目標訊息，Gift Wrap 包封。
  * **收回訊息（Unsend）：** NIP-09 刪除（kind 5）指向目標訊息，收件端移除。
  * **限時訊息（Disappearing）：** 送訊即帶較短 NIP-40 過期（rumor 內層供收件端到期隱藏、外層 wrap 同步縮短以利中繼清除）；客戶端到期即顯示「訊息已到期」，與「閱後即焚」精神一致（見 ADR-0013）。
* **M7 富媒體**
  * **語音訊息／媒體相簿：** 錄音與圖片複用 WebRTC 檔案分塊傳輸，本機 SQLite 聚合為相簿。
  * **貼圖（Sticker）：** 內建貼圖包以 `pack/id` 參照、客戶端渲染；自訂貼圖走媒體傳輸。
* **M8 語音／視訊通話**
  * WebRTC media track；通話信令（offer/answer/candidate/hangup）複用既有 SDP 信令通道（NIP-59 包封的 ephemeral）。**媒體全程 P2P、不經中繼。**
* **M9 聯絡人與群組**
  * **QR 加好友：** 交換 `npub` + 雙向同意流程（延伸既有聯絡人同意機制，見 §10）。
  * **群組聊天：** 加密方案已定案（見 ADR-0027）：v1 採 **Gift-Wrap 成對扇出 + 帶內群組狀態**（每則群訊對每位成員各發一個 NIP-17/59 Gift Wrap，rumor 帶 `groupId`；隱私優先、複用既有機制、移除成員免 rekey），限私密好友小群；**MLS（NIP-EE）延後**為未來升級（需 PCS/更大群時，與 1:1 棘輪一併評估）。

### 明確排除（與本專案精神衝突）

* **LINE Pay／錢包／點數：** 非金融 App，超出範疇。
* **官方帳號／企業帳號／廣告／貼圖商店金流：** 中心化營利模式，違背去中心化與 AGPL 精神。
* **集中式雲端聊天備份：** 違反「明文不上雲、設備全毀即帳號死亡」的核心設計。
* **Open Chat 公開大群／VOOM 社群動態牆：** 會將產品從私密即時通拉成社群平台，與定位不符。

### 預期與事實的區分

* **預期：** 開發者與市場普遍預期，去中心化且跨平台的通訊軟體，理應能輕易在瀏覽器（網頁版）上流暢運行，且純 P2P 網路就能解決所有的傳輸需求。
* **事實：**
1. **瀏覽器儲存的脆弱性：** 現代瀏覽器（如 Chrome, Safari）會在設備空間不足時，靜默且強制地清空 IndexedDB。在沒有中央伺服器備份的架構下，網頁版會導致使用者面臨不可挽回的資料遺失。唯有編譯為二進位執行檔（Tauri/React Native）才能取得作業系統的持久化儲存權限。
2. **P2P 的物理限制：** 純 WebRTC P2P 絕對無法處理「離線留言」，且在對稱式 NAT 環境下有 10%~15% 的連線失敗率。必須引入 Cloudflare 作為 Nostr 中繼站來處理非同步狀態，這是維持通訊可靠性的工程事實。
3. **「密文上雲」不等於「隱私」：** 即使內容加密，Nostr 事件的寄件者與收件者欄位預設為明文，會把社交圖譜洩漏給中繼站。唯有導入 NIP-17/59 Gift Wrap 隱藏元資料，「極致隱私」才成立。

### 資料來源與推論過程

1. **RFC 8305: Happy Eyeballs Version 2**
* **來源網址：** [https://datatracker.ietf.org/doc/html/rfc8305](https://datatracker.ietf.org/doc/html/rfc8305)
* **推論過程：** IETF 制定的雙軌連線競速標準，客觀支撐了我們在「QR Code 設備同步」時，同時發起 LAN（內網直連）與 WAN（WebRTC）請求並中斷較慢者的設計，確保了在複雜網路環境下的最高同步成功率。


2. **Nostr 官方 GitHub：NIP-40 (Expiration) 與 NIP-01 (Ephemeral)**
* **來源網址：** [https://github.com/nostr-protocol/nips/blob/master/40.md](https://github.com/nostr-protocol/nips/blob/master/40.md) , [https://github.com/nostr-protocol/nips/blob/master/01.md](https://github.com/nostr-protocol/nips/blob/master/01.md)
* **推論過程：** 協議明確賦予了事件自動銷毀與「純廣播不儲存」的屬性。這證實了利用 Cloudflare D1 作為中繼站，能在不突破免費層級限制的前提下，完美處理離線留言與線上狀態廣播的技術可行性。


3. **Nostr 官方 GitHub：NIP-44 (加密)、NIP-17 (私訊)、NIP-59 (Gift Wrap)、NIP-42 (AUTH)、NIP-13 (PoW)**
* **來源網址：** [44](https://github.com/nostr-protocol/nips/blob/master/44.md) , [17](https://github.com/nostr-protocol/nips/blob/master/17.md) , [59](https://github.com/nostr-protocol/nips/blob/master/59.md) , [42](https://github.com/nostr-protocol/nips/blob/master/42.md) , [13](https://github.com/nostr-protocol/nips/blob/master/13.md)
* **推論過程：** NIP-04（舊 kind 4）已被棄用且洩漏訊息長度，故內容加密改用 NIP-44；為避免社交圖譜洩漏，私訊以 NIP-17/59 Gift Wrap 隱藏收發雙方；NIP-42/13 提供中繼站存取控制與抗濫用，支撐「永久免費且不被灌爆」的營運前提。


4. **W3C Storage API 規範**
* **來源網址：** [https://developer.mozilla.org/en-US/docs/Web/API/Storage_API/Storage_quotas_and_eviction_criteria](https://www.google.com/search?q=https%3A%2F%2Fdeveloper.mozilla.org%2Fen-US%2Fdocs%2FWeb%2FAPI%2FStorage_API%2FStorage_quotas_and_eviction_criteria)
* **推論過程：** 規範指出瀏覽器具備主動驅逐（Eviction）本地資料的權限。這提供了我們「徹底捨棄網頁版開發」這項決策背後的絕對事實依據。
