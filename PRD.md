# 去中心化即時通訊軟體 (專案代號：Cinder) 技術規格書

## 1. 系統概述

本專案旨在開發一款具備極致隱私、無中央資料庫（Serverless），且高度還原早期 MSN Messenger 經典互動體驗（如震動、音樂狀態）的跨平台即時通訊軟體。系統透過融合 Nostr 協議與 WebRTC 技術，達成「零伺服器維護成本」的營運目標。

「極致隱私」在本專案有明確的工程定義，詳見〈6. 威脅模型〉。簡言之：**訊息內容與社交圖譜（誰跟誰聊）對中繼站皆不可見**；殘餘可觀測項目（如概略上線時間）被明確界定並採取緩解。

## 2. 核心架構設計 (雙軌混合網路)

系統將網路通訊拆分為「狀態與信令」及「巨量資料傳輸」雙引擎，並根據連線狀態動態切換：

* **引擎 A：Nostr 協議 (Cloudflare Workers + Durable Object SQLite)**
* **定位：** 負責離線留言暫存、線上狀態廣播、以及 WebRTC 的初始信令交換。
* **持久層：** 早期規劃 D1，實作改用 **Durable Object 內建 SQLite ＋ WebSocket Hibernation**
  （ADR-0056）——同步、免額外綁定、跨休眠存活，且每則進站 WS 訊息才計一次 DO 請求。
* **資料生命週期 (NIP-40)：** 所有寫入持久層的加密訊息皆押上 7 天過期時間戳。中繼站定時自動
  銷毀資料，確保永久免費額度不被耗盡。
* **無狀態廣播 (Ephemeral Events)：** 種類 20000-29999 的事件（線上狀態、正在聆聽音樂、
  WebRTC SDP 交換）僅做記憶體轉發，絕不寫入持久層。**注意：Ephemeral 不寫 DB，但仍消耗
  Worker/DO 請求數與 CPU，並非「零消耗」；容量估算見〈8〉。**


* **引擎 B：WebRTC 直連 (點對點 P2P)**
* **定位：** 負責即時互動與大流量傳輸。
* **無縫升級：** 雙方上線並透過 Nostr 完成 SDP 交換後，即刻打通 P2P 資料通道。後續的「震動 (Nudge)」、「動畫快遞」與大檔案傳輸皆繞過中繼站，達成毫秒級延遲。
* **降級保底：** 對稱式 NAT 下 P2P 有 10%~15% 失敗率。為避免核心功能在這些情境完全失效，需提供 TURN 中繼與「經 Nostr 中繼的降級資料路徑」作為保底，詳見〈9. 可靠性與降級策略〉。



## 3. 終端平台與儲存策略

> **設計反轉（重要）：** 早期規格「徹底捨棄網頁版」。實作階段改為**三平台並存**——桌面（Tauri）、
> 網頁（瀏覽器）、行動（react-native-web）。網頁版的資料遺失風險（瀏覽器可驅逐儲存）是真的，
> 但改以「必填本地密碼＋明確接受的殘餘風險」緩解，而非整個放棄（ADR-0104 本地啟動器、
> ADR-0112 web 靜態加密、ADR-0122 身分不被靜默替換）。

* **桌面端 (Rust Tauri + React/TypeScript)：** 系統的第一優先平台。私鑰存入 **OS 安全儲存**
  （keyring：macOS Keychain / Windows Credential Manager / Linux Secret Service，ADR-0053）；
  對話資料以 **AES-256-GCM 加密 blob 逐部位落地**（非 SQLite；ADR-0054／0110）。「關窗仍在線」
  以**系統匣隱藏**達成（webview 續存）——原本規劃的 Rust 背景連線模組已於 ADR-0105 退役
  （引擎早已改為多中繼連線池，該模組的單一連線假設過時）。
* **網頁端 (瀏覽器)：** 與桌面共用同一份 React/引擎程式碼，差在**沒有 OS 金鑰庫**。因此
  nsec 以 **Argon2id 本地密碼包裹**後落 localStorage（磁碟上只有密文，KEK 從不落盤；
  ADR-0112／0122），對話資料以 nsec 導出的金鑰加密，封存走 **OPFS 檔案**（ADR-0111）。
  已知殘餘風險：瀏覽器可能驅逐儲存——緩解為必填本地密碼＋可複製的 nsec 備份（見 §7）。
* **行動端 (react-native-web)：** 接真實中繼（ADR-0086），儲存走 localStorage＋OPFS，
  「記住我」以 Argon2id 包裹 nsec（ADR-0117）。**不做推播（APNs/FCM）**（ADR-0116）：
  接受 iOS/Android 的背景休眠限制，而非引入「無聲推播」這個中央依賴與「何時喚醒誰」的元資料洩漏。
* **唯一真相來源 (SoT)：** **本機的加密儲存**（狀態常駐記憶體＋逐部位加密持久化，ADR-0110）。
  這為未來本地 AI 助理（讀結構化對話紀錄做 RAG 摘要）提供絕對隱私的封閉環境。
* **靜態落地加密 (At-Rest)：** 私鑰與對話資料**不得以明文落地**。桌面以 OS 金鑰庫＋AES blob；
  web/mobile 以 Argon2id 包裹的 nsec＋nsec 導出的資料金鑰。共用設備可再啟用**本地密碼**
  （ADR-0067）：Argon2id 衍生金鑰包裹 nsec 與資料金鑰、支援隱藏身分與閒置自動上鎖。

## 4. 帳號、身分與多設備同步機制

* **零知識身分驗證：** 捨棄傳統帳號密碼。首次開啟軟體時，於本機端隨機生成 secp256k1 橢圓曲線公私鑰對（Nostr/NIP-01 標準，簽章採 BIP-340 Schnorr）。公鑰（`npub`）即為全網唯一 ID，顯示名稱允許無限重複（Zooko 三角難題之妥協方案）。
* **金鑰外洩的後果（已知取捨）：** 身分等同私鑰，**不提供金鑰輪替或撤銷**。私鑰外洩 = 永久可被冒名，且因採靜態 ECDH，外洩亦可解密歷史訊息（無前向保密）。此取捨與緩解見〈7. 隱私與加密設計〉與對應 ADR。
* **多設備同步 / 換機搬家（配對克隆）：**
* 實作採 **WebRTC 配對傳輸**（ADR-0072／0118）：舊機產生一次性載荷（帶會合 relay ＋ 一次性
  AES-256-GCM 金鑰），新機貼上後雙方經 WebRTC 打通，把全量捆包（身分＋聯絡人＋歷史）
  端到端加密傳輸，**內容不經中繼**。（早期規劃的「掃 QR＋LAN 內網 IP＋Happy Eyeballs 競速」
  未實作；實際只用 WebRTC transport。）
* **通道安全：** 雙方顯示 **SAS 短驗證碼**，使用者**比對相符才確認送出**——沒有它，中間人可冒充
  新機把整包資料（含 nsec）騙走。金鑰用後即焚。
* **同步後的持續一致性：** 首次整包搬家後，各設備並非彼此孤立。詳見〈11. 多設備持續同步〉。


* **極端災難復原（設計反轉）：** 早期規格「不提供任何備份機制」。實作階段補上了**多條 opt-in
  的換機／備份路徑**（仍不採 BIP-39 助記詞）：可複製的 nsec 備份、加密備份碼（NIP-49 使用者自持，
  ADR-0070）、加密雲端快照（relay 只見密文，ADR-0071）、桌面配對克隆（一次性 P2P 全量搬家＋
  SAS 互認，ADR-0072／0118）。這些都由使用者主動啟用；若使用者未做任何備份，設備全毀仍代表
  帳號永久死亡（靜態 ECDH，無第二把金鑰可救）。

## 6. 威脅模型 (Threat Model)

「極致隱私」必須是可被驗證的工程界線，而非行銷語言。本節定義對手、保護標的與明確不保護的範圍。

* **對手 (Adversaries)：**
1. **中繼站營運者 / 託管方（Cloudflare）：** 假設可讀取所有經過的事件與其元資料、可被傳喚。
2. **被動網路監聽者：** 可觀測連線時間與流量大小。
3. **惡意聯絡人：** 已是你的好友，但試圖去匿名化或騷擾。
4. **設備竊取者：** 取得已解鎖或上鎖的實體裝置。

* **必須保護 (In-Scope)：**
- 訊息**內容**：對中繼站、監聽者一律密文（NIP-44）。
- **訊息層社交圖譜**：**誰傳訊息給誰**，對中繼站不可見（NIP-17/59 Gift Wrap——寄件者為一次性臨時金鑰、只有收件人 pubkey 明文）。**注意：這是訊息層**；在線狀態的訂閱過濾器仍會洩漏你的**聯絡人集合**（見殘餘風險），兩者是不同的洩漏面。
- **靜態資料**：私鑰與資料庫在裝置上加密落地（對抗設備竊取，見 §3）。

* **已知殘餘風險與緩解 (Accepted Residual Risks)：**
- **上線時序（who's online when）：** 心跳以真名簽章廣播，天然把「某 npub 此刻在線」的**時序**暴露給中繼站——這是廣播式 presence 的固有代價，藏不掉。緩解：僅在雙向同意聯絡人間訂閱、隨機抖動（jitter）、可選「隱身上線」、與聯絡人 P2P 直連時**抑制中繼心跳**（ADR-0088）。
- **🔴 在線訂閱洩漏聯絡人清單：** 要收聯絡人的在線狀態，客戶端須訂閱 `{kinds:[20000], authors:[聯絡人清單]}`——這個 REQ 等於把你的**聯絡人集合**交給中繼站。訊息層的社交圖譜由 Gift Wrap 藏住（見 In-Scope），但**在線訂閱仍洩漏你的聯絡人是誰**。徹底修需把在線也封裝定址（發佈數乘以聯絡人數，撞免費層容量），暫緩；詳見 `docs/relay-metadata-observability.md`（M2/M7）。
- **在線狀態的「內容」已不再洩漏（記錄改善）：** 狀態文字（「我在發呆」）、正在聽的音樂、正在輸入中、敲一下——這些**內容**曾以明文廣播給中繼站，現已改為 **NIP-59 封裝、只在改變/對方上線時發**（ADR-0120 typing/nudge、ADR-0129 狀態文字/音樂），中繼站**看不到內容**（只剩上述時序）。
- **WebRTC 真實 IP 揭露：** P2P 會將 IP 暴露給對端聯絡人。緩解：預設提供 TURN 中繼選項以遮蔽 IP；對非信任聯絡人預設不直連。
- **無前向保密 / 後妥協安全：** 靜態 ECDH 下單次私鑰外洩可解全部歷史與未來訊息。此為與 Nostr 生態相容性的取捨；高敏感場景建議搭配 WebRTC 即時通道並考慮未來導入棘輪（ratchet）機制（見 ADR）。
- **~~行動端推播元資料~~（已消除）：** 早期規劃以 APNs/FCM 無聲推播喚醒行動端，會洩漏「何時喚醒誰」。**ADR-0116 決定不做推播**——行動端接受背景休眠限制，故無此洩漏。

* **明確不在保護範圍 (Out-of-Scope)：**
- 已遭入侵（rootkit/惡意軟體）的作業系統。
- 對端聯絡人自行轉貼/截圖你的訊息。
- 流量分析等級的全球被動對手（national-scale）。

## 7. 隱私與加密設計

* **內容加密：** 一律使用 **NIP-44（版本化 AEAD 加密）**，取代已棄用且不安全的 NIP-04（舊 kind 4）。
* **元資料隱藏：** 私訊採 **NIP-17 + NIP-59 Gift Wrap**：真實事件（kind 14）先封入 kind 13 seal，再以一次性金鑰封入 kind 1059 Gift Wrap，使中繼站只看到「一則指向某收件人臨時金鑰的密文」，無法還原寄件者與社交圖譜。
* **加密原語一致性：** 加解密、簽章與金鑰衍生集中於 `packages/core`，桌面/行動/Worker 不得各自重造，避免多重實作造成不一致或漏洞（SSOT）。選型（Web Crypto vs Rust `ed25519-dalek`/`ring`）以一份 ADR 定案。
* **金鑰落地：** 見 §3 靜態落地加密。
* **前向保密取捨：** 預設沿用 Nostr 靜態金鑰模型以維持生態相容；其侷限已於威脅模型載明，未來是否導入 Double Ratchet 類機制由專屬 ADR 評估。

## 8. 中繼站濫用防護與容量模型

開放中繼站若無防護，任何人都能灌爆事件以耗盡免費額度或塞滿 7 天留言視窗，直接威脅「永久免費」目標。

* **存取控制 (NIP-42)：** 訂閱與發布前要求 AUTH（ADR-0057，已啟用）；且**訂閱必須具名**——
  帶 `#p`（等於自己）或帶 `authors`，擋掉「一鍵收割全站線上名冊」的無 scope filter（ADR-0123）。
* **抗濫用：** 中繼核心支援對持久化事件要求 **NIP-13 PoW**（`minPow`）與每 pubkey 速率／大小上限；
  **生產目前未強制 PoW**（`worker.ts` 未設 `minPow`），保留為濫用升溫時的旋鈕。訂閱數上限已啟用
  （`maxSubscriptions`，ADR-0119）。
* **過期與配額：** 沿用 NIP-40 7 天過期；對單一收件人佇列設容量上限，超量丟棄最舊。
* **容量重估（取代「零消耗」說法）：** Cloudflare 免費層約 10 萬請求/日。心跳與訂閱轉發會線性
  放大請求數。**已定案並實作**（ADR-0109）：自適應心跳（活躍 60s／閒置 300s）＋合併 REQ ＋
  增量收件箱，把心跳從請求量的大宗壓下來。DO 的**外送事件不計請求**，僅進站 WS 訊息計費。

## 9. 可靠性與降級策略

* **P2P 失敗保底：** 震動與檔案傳輸不得「僅」在 WebRTC 可用時運作。連線建立失敗（對稱 NAT 等）時：
- 檔案傳輸與 Nudge 走 **TURN 中繼**維持可用；
- 若 TURN 亦不可用，Nudge 降級為「經 Nostr 中繼的延遲送達」（喪失毫秒級同步感但不丟失意圖）。
* **離線留言過期的 UX 對策：** 7 天硬過期會造成超時未取的訊息永久消失。需於產品層面：對寄件者顯示「未送達/已過期」狀態、對長期離線者在重新上線時提示可能的訊息缺口；不得讓使用者誤以為訊息必達。
* **重放與時鐘信任：** 對 Ephemeral 與心跳採用 `created_at` 有效窗口與一次性 nonce/事件 id 去重，避免重放偽造上線或重放舊 SDP；跨裝置排序不可僅信任 `created_at`，需搭配本機接收序與 Lamport/向量時鐘策略。

## 10. 聯絡人同意與封鎖

* **訊息請求（已實作，ADR-0121）：** 任何人僅憑 npub 即可寄送 Gift Wrap 留言。陌生人的訊息進入
  **訊息請求區**，**不是**聯絡人清單——在你接受之前，他碰不到任何注意力管道：不跳通知、不點亮
  未讀徽章、不能敲你（nudge）、不顯示 typing、看不到你的上線狀態、拿不到你的個人檔、收不到
  已讀回條。你「接受」或「主動回覆」才成為聯絡人。（早期規格把這寫成規劃中的「隔離區」；
  已落地。**已知限制：** 訊息本身仍會抵達中繼〔Nostr 擋不掉〕，這是客戶端的顯示層防禦；
  請求區的防洪〔數量上限／速率／全部刪除〕尚未做。）
* **封鎖名單：** 本機維護封鎖清單，封鎖後不再解封其 Gift Wrap、不回應其 SDP、不廣播心跳予對方；
  封鎖一個請求者會連同該請求一起清掉。

## 11. 多設備持續同步

首次 QR 整包同步只是起點；多設備若各自為政必然發散。

* **持續收斂：** 同一身分的多台設備需就「新訊息、已讀位置、聯絡人/封鎖清單變更」持續對帳。實作上可透過自己加密寄給自己的同步事件（NIP-17 自封）或 P2P 設備間直連，將增量變更廣播至本人其他設備。
* **衝突解法：** 訊息以事件 id 去重；可變狀態（已讀、暱稱、封鎖）採 last-writer-wins 或 CRDT 策略，於 ADR 定案。
* **邊界更新：** 多設備同步協定一旦定版，須同步更新 `ARCHITECTURE.md` 的事件契約。

## 12. 功能路線圖（借鏡 LINE，M6–M9）

在 MSN 經典體驗（M1–M5）之外，借鏡現代即時通的熱門功能，於後續里程碑納入。**一律沿用既有隱私機制**（NIP-44 加密 + NIP-17/59 Gift Wrap 隱藏元資料；即時媒體走 WebRTC P2P、不經中繼）。決策與範疇界線見 `docs/adr/0010`。

* **M6 訊息互動**
  * **訊息回應（Reaction）：** 對訊息按 emoji（NIP-25 kind 7），`e` tag 指向目標訊息，Gift Wrap 包封。
  * **收回訊息（Unsend）：** NIP-09 刪除（kind 5）指向目標訊息，收件端移除。
  * **限時訊息（Disappearing）：** 送訊即帶較短 NIP-40 過期（rumor 內層供收件端到期隱藏、外層 wrap 同步縮短以利中繼清除）；客戶端到期即顯示「訊息已到期」，與「閱後即焚」精神一致（見 ADR-0013）。
* **M7 富媒體**
  * **語音訊息／媒體相簿：** 錄音與圖片複用 WebRTC 檔案分塊傳輸，本機儲存聚合為相簿。
  * **貼圖（Sticker）：** 內建貼圖包以 `pack/id` 參照、客戶端渲染；自訂貼圖走媒體傳輸。
* **M8 語音／視訊通話**
  * WebRTC media track；通話信令（offer/answer/candidate/hangup）複用既有 SDP 信令通道（NIP-59 包封的 ephemeral）。**媒體全程 P2P、不經中繼。**
* **M9 聯絡人與群組**
  * **QR 加好友：** 交換 `npub` + 雙向同意流程（延伸既有聯絡人同意機制，見 §10）。
  * **群組聊天：** 加密方案已定案（見 ADR-0027）：v1 採 **Gift-Wrap 成對扇出 + 帶內群組狀態**（每則群訊對每位成員各發一個 NIP-17/59 Gift Wrap，rumor 帶 `groupId`；隱私優先、複用既有機制、移除成員免 rekey），限私密好友小群；**MLS（NIP-EE）延後**為未來升級（需 PCS/更大群時，與 1:1 棘輪一併評估）。

### 明確排除（與本專案精神衝突）

* **LINE Pay／錢包／點數：** 非金融 App，超出範疇。
* **官方帳號／企業帳號／廣告／貼圖商店金流：** 中心化營利模式，違背去中心化與 AGPL 精神。
* **集中式雲端聊天備份：** 違反「明文不上雲、設備全毀即帳號死亡」的核心設計。
* **Open Chat 公開大群／VOOM 社群動態牆：** 會將產品從私密即時通拉成社群平台，與定位不符。

> **界線澄清（捐款導流 ≠ 上述排除）：** 上列「廣告」與「金流」排除的是**產品自身的中心化營利與第三方廣告聯播**。**自願、去中心化的捐款導流**——relay 營運者自報的贊助入口（`docs/adr/0089`）與官方捐款/資金透明度（`docs/adr/0090`）——採**純外部連結、無站內托管/錢包/抽成、無第三方廣告、行動端不放**，用於支撐社群 relay 與官方節點的永續，**不屬本清單排除項**。

### 預期與事實的區分

* **預期：** 開發者與市場普遍預期，去中心化且跨平台的通訊軟體，理應能輕易在瀏覽器（網頁版）上流暢運行，且純 P2P 網路就能解決所有的傳輸需求。
* **事實：**
1. **瀏覽器儲存的脆弱性（後續已緩解，非放棄）：** 現代瀏覽器會在設備空間不足時驅逐儲存。
   早期據此「徹底捨棄網頁版」；實作階段改為**支援網頁版但誠實標示殘餘風險**——nsec 以必填的
   Argon2id 本地密碼包裹、資料以 nsec 導出金鑰加密、身分絕不被靜默替換（拿不到金鑰時大聲失敗，
   ADR-0122），並提供 nsec 備份作為救回途徑。桌面（Tauri）仍是第一優先平台，擁有 OS 持久化
   與金鑰庫；網頁版是「可用但風險自負」的第二選擇。
2. **P2P 的物理限制：** 純 WebRTC P2P 絕對無法處理「離線留言」，且在對稱式 NAT 環境下有
   10%~15% 的連線失敗率。必須引入 Cloudflare 作為 Nostr 中繼站來處理非同步狀態，這是維持
   通訊可靠性的工程事實。
3. **「密文上雲」不等於「隱私」：** 即使內容加密，Nostr 事件的寄件者與收件者欄位預設為明文，會把社交圖譜洩漏給中繼站。唯有導入 NIP-17/59 Gift Wrap 隱藏元資料，「極致隱私」才成立。

## 13. 企業模式（自架封閉節點 + 多身分並存）

> 面向企業內部通訊的封閉部署型態。**相容並不取代**既有的開放模式；核心隱私鐵則（明文不上雲、E2E、中繼零狀態）完全維持。設計決策見 ADR-0044（封閉 allowlist 中繼 + 自架）、ADR-0045（多身分）、ADR-0046（成員判定與對外通訊邊界）。

### 13.1 目標與範疇

* **同一企業只用同一節點：** 企業自架一座中繼，內部成員的通訊只經此節點；**刻意不聯邦化**（跨企業互通與「單節點」互斥，不在範疇）。
* **可自架於本地：** 不限 Cloudflare Worker，可跑在企業內網伺服器或單機。
* **外部客戶不進系統：** 系統僅服務內部成員；企業的外部客戶並非使用者。
* **明確排除：** 法遵歸檔／eDiscovery／DLP／內容監督——這些**需要伺服器讀明文**，與 E2E 根本衝突，不在本模式；若客戶要求須另立獨立「受監督版」ADR（涉及金鑰託管／端點擷取的隱私取捨）。

### 13.2 成員判定（allowlist）

* **「內部成員」的唯一判準：** 該成員的 hex pubkey 是否在企業中繼的**發布 allowlist**（`allowedAuthors`）內。非成員的任何事件（含上線心跳）於驗簽後一律拒收（`blocked:`，永久性）。→ 外部客戶天然無法進入系統。
* **佈建（企業身分 → npub）：** 由管理者把成員 pubkey 加入名單；「公司身分（HR／AD／SSO）對應到 npub」發生在此佈建步驟。**系統無中央目錄**，中繼只認得「是否為名單成員」。
* **離職：** 把 npub 移出名單即時生效（下次不放行），符合既有「移除即不放行」語意。

### 13.3 自架部署

* **與 Cloudflare 解耦：** 中繼核心 `RelayCore` 為傳輸無關（`worker.ts` 僅注入 Cloudflare WS 外殼；測試/開發已在 Worker 外執行）。自架＝以 Node／Deno／Bun／Docker 的 WebSocket server 包核心，跑內網或單機。
* **資料落地與保留：** 自架後保留期、每收件人上限、速率／PoW 全由企業內控。
* **讀取層把關：** allowlist 只擋「發布」；封閉節點的「不被外部連上」由企業**自架於私網／VPN／防火牆後**達成（無 NIP-42 亦足夠；且取到的皆 E2E 密文）。

### 13.4 多身分並存與切換

* **同一 App 內工作＋個人身分並存、一鍵切換**（ADR-0045）。工作身分連公司座、受 allowlist、**鎖定單座**（關閉 relay pool／錨點／漫遊／home 遞補，不得漫遊出公司節點）；個人身分走開放模式。
* **資料硬隔離：** 每身分的聯絡人／訊息／群組／貼圖庫等以 pubkey 命名空間（`nb.<pubkey>.*`）隔離，互不交雜（合規上乾淨）。既有單一身分維持舊鍵、首次載入自動遷移為 legacy 設定檔，向後相容。
* **一次一個作用中身分：** 切換以重載乾淨重建狀態；同時在線為後續可選項。

### 13.5 對外通訊邊界（重要界線）

* **使用者永遠保有以「個人身分」對外的能力：** 金鑰自主、軟體開源（AGPL），一個人可持多把金鑰；企業 allowlist 只約束「工作身分」在「公司節點」，管不到個人身分在公開 relay 的活動。
* **企業能控制的是「身分＋節點」，不是「人」：** 企業可鎖定自家 App 建置（工作身分只連公司座）、可把工作 npub 移出名單，但**擋不住員工另用個人金鑰＋一般版 App 對外**。
* **真正禁止對外只能靠系統外的裝置管理（MDM／端點）：** 此為裝置治理，不在本系統；文件明確告知企業此界線。

### 13.6 換裝置/遺失還原（身分輪替，非金鑰託管）

* **不採金鑰託管（無後門）：** 企業「換裝置/遺失還原」需求**不以託管私鑰達成**——託管靜態金鑰＝公司持有可回溯、永久、全解密的後門（因無前向保密），等同 §13.1 排除的「受監督版」能力。決策見 ADR-0052。
* **改採工作身分輪替：** 換裝置/遺失時，員工於新設備**自產**新工作身分金鑰（私鑰永不離開員工端）；管理者以簽章名冊把舊 npub 標 `supersededBy`、加入新 npub（allowlist 同步排除舊金鑰）。成員端自動 remap——把本機聯絡人/群成員與 1:1 歷史從舊 npub 接續到新 npub，並提示「◯◯ 已更新金鑰」。
* **與 §4 一致：** 不違反「不提供金鑰輪替/撤銷」——§4 指開放模式協定層；企業輪替是**公司範圍**的名冊/allowlist 動作，非全網撤銷。個人身分不受影響。
* **「不想丟歷史」＝多設備冗餘，非託管：** 建議員工登記 **≥2 台設備**（如桌機＋手機，複用 M4 QR 配對與持續同步）。掉單台→另一台仍有金鑰與完整歷史、免輪替；僅全部設備皆毀才走輪替（歷史依「設備死亡」承諾消失）。
* **邊界：** 全設備皆毀時舊歷史不可復原（刻意）；輪替接續依賴組織通訊錄（以名冊/組織群認得的同事自動換人）。

### 13.7 實作現況與後續

* **已完成：** 中繼發布 allowlist（`RelayCore.allowedAuthors`）；客戶端多身分設定檔、命名空間隔離、切換器與工作身分鎖定建置；成員名冊佈建工具與企業通訊錄（G1）、集中政策＋強制 TURN 接入 WebRTC（G2）、組織群組/公告（G3）、**工作身分輪替（G4，ADR-0052）**。
* **後續：** 自架 WS 外殼與部署文件；G5（SSO 綁定 / 連線元資料稽核）；輪替提示 i18n、Rust 原生 store 平價；多身分切換列同時在線（可選）。
