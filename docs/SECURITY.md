# 安全政策與自評（Security Policy & Self-Assessment）

> 本文件為 **Phase F4** 的內部安全自評與漏洞回報政策，為未來第三方稽核鋪路。
> 威脅模型的權威定義見 `PRD.md §6`；加密設計見 `PRD.md §7`、`ARCHITECTURE.md §5`；
> 各項決策理由見 `docs/adr/`。

## ⚠️ 稽核狀態

**本專案的端到端加密與金鑰處理尚未經第三方獨立稽核。** 請勿在生命安全等級的高風險
情境倚賴本軟體。歡迎安全研究者檢視（AGPL-3.0，原始碼公開）。

## 回報漏洞

- 請**勿**公開開 issue 揭露安全漏洞。
- 請透過 GitHub **Security Advisories**（私密回報）通報，或聯繫維護者的私密管道。
- 請附：影響範圍、重現步驟、受影響版本/commit，以及（若有）建議修補方向。
- 我們會盡力在合理時間內確認、修補，並於修補後協調揭露。

## 加密原語盤點（SSOT：`packages/core`）

| 面向 | 採用 | 備註 |
| --- | --- | --- |
| 身分/簽章 | secp256k1 + BIP-340 Schnorr（`@noble/curves`） | NIP-01；ADR-0004 |
| 內容加密 | NIP-44 v2（版本化 AEAD，經 `nostr-tools`） | 取代不安全的 NIP-04；ADR-0007 |
| 元資料隱藏 | NIP-17 + NIP-59 Gift Wrap（kind 14→13→1059） | 對**只看事件內容/靜態資料**的中繼與其他客戶端：隱藏**寄件人**（一次性金鑰）、收件人 pubkey 明文供路由。⚠️ 對**主動關聯「連線認證身分↔收發」**的中繼，寄件人（發布連線=真名）與聯絡人集合（presence 訂閱）皆可被歸屬（ADR-0237）；ADR-0002 |
| 即時通道 | WebRTC DTLS（P2P，具前向保密） | 檔案/語音/通話媒體不經中繼 |
| 靜態落地 | SQLCipher（`sqlcipher` feature） | 桌面原生 DB 加密；ADR-0020 |
| 隨機性 | `@noble/hashes` `randomBytes` / WebCrypto | groupId、一次性金鑰、nonce |

## 威脅模型盤點（對照 PRD §6：對手 → 已實作緩解 → 殘餘風險）

### 中繼站營運者 / 可傳喚的託管方
- **內容**：一律 NIP-44 密文；中繼只見密文。✅
- **社交圖譜（訊息層）**：私訊、回應、收回、限時、**群組**皆走 NIP-59 Gift Wrap（外層作者為
  一次性金鑰），**對只看事件的中繼**無法從事件本身還原「誰**傳訊息**給誰」。✅（群組亦為成對扇出，見 ADR-0027）
- **🔴 殘餘——連線歸屬打穿事件層寄件人隱藏（ADR-0237）**：反爬蟲的 NIP-42 AUTH 讓每條連線都綁真名，
  而發布走同一條已認證連線——一個把「認證身分↔發布」關聯的中繼（零成本，本就握有 authState）看得到
  「真名 A 的連線發了一則指向收件人 #p 的 Gift Wrap」＝**送出邊**，即使外層作者是臨時金鑰。故事件層
  寄件人隱藏只對「不主動關聯連線」的中繼成立。緩解：自架（對手＝自己）；根治需傳輸匿名，見 ADR-0237。
- **🔴 殘餘——在線訂閱洩漏聯絡人集合**：要收聯絡人在線狀態，客戶端須以真名 AUTH 後訂閱
  `{kinds:[20000], authors:[聯絡人清單]}`——這個 REQ **把你的聯絡人集合直接交給 home relay**。
  訊息層社交圖譜由 Gift Wrap 藏住，但**在線層沒有**。緩解：P2P presence 卸載（ADR-0088e）、
  自架（洩漏對象＝自己的節點）；徹底修需把在線也封裝定址（發佈數 × 聯絡人數，撞免費層容量），
  暫緩，見 PRD §6 與 `docs/relay-metadata-observability.md` M7。
- **殘餘——時序/大小**：上線**時序**（心跳）、事件**大小/時間**可觀測。緩解：心跳僅對雙向同意
  聯絡人廣播、加入 **jitter**（ADR-0006 F5）、音樂併入心跳減少事件；未做流量填充（padding）。

### 被動網路監聽者
- 連線時間、流量大小可觀測（同上時序殘餘）。TLS/WSS 由部署層負責（中繼為 Cloudflare）。

### 惡意聯絡人（已是好友）
- **去匿名化**：WebRTC 直連會向對端揭露 IP。緩解：可注入 TURN（`rtcConfig`）遮蔽；對非
  信任聯絡人預設不直連（產品層）。**殘餘**：預設無公共 TURN（見已知限制 #5、ADR-0243）。
- **騷擾/spam**：加好友需雙向同意、封鎖名單（本機不再解其 Gift Wrap）。ADR-0014。
- **截圖/轉貼**：明確不在保護範圍（收回/限時為軟性、非強制）。

### 設備竊取者
- 私鑰與 DB 加密落地：DB→SQLCipher（ADR-0020）、私鑰→**OS 金鑰庫（B5，已實作，ADR-0053）**——
  桌面版經 `keyring`（Windows Credential Manager／macOS Keychain／Linux Secret Service）託管 nsec；
  舊版殘留 localStorage 的明文 nsec 於開機**遷入金鑰庫並抹除**（`key_set`／`key_get`／`key_delete`）。
- **殘餘（現況）**：正式桌面版 at-rest ＝私鑰（OS 金鑰庫）＋DB（SQLCipher）。瀏覽器示範/開發模式：
  應用資料以 nsec 導出 DEK 靜態加密（ADR-0112）；nsec 本身以**使用者密碼包裹**後落盤（passlock）
  或**不持久化**（未設密碼時）。**桌面版尚未程式碼簽章**（SmartScreen 警告）為發佈面殘餘、非 at-rest 缺口。

### 前向保密 / 後妥協安全（跨對手）
- **靜態 ECDH，無 FS/PCS**：單次私鑰外洩可解**全部歷史與未來**離線訊息。決策見
  **ADR-0028**：即時走 DTLS（已具 PFS）；離線留言之 FS/PCS 未來統一交由 MLS。
  **此為目前最重大的已知密碼學限制，稽核應優先檢視。**

## 已知限制清單（供稽核聚焦）

1. **離線訊息無前向保密**（ADR-0028）——最高優先。
2. **無金鑰輪替/撤銷**：私鑰即身分，外洩＝永久可冒名（PRD §4）。
3. **at-rest 加密（已具備）**：正式桌面版＝私鑰經 OS 金鑰庫（B5，ADR-0053，已實作）＋DB 經 SQLCipher；Web/開發模式＝應用資料以 nsec 導出 DEK 加密（ADR-0112）、nsec 以密碼包裹（passlock）或不持久化。殘餘＝桌面版尚未程式碼簽章（發佈面，非 at-rest）。
4. **顯示名稱不驗證/不傳遞**：以 npub 為唯一真實識別；名稱純本機、可重複（Zooko 取捨）。
5. **無公共 TURN（ADR-0243）**：一般使用者僅 STUN。兩個後果——(a) P2P 直連向聯絡人揭露 IP；
   (b) **對稱 NAT／嚴格防火牆下通話接不通**（即時影音只能 P2P/TURN、無文字中繼退路；**檔案有 relay
   退路、文字/nudge 本走 relay，不受影響**）。企業/自架可配 `turnServers`＋`forceTurn`（ADR-0048）。
   **明確標示、非靜默失敗**；公共 TURN 為成本決策（Cloudflare TURN 為候選）。TURN 一旦使用即見兩端 IP
   與通話時序/流量（不見內容，E2E 密文）——多一個元資料可見方，自架可避。
6. **群組成員清單一致性**依賴帶內控制訊息與多設備收斂；**已強化授權**（審查後，ADR-0027
   補述）：入群需你在名單內且建立者未被封鎖、`admin` 強制為驗證寄件人、加/移除限管理者、
   發訊須為成員。惡意管理者對其群內仍有成員操作權（設計內）；陌生人可發起含你的群（同
   「陌生人 DM」模型）但無法污染聯絡人或冒充 admin。通話亦已遵守封鎖名單。
7. **時序元資料**：上線/活動時間對中繼可觀測（已加 jitter，未做流量填充）。
8. **重放/時鐘**：ephemeral 以 `created_at` 窗口 + 事件 id 去重防重放（PRD §9）；跨裝置排序
   不僅信任 `created_at`。
9. **相依信任**：`nostr-tools`、`@noble/*`、`qrcode-generator`、`rusqlite`、`tokio-tungstenite`
   等第三方；供應鏈風險依 lockfile 鎖定版本。

## 建議的第三方稽核範圍

- `packages/core` 的加密路徑：Gift Wrap 封裝/拆封、NIP-44 使用、簽章驗證、群組扇出、
  一次性金鑰產生與 nonce 唯一性。
- 金鑰生命週期：產生、儲存（SQLCipher/OS 金鑰庫）、備份缺失的影響。
- 濫用防護：PoW、速率/大小上限、NIP-42 AUTH（relay，部署階段）。
- 上述「已知限制」逐項之實作正確性與是否有繞過。
