# 0232. 訊息互動強化：一鍵複製、點文字開串、串回覆同傳主對話

- 狀態：已接受
- 日期：2026-07-22
- 相關文件：ADR-0051（對話串：NIP-10 reply e-tag 攜於加密 rumor 內層）、ADR-0027（群組 Gift-Wrap 扇出）、ADR-0107（自封副本）、ADR-0183（thread-util 上移共用引擎）

## 背景與問題

三個訊息互動缺口：(1) 複製訊息文字要手動反白，長訊息麻煩；(2) 開對話串只能點小小的 🧵 鈕或回覆數，命中面積小；(3) 串內回覆只存在於面板（`mainMessages` 硬排除），沒有 Slack「also send to #channel」的選項——想讓主對話的人也看到串內結論時，只能手動複製貼上。

## 考量的選項

- **(3) 客端雙發**（勾選＝送兩則：一則進串、一則進主）：訊息重複、兩則 id 不同（回應/回條/收回對不上）。**否決**。
- **(3) 旗標 tag（採用）**：同一則訊息帶「也顯示於主對話」旗標，兩端渲染時決定；單一 id、語意乾淨。

## 決策

1. **一鍵複製**：`MessageLine` 加 📋 鈕（hover 浮現，樣式比照收回鈕）——複製**可見文字**（`splitAssetManifest` 後的 bodyText，不含行內資產 manifest JSON）；成功顯示 ✓ 1.5 秒。貼圖／檔案訊息不顯示。
2. **點文字開串**：`.text` 區塊點擊＝開啟該訊息所屬串（`openThread(rootIdOf(m))`——根訊息開自己的串、alsoMain 回覆開回原串）。**防誤觸**：點擊目標在 `a`/`button` 內（連結、顯示全文、遮罩展開）不攔；有反白選取不觸發。🧵 鈕保留（鍵盤/無障礙路徑）。
3. **串回覆同傳主對話（仿 Slack）**：
   - **契約**：rumor 內層加 app 專屬 `["also-main"]` tag（`alsoMainTag`/`alsoMain`，core `thread.ts`）；**reply e-tag 照舊**（NIP-10 相容）、隨 Gift Wrap 加密，中繼看不到。1:1（`wrapMessage`）與群組（`wrapGroupMessage`）皆支援；自封副本自然同步到自己其他裝置（ADR-0107）。
   - **引擎**：`ChatMessage.alsoMain?: boolean`；`sendMessage`/`sendGroupMessage` 加第 6/5 參數；收端讀 `alsoMain(rumor)` 落地。`mainMessages` 改為「排除回覆，**但 alsoMain 回覆保留**」——桌面與行動端（共用 thread-util）同時生效。
   - **UI**：串 composer 下方勾選「同時傳到主對話」（每次送出後重置）；alsoMain 訊息顯示 🧵 徽章（主對話與串內皆標，說明它同時存在兩處）。

## 理由

- 旗標 tag 讓「一則訊息、兩處顯示」以單一 id 成立——回應、回條、收回、串聚合全部照舊對得上；客端雙發全都會壞。
- 攜於加密內層＝中繼與外部觀察者看不到串結構與旗標，維持 ADR-0051 的隱私邊界；非本 app 的 NIP-17 客戶端讀到的仍是合法 NIP-10 回覆（旗標被忽略、僅顯示位置不同）。
- 點文字開串以「連結/按鈕優先＋選取防護」避免與既有互動（風險連結確認、遮罩展開、文字選取）衝突。

## 後果

- **正面**：複製、開串、串結論回流主對話三個高頻互動一步到位；跨端（desktop/mobile 檢視）語意一致。
- **負面／已知殘餘風險**：
  - 行動端 UI 尚未提供勾選（檢視 alsoMain 回覆會顯示於主列表，但其 composer 無此選項）——列後續。
  - 舊版客戶端收到 alsoMain 回覆：不認旗標 → 只顯示在串內（原行為），無害降級。
  - 點文字開串對純滑鼠使用者是新行為，誤觸以雙重防護緩解；不喜歡可再議設定開關。
- **後續**：行動端串 composer 加同款勾選；主列表 alsoMain 訊息可考慮顯示「來自：〈根訊息摘要〉」引用列（目前僅 🧵 徽章）。
