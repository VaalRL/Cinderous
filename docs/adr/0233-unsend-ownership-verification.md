# 0233. 收回訊息擁有者驗證（只有原寄件人能收回）

- 狀態：已接受
- 日期：2026-07-22
- 相關文件：ADR-0012（收回 v1：NIP-09 kind 5 經 Gift Wrap）、ADR-0107（自封副本／pendingReceipts 先到後補模式）、ADR-0119（群組收回扇出修正）

## 背景與問題

收回機制（ADR-0012）的接收端**不驗證收回者身分**：收到 kind 5 就照 `e` tag 標記目標為已收回。NIP-09 慣例是客戶端應檢查刪除事件作者＝目標事件作者，我們漏了。影響：

- **1:1**：對端可偽造 kind 5 指向**你自己的**訊息 id（同一對話雙方都知道所有 rumor id）→ 你這邊自己的訊息顯示「已收回」。影響輕微（對方本就是對話當事人）但語意錯誤。
- **群組**：**任何成員可收回任何人的訊息**（對全群生效）——UI 只把「收回」鈕限制在自己的訊息上，協定層沒有第二道防線；惡意成員可竄改群組對話呈現。

## 決策

1. **接收端擁有者驗證**（`applyDeletion`）：收到 kind 5 先以 `storage.findMessage(target)` 找到目標訊息，判定作者——自己送的（`outgoing`）＝自己；群訊看 `sender`；1:1 進訊＝對話對象（`contact`）。**作者 ≠ 收回寄件人 → 拒收、不標記**。寄件人身分由 Gift Wrap seal 驗證（既有機制），不可偽造。
2. **先到後補**（`pendingDeletes`）：目標訊息尚未抵達（中繼回放順序亂）時暫存（Map 保序、上限 1024 逐出最舊——比照 `pendingReceipts`），於訊息落地（1:1 文字／檔案 metadata／群訊三個落地點）後**驗證通過才**補套用。
3. **儲存層**：`AppStorage` 新增 `findMessage(id)`（跨對話以既有 `byId` 索引查找；memory／local／tauri 三實作補齊）。
4. **本機自發收回不設閘**：`unsendMessage` 的本機立即標記維持原樣——在自己裝置上隱藏訊息是使用者自己的選擇；防的是**別人**收回你的訊息。多裝置自封副本走同一驗證（自己的訊息 `outgoing`＝作者是自己，天然通過）。

## 理由

- 與 NIP-09 慣例對齊；驗證建立在既有的 seal 寄件人驗證上，零新密碼學。
- 「先到後補」不因網路順序犧牲合法收回，且沿用 `pendingReceipts` 的有界暫存模式（同型問題同型解）。
- 拒收採靜默丟棄（不通知、不記錄 UI）：偽造者得不到任何回饋。

## 後果

- **正面**：群組成員無法再刪他人訊息；1:1 對端無法讓你的訊息「被收回」。
- **負面／已知殘餘**：
  - 目標訊息已因保留上限（ADR-0094）修剪時，`findMessage` 找不到 → 合法收回進等待區直到被逐出（訊息都已不存在，收回與否無視覺差異；可接受）。
  - 收回者「在自己裝置上」仍可標記任何訊息為已收回（僅影響自己視角，非攻擊面）。
- 測試：1:1 偽造拒收、群組非作者拒收＋作者生效（engine 296 綠）；既有收回/多裝置測試不變。
