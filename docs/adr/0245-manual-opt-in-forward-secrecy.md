# 0245. 手動觸發、opt-in 的前向保密：把 0238 的輪替降階為使用者控制

- 狀態：提議中（實作計劃；沿用 ADR-0238 的加密機制、調整輪替模型；**上線前需外部審計**）
- 日期：2026-07-24
- 相關文件：**ADR-0238**（輪替加密子鑰 FS 的加密設計——本 ADR 沿用其機制，只改輪替觸發/預設/備份）、
  ADR-0236（FS 方向決策）、ADR-0237（元資料可連結性）、ADR-0240（真隱身）、ADR-0129（無內容心跳）、
  ADR-0061/0066（加密個人檔）、ADR-0070/0073（備份碼/救援）、ADR-0107/0071/0072（多設備同步/快照/配對）、
  `PRD.md §4／§6`

## 背景與問題

ADR-0238 把 FS 設計成**自動、每週 epoch 輪替**的粗粒度 FS。落地評估時使用者指出：自動連續 FS **方便性
下降不少**——離線超過 grace（14 天）的訊息會解不開、背景金鑰 churn、快照/備份要一直老化。使用者提案：
**先實作但預設不啟用、保留手動觸發更換、更換時自動更新散落的私鑰、並文字提示會斷掉什麼。** 本 ADR 據此
把 0238 的**加密機制原封保留**，只把**輪替模型**從「自動連續」降階為「**手動、opt-in、可當緊急按鈕**」，
並定案討論中釐清的幾個子問題（備份、EK 發現訂閱、「誰學到了」的可觀測性）。

**沿用 0238 不變的核心**：IK（npub）不動；隨機加密子鑰 EK（非 nsec 導出）、公鑰 IK 簽章；寄件人 retarget
Gift Wrap（wrap＋seal）到收件人當前 EK；EK 是**傳輸金鑰**、與 at-rest 儲存金鑰（nsec 導出）兩回事，故刪 EK
得 FS 又不弄丟本機歷史；kind 10040 公告＋rumor 內嵌雙通道發現；降級保護（capability＋TOFU）。

## 考量的選項

- **選項 A：ADR-0238 原案（自動每週 epoch、always-on）。** FS 最強（連續、~1–2 週粒度），但方便性差
  （離線>14 天失效、背景 churn、快照/備份持續老化），且 default-on 打擾既有使用者。
- **選項 B（本 ADR）：手動觸發、opt-in、預設關。** FS 強度＝「你多常換」，但方便性高、使用者可控（定期
  衛生＋懷疑被入侵時的緊急按鈕）、實作更簡單（免 epoch 排程）、不打擾既有使用者。
- **選項 C：不做 FS。** 維持現況「靜態金鑰、nsec 洩＝全解」。

## 決策

**採選項 B：沿用 ADR-0238 的加密機制，輪替改為手動/opt-in，並定案下列子問題。**

### 1. 觸發與預設

- **手動觸發**：使用者按「立即更換加密金鑰」才輪替（取代 0238 §1 的「每週 UTC epoch 自動」）。「當前 EK」
  ＝最近一次生成那把。**免 epoch 排程/週界計算**——0238 最麻煩的一塊直接省掉。
- **預設關（opt-in）**：`fs:ek-v1` 預設不啟用；關閉＝現行靜態 Gift Wrap（向後相容）。開啟才生成 EK、
  宣告 capability、發 10040、內嵌 rumor。
- **仍需 grace**：換完**不立刻刪**舊 EK（否則「還沒學到你新鑰的聯絡人」送到舊鑰的在途訊息會當場解不開）；
  保留舊 EK 一段（預設 7 天，可調），期間新舊 EK 皆可解，grace 後才刪。grace 是全域上限，不隨企業 TTL 放大。

### 2. 備份碼＝身分-only（不含 EK）

- **備份碼只帶身分（nsec），不帶 EK 私鑰**（修正討論中「更換時更新備份碼」的方向）。理由：凡進**持久
  備份**的 EK 就**不具 FS**（備份外洩即曝光那批訊息），且每次換鑰要重存備份碼＝正是要避免的不方便。
- 效果：備份碼**一次存終身、不隨換鑰改變**；FS 更強（寫下來的備份永不含 EK）。代價：**全裝置同時遺失後
  從備份救回時，少數「剛到、尚未落地封存」的在途訊息救不回**（多裝置 E2E 通性）。EK 私鑰只在**活著的
  裝置之間同步**（ADR-0107/0071/0072），不進備份碼。

### 3. EK 發現訂閱＝搭 presence 便車（同進退）

- 對方要送你、需你當前 EK。發現走 0238 的雙通道：**kind 10040**（首次接觸/查詢）＋**rumor 內嵌**（對話中
  即時、零元資料）。
- **10040 的訂閱搭既有 presence 訂閱**：`{kinds:[10040], authors:[聯絡人]}` 與心跳訂閱**同一批 authors**——
  聯絡人集合早因 presence 交給中繼（ADR-0237 已知殘餘），故 10040 訂閱**不新增獨立隱私面**。
- **必與真隱身同進退**（ADR-0240）：隱身時心跳訂閱被拿掉；10040 訂閱**也一起拿掉**，否則會把真隱身特地
  關掉的聯絡人集合洩漏又打開。故：非隱身聯絡人加了你就會訂 10040、學得到；隱身聯絡人不訂。

### 4. 「誰學到新 EK」的可觀測性與安全刪除

- **可精確知「已確認學到」**：收到對方用你**新 EK** 加密的訊息＝確認他已學到。每聯絡人記「最後對我用哪把 EK」。
- **無法確知「未學到」**：沉默模稜兩可（可能已從 10040 學到、只是沒送你訊息）；且 10040/rumor 都**無回執**。
- **「非隱身且近期上線」≈ 已學到**（他有訂 10040、上線即收到新公告）——但這是**啟發式、非證明**（時序/
  多裝置/推送不確定）。
- **故安全刪舊 EK 靠 grace 時間、不靠點名**：UI 可把聯絡人分「已確認新鑰 / 尚未確認」best-effort 呈現，
  但刪除以 grace 為準（grace 後仍送舊鑰者接不到，明確接受）。

### 5. 換鑰時的文字提示（誠實、必做）

按「更換」時彈窗提示：
- 還沒學到你新鑰的聯絡人（列「已確認 / 尚未確認」）、離線中的人，送到你**舊鑰**的在途訊息，**grace 後可能
  收不到**。
- 你的**舊備份碼**仍救得回**身分與本機封存歷史**，但救不回「本次換鑰後才到、且全裝置遺失前未落地」的訊息。
- 沒同步到新 EK 的其他裝置需先上線同步。

### 6. 降級保護、刪除紀律、範圍（沿用 0238）

- 降級：`fs:ek-v1` 在**簽章個人檔**內（不可偽造）＋TOFU 釘選；已釘選者找不到可用 EK **不得靜默退回靜態**，
  先試「上一把 EK＋近期 rumor 內嵌 EK」排除空窗，仍無→**警告/暫緩/重試**。未釘選者允許靜態退回。
- 刪除紀律：grace 後於**所有裝置**刪 `priv(EK)`、**從快照/配對包老化剔除**；安全刪除 best-effort（記為殘餘）。
- 範圍：**僅 1:1**、群組不納 v1、不追溯；文案上線前不得宣稱 FS。

## 理由

- **為何手動/opt-in 優於自動連續（對本專案）**：FS 的自動連續版把「離線>14 天訊息失效」「背景 churn」
  「快照/備份持續老化」壓在**全體**使用者身上換取連續性；而本專案定位本地優先、方便性重要。手動版讓
  **想要 FS 的人自己控制**（定期衛生＋緊急按鈕），其餘人零打擾——用「強度換方便＋可控」，且實作更簡單。
- **為何備份碼身分-only**：FS（刪鑰）與持久備份（留鑰可救）是**根本張力**——凡進備份的鑰就不 FS。身分-only
  同時**更方便**（備份一次存終身）又**更 FS**（備份永不含 EK），只犧牲「全裝置遺失時少數未落地在途訊息」。
- **為何 10040 訂閱搭 presence**：聯絡人集合已因 presence 交給中繼，10040 同 authors ＝零新增隱私面；隨真隱身
  同進退即守住「要完整隱私就隱身/自架」的既有結論（ADR-0237/0240 不變）。
- **為何刪鑰靠 grace 非點名**：「已學到」可觀測、「未學到」不可知（沉默/離線/隱身），且無回執——只能以
  grace 時間兜底。「上線≈學到」可當 UI 參考色，不可當刪除依據。
- **為何仍要外部審計**：改的是輪替模型、不是加密機制的正確性標準；FS 的密碼學正確性照樣要外部審計才上線。

## 後果

- 正面：
  - 得到**使用者控制的粗粒度 FS**（緊急按鈕＋定期衛生）：換鑰＋刪舊鑰後，被側錄的舊密文於 nsec 日後失竊
    時解不開；且不弄丟本機歷史。
  - **方便性顧慮化解**：預設關（不打擾）、手動（無背景 churn/離線失效驚嚇）、備份碼一次存終身。
  - **實作更簡單**（免 epoch 排程）、複用 0238 加密機制與既有同步/快照/配對；EK 發現搭 presence 訂閱零新增隱私面。
- 負面 / 已知殘餘風險：
  - **FS 強度取決於使用者多常換**；不換＝無 FS（同現況）。文案須誠實（「支援手動輪替取得 FS」，非「具 FS」）。
  - 無 PCS；grace 後送舊鑰的在途訊息接不到；race 期間 EK 未及同步→暫時未解待重試；TOFU 首用弱點；
    安全刪除 best-effort；**本機明文封存仍是最大長期暴露面**（FS 管不到）；kind 10040 洩漏「此 npub 用 FS 輪替」極小元資料。
  - 全裝置同時遺失＋只有身分-only 備份→少數未落地在途訊息救不回。
- 後續行動 / 待辦（分階段實作計劃）：
  1. ✅ **Phase 0 完成 — core 純函式**：`packages/core/src/subkey.ts`——`generateEncryptionKey`、
     `buildEkAnnounce`/`readEkAnnounce`（kind 10040 IK 簽章、驗 kind/簽章/pk）、`withEkHint`/`ekHintOf`
     （rumor 內嵌）、`openWrapWithEks`（多鑰依序解封）。**關鍵發現：nip59.sealAndWrap/openWrap 已完全
     參數化在 recipientPk/Sk、nip44 對話金鑰對稱 → retarget 到 EK 不需改 nip59**，呼叫端改傳 EK 金鑰即可，
     `#p` 仍身分供路由。TDD 11 測含 **FS 核心**（刪 EK 後即使 IK 也解不開被側錄密文）、向後相容退回 IK、
     多 EK 解封、全錯拋、10040 驗簽、hint 往返。core 451 綠。
  2. **Phase 1 — 引擎（opt-in、手動）**：storage 存 EK（當前＋grace 內舊把）／每聯絡人 learned-EK／FS 釘選；
     backend `rotateEncryptionKey()`（手動：生成→發 10040→之後訊息內嵌→保留舊 EK 至 grace→排刪）；發布路徑

     多裝置 EK 同步（0107/0071/0072）；快照/配對老化剔除；備份碼保持身分-only。
  3. **Phase 2 — UI（desktop）**：設定頁「啟用前向保密／立即更換金鑰」；換鑰**警告彈窗**（列已確認/尚未確認
     聯絡人、備份影響、其他裝置同步）；每聯絡人 FS 狀態（已確認新鑰/尚未確認）；誠實文案（不宣稱 FS）。
