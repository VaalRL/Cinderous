## 決策後記（2026-07-12）：暫緩，維持 ADR-0028

評估後**暫不採納本升級**，維持 ADR-0028（靜態金鑰＋即時走 DTLS＋誠實揭露）。本 ADR
以「已棄用」保留，作為**設計討論與觸發條件的紀錄**；未來若滿足 §4 觸發條件，可另立新
ADR 承接此設計。暫緩的兩顆決定性砝碼：

1. **relay 不穩定 × MLS 有狀態耦合，放大失效爆炸半徑**：現行靜態方案每則 Gift Wrap
   **自足**，掉一則就只掉那一則（孤立、自癒）；MLS 下訊息依賴所在 epoch，掉/亂序/過期一個
   **commit** 可能讓**後續整段對話在復原前解不開**（相關、連鎖）。而此 relay 生態本就
   注定不穩定（免費層休眠 ADR-0059、社群自架可能睡著 ADR-0075、7 天硬過期 ADR-0065、
   盡力而為不保證有序 ADR-0027）——不穩定是**設計前提而非邊角案例**。尖角殘留：**單裝置
   使用者 × 全程離線 >7 天 × 關鍵 commit 過期**，追趕來源已被 TTL 銷毀，只能靠對方/管理者
   當下可達的 Welcome/re-add 復原，否則卡住。G1–G4 護欄能吸收日常抖動，但無法完全消除此尖角。
2. **groupId 元資料退步（修不掉的取捨）**：現行成對 Gift-Wrap 扇出對中繼**完全不暴露
   群組/成員**；NIP-EE/MLS 會使 **groupId 可觀測**。對「極致隱私」為核心價值的專案，這是
   **永久取捨、非 bug**，比可工程化的 relay 抖動更傷根本。

結論：以「靜態金鑰＋即時 DTLS 承擔即時 FS」所留的、**已誠實揭露**的離線留言 FS 風險，
換取「精簡、可靠、群組元資料零暴露」，在現階段（未稽核、免費層、社群自架 relay）ROI 更佳。

## 背景與問題

現況 1:1 與群組（ADR-0027）皆採 **NIP-44 靜態 ECDH**：固定不變的共享秘密加密所有
訊息，等於「一把金鑰解過去＋未來全部」。此性質被兩件事放大成實質風險：

1. **中繼在威脅模型中即為對手**（PRD §6，頭號對手）：惡意 relay 可違背「7 天銷毀」，
   把 Gift Wrap 密文**永久囤積**。
2. **協定不提供金鑰輪替/撤銷**（PRD §4）：靜態私鑰**將來某日**外洩（設備遭竊、脅迫、
   供應鏈），即可把 relay 囤積多年的密文**一次全解**——典型的 harvest-now,
   decrypt-later。

即時通道（WebRTC 資料通道走 DTLS）**已具傳輸層前向保密**（ADR-0028 關鍵觀察 1），
故 **FS 缺口僅在「經中繼的離線 Gift Wrap 留言」這一條路**。本 ADR 決定如何封住這條路，
並確立「升級後仍讓正常使用者無痛運作」的可靠性護欄為驗收條件。

要修補的密碼學性質有二：

- **前向保密（FS）**：拿到現在的金鑰推不回過去 → relay 囤的歷史密文，日後金鑰外洩也解不開。
- **後妥協安全（PCS／自癒）**：金鑰一度外洩後未來自動恢復保密 → 把單次外洩的爆炸半徑
  從「永遠」縮成「一個窗口」。

## 考量的選項

- **選項 A：維持現狀（ADR-0028）**——靜態金鑰，離線留言無 FS/PCS，僅靠即時 DTLS 與
  誠實揭露。爆炸半徑＝帳號終身。
- **選項 B：為 1:1 單獨導入 Double Ratchet**——1:1 得 FS/PCS，但群組（ADR-0027 扇出）
  仍無；需維護**兩套**密碼子系統、姿態不一致、雙倍稽核面。
- **選項 C（本 ADR 採用）：統一採 MLS（RFC 9420 / NIP-EE）**——把 1:1 視為 2 人群組，
  與多人群組**同一套機制**取得 FS+PCS；重用已稽核的 **OpenMLS（Rust/Tauri 層）**。

## 決策

### 1. 以 MLS 統一 1:1 與群組的 FS/PCS

- 1:1 視為 2 人 MLS 群組；多人群組共用同一 TreeKEM 機制。成員為樹的葉子，`commit`
  旋轉 epoch 金鑰：舊 epoch 金鑰刪除（**FS**），成員 rekey 修復先前外洩（**PCS**），
  加/移除成員 **O(log N)**。
- **實作邊界（跨層例外，需明記）**：MLS 密碼核心用**已稽核的 OpenMLS（Rust）**跑在
  **Tauri 原生層**，**不**在 `packages/core`（TS SSOT）重寫 TreeKEM。此為對「邏輯一律
  收斂 core」預設的**刻意例外**，理由是「不自造密碼原語、優先重用稽核過的函式庫」。
  行動端於 RN 工具鏈就緒後對應原生綁定；瀏覽器 demo 後端不支援 MLS（維持靜態或停用）。
- **Nostr 綁定採 NIP-EE**：KeyPackage（預金鑰）與群組 commit/應用訊息以 Nostr 事件遞送，
  **外層維持 NIP-17/59 Gift Wrap**。relay 全程只轉發密文與預金鑰庫，**零伺服器狀態與
  只轉發密文的隱私鐵則不變**。

### 2. 正常使用者可靠性護欄（驗收條件，缺一不可）

棘輪的安全性來自「用完即刪金鑰」，這正是可靠性的天敵：狀態一旦對不齊，訊息即
**永久解不開**。以下四條護欄把「為 FS 刪金鑰」與「訊息會遲到／亂序／多裝置／重灌」
調和，**列為升級的驗收條件**：

- **G1 非同步（預金鑰）**：每人預先發佈一批**一次性預金鑰**＋一顆**簽章 last-resort
  預金鑰**至中繼；對方離線時發訊方抓預金鑰 bootstrap 會話，訊息於 ≤7 天窗口等取件
  （對齊 ADR-0065，遞送模型不變）。**護欄**：一次性用盡必退回 last-resort，杜絕
  「某人暫時無法被傳訊」；客戶端上線自動補充庫存。
- **G2 亂序/遲到（跳號金鑰快取）**：收到跳號訊息時，先推導並**有界快取**被跳過的
  per-message key，使遲到/亂序訊息仍可解。快取**有界＋壽命對齊 7 天 TTL**：吸收遲到又
  不破壞 FS；逾窗未到者比照現況「超時未取件」消失（無退步）。與 ADR-0034 event id 去重相容。
- **G3 多裝置（裝置即群成員）**：把「你＋對方的所有裝置」建成 MLS 群，新增/移除裝置＝
  O(log N) commit。**新裝置歷史不從 relay 回解**（FS 使舊金鑰已抹除）——改走 **ADR-0072
  桌面配對克隆**：一次性 P2P 全量搬明文歷史＋身分（SAS 互認、不經中繼），新裝置再以新葉子
  入群接手未來訊息。全部裝置皆毀＝帳號死亡（同 PRD §4，無退步）。
- **G4 自動復原（永不顯示「解不開」）**：持續解密失敗 → **自動會話重置**（抓新預金鑰
  重握手）；對方重灌/輪替 → 顯示「安全金鑰已更新」溫和提示後自動重建，不顯示錯誤牆。
  棘輪/MLS 狀態於加密 SQLite（SQLCipher，ADR-0020）**交易式原子持久化**並跨裝置同步
  （ADR-0009），避免半個棘輪步導致 desync。

### 3. 群組有序遞送與 epoch 追趕（MLS 特有）

- MLS `commit` 推進 epoch **須有序**，否則成員 epoch 不同步即解不開。重連時客戶端
  **自動抓漏掉的 commit、按序重放至目前 epoch** 才顯示新訊息（無感）。遞送可靠性藉
  **ADR-0041 外送匣（OK 感知重試＋重連補送）** 承載。
- **commit 過期復原**：關鍵 commit 若已逾 7 天被清，提供 **Welcome/re-add 自動重新入群**
  復原路徑（把該成員當新加入），而非卡死。此為「盡力而為中繼 × MLS 有序 epoch」張力
  （ADR-0027 選項 C 已標記）的必解項。

### 4. 觸發條件（沿用 ADR-0028，何時才啟動本升級）

威脅模型升級為需對抗設備/金鑰**事後**外洩、第三方稽核（F4）建議、或明確高敏感使用者
需求三者之一出現，才把本 ADR 由「提議中」推進為施工。

## 理由

- **一致性**：MLS 一套同時覆蓋 1:1 與群組，避免選項 B 的兩套機制與姿態不一致。
- **廢掉惡意 relay 的超能力**：每則/每 epoch 用各自不同、推導後即抹除的金鑰；relay 囤得再多
  再久，事後金鑰外洩也回推不出歷史 per-message key（FS）——**囤積的密文倉庫對歷史解密變廢鐵**，
  直接中和 harvest-now-decrypt-later。PCS 再把外洩爆炸半徑縮成一個窗口。
- **最小新稽核面**：重用 OpenMLS（不自造 TreeKEM）；可靠性所需的多裝置同步（0009）、
  配對克隆（0072）、外送匣（0041）**地基已存在**，新工作集中在 MLS 有序遞送與 epoch 追趕。
- **可行性已被既有零件支撐**：G1–G4 對應的機制多為既有能力的延伸，而非從零長出。

## 後果

- **正面**：封住「經中繼離線留言」唯一的 FS 缺口；1:1 與群組取得 FS+PCS 且姿態一致；
  移除群組成員即失去未來讀取（比現行「下次扇出略過」更強的 PCS）；隱私鐵則與 relay 零狀態不變。
- **負面 / 已知殘餘風險**：
  - **元資料不受益**：MLS 只補「內容」FS/PCS；relay 仍見**收件人 pubkey、上線時序、IP、
    訊息大小**（威脅模型殘餘）——需 Gift Wrap/TURN/jitter/隱身另解，非本 ADR 範圍。
  - **groupId 元資料退步**：現行成對 Gift-Wrap 扇出（ADR-0027）對中繼**完全不暴露群組/成員**；
    NIP-EE/MLS 會使 **groupId 可觀測**——補內容 FS 卻在群組元資料隱藏上退一步，為明確取捨。
  - **狀態脆弱性**：棘輪「不能弄丟狀態」，持久化與跨裝置同步的正確性要求高於現況，是主要新稽核面。
  - **多裝置複雜度**：裝置即群成員會增加 epoch 變動頻率，須與 ADR-0009 一併設計。
  - **跨層邊界**：MLS 走 Rust/OpenMLS 而非 TS core，破例需在實作時嚴守「僅密碼核心下沉、
    契約仍由 core 定義」以免多重 SSOT。
- **後續行動 / 待辦**（本 ADR 接受後）：
  1. 定案 NIP-EE 事件契約（KeyPackage/commit/Welcome kind 與 tag），更新 ARCHITECTURE §5。
  2. Tauri 層整合 OpenMLS 與金鑰庫/加密儲存（ADR-0020/0054）。
  3. 實作 G1–G4 護欄與 §3 epoch 追趕/復原，補測試（含亂序、遲到、重灌、加/移除裝置情境）。
  4. F4 安全自評把「離線留言無 FS」風險項改為「已由 MLS 覆蓋」，並列 groupId 元資料取捨。
  5. 接受後把 ADR-0028 標記為「被取代 by 0091」、更新 PRD §6/§7 揭露文字。
