# 0179. 行動端離職接管（企業主端，加密託管）＋公司儲存槽收檔桌面限定提示

- 狀態：已接受（已實作）
- 日期：2026-07-17
- 相關文件：ADR-0163（入職金鑰託管＋離職接管）、0112（靜態加密／絕不明文存 nsec）、0161
  （公司儲存槽）、0178（行動端企業主管理 v1）、0177（行動端儲存槽員工端）

## 背景與問題

ADR-0178（企業主管理 v1）明載兩項留待：**離職接管**（D-v2）與**企業主收儲存槽落盤**（路線圖 E）。
使用者裁示：**離職接管要在手機實作；企業主收儲存槽不做，但手機介面上要文字提示**。

離職接管的核心是**持久化員工的託管私鑰**（`onOrgEscrow`）——這踩紅線：桌面把託管 nsec **明文**
存 localStorage，行動端 web 沒有 OS 層保護，明文存員工私鑰＝XSS 一次撈走全公司金鑰。

## 決策

### 1. 託管清單**加密**落盤（行動端與桌面的關鍵差異）

新增 `apps/mobile/src/org-escrow.ts`：純函式（upsert/remove/offboardedEntries）同桌面，但
load/save 以**企業主自己 nsec 導出的金鑰**（ADR-0112 `deriveStorageKey`＋`sealValue`/`openValue`）
封裝——`nb.orgEscrow.<admin>` 落盤為密文（`c1:` 前綴），離開企業主 nsec 就解不開；企業主 nsec
本就不明文落盤（ADR-0112 紅線不破）。`onOrgEscrow` 到達即加密 upsert。

### 2. 離職接管流程

- **標記離職**＝從名冊移除該成員並重發布（ADR-0178 名冊畫面已可）；**離職**＝託管中但不在
  現行名冊在世成員（`offboardedEntries`）。
- **接管查看**：以託管私鑰在本機登入該離職員工身分（`signInWith` 加 `overrideRelay` 指向其
  公司座）查看中繼站仍保留的歷史；**立即設隱身**（離職員工不應顯示在線、不再廣播）。
- **刪除託管**：企業主決定不再保留 → `removeEscrow` 後重新加密落盤。
- UI 在組織名冊畫面新增「離職帳號接管」區（`offboarded` 只帶 pubkey＋name，**nsec 不進 UI 元件**，
  留在 App 層加密儲存）。

### 3. 公司儲存槽收檔＝桌面限定，手機**文字提示**

企業主端收檔落盤需真實檔案系統（桌面 Rust/Tauri 寫檔＋`index.jsonl`），手機 web 沒有——**不實作**。
在組織名冊畫面加一段明確提示（`settings_vaultDesktopOnly`）：收檔落盤僅桌面，員工可從手機存入、
企業主收檔請用桌面。

## 後果

- 正面：企業主可**純從手機**完成離職接管（標記離職→接管查看歷史→刪除託管），且託管私鑰在手機
  **加密**儲存（比桌面明文更嚴）。企業主管理鏈路（建立/名冊/邀請/離職接管）在手機齊備。
- 已知限制／取捨：
  - **接管查看為 transient**（不寫登錄）：查看一次，需要再看就再接管（託管持久，隨時可接）；
    未做「以離職身分常駐於切換器（🗄）」——桌面有，行動端後續可補。
  - **企業主收儲存槽落盤仍桌面**（路線圖 E，原生 FS）；手機已明確文字提示，不留模糊。
  - 接管以隱身避免廣播；不做更強的「orgOffboarded 完全不連線只讀本地」隔離——v1 夠用。
- 路線圖：**A~D 全數落地**（D-v2 離職接管完成）；唯一剩下的 **E（企業主收儲存槽落盤）** 為
  平台限制（需原生建置），已於手機介面文字標明。
- 測試：`org-escrow.test`（加密往返/密文不含明文 nsec/錯金鑰回空/純函式/SSR 安全）；
  `RosterAdmin.test` 補離職接管區與儲存槽提示；mobile 188 綠燈、i18n/mobile typecheck 通過。
