# 0145. 新增身分先選類型（個人／組織）

- 狀態：已接受（已實作）
- 日期：2026-07-15
- 相關文件：**ADR-0046/0047（企業/組織身分：`enterprise` 旗標＋管理者名冊訂閱）**、
  0045/0138（多身分登錄）、0070（加密備份碼匯入）、0122（瀏覽器新身分本地密碼必填）、
  0139（統一自訂對話框）、0141（統一按鈕系統）

## 背景與問題

「新增身分」彈窗過去是**單一表單**：名稱、relay、（隱藏在後段的）一個「這是組織身分」核取方塊、
管理者 npub 欄、匯入 nsec/備份碼。組織身分（ADR-0047：訂閱管理者發佈的名冊、自動採用通訊錄與政策）
與個人身分（一般帳號、聯絡人自己管理）在概念上是**兩種不同的東西**，卻被塞進同一張表單，使用者：

1. 不易察覺自己能建立「組織身分」，也不懂那個核取方塊代表什麼取捨；
2. 個人身分的使用者仍會看到與己無關的管理者 npub 欄（雜訊）。

使用者要求：「點擊新增身份時跳出彈窗選擇要新增個人身分還是組織身份。」

## 決策

### 1. 兩步流程：先選類型，再填表單

`AddIdentityModal` 改為兩步。以 `mode: "personal" | "org" | null` 取代原本的 `enterprise` 布林，
`enterprise = mode === "org"`：

- **第一步（`mode === null`）**：顯示 `.addid-choose`，兩個大按鈕——
  - 👤 **個人身分**（`addid-mode-personal`）：一般帳號；聯絡人由你自己管理。
  - 🏢 **組織身份**（`addid-mode-org`）：工作身分；訂閱組織管理者發佈的名冊，自動採用通訊錄與政策
    （可選填管理者 npub）。
  - 每顆按鈕含圖示＋標題＋一行說明（`addid-mode__ic/__name/__hint`），讓兩種身分的差異在**選之前**就講清楚。
- **第二步（選定後）**：進入表單，頂端有「← 改選類型」（`addid-change-mode`，`setMode(null)` 回第一步）
  與目前類型標示（`🏢 組織身份` / `👤 個人身分`）。**管理者 npub 欄（`addid-admin`）只在組織身分出現**；
  名稱、relay、本地密碼（瀏覽器必填，ADR-0122）、匯入 nsec/備份碼（ADR-0070）維持不變。

### 2. `initialMode` prop（測試／深連結）

新增可選 `initialMode?: "personal" | "org" | null`（預設 `null`＝先選類型），初始化 `mode`。用於
SSR 測試直接渲染某類型表單，未來也可支援「直接建組織身分」的深連結入口。`onAdd` 簽章不變
（`(name, relayUrl, enterprise, opts)`），呼叫端零改動。

## 後果

- 正面：
  - 「個人」與「組織」在**選之前**就以圖示＋說明講清楚差異；個人身分不再看到與己無關的管理者欄。
  - 組織身分（ADR-0047）從隱藏核取方塊升級為**平等的一級入口**，更易被發現。
  - 沿用既有 `onAdd`／`enterprise` 語意與 ADR-0141 按鈕樣式，無資料流變更、無新後端。
  - 測試 +3（SSR：預設顯示選類型步驟且尚無表單、選組織→出現管理者欄、選個人→無管理者欄），
    既有 3 題改以 `initialMode="personal"` 直接渲染表單。全綠 desktop 326／i18n 8。

- 已知限制：
  - 目前只有桌面版走兩步選類型；行動端多身分（ADR-0138）的新增流程維持既有表單，待後續對齊。
  - 「原地把個人身分升級成組織」不在本 ADR 範圍——建立組織身分仍是新增一個身分（新金鑰）。
