# 0237. 中繼層元資料可連結性：連線歸屬打穿事件層隱藏

- 狀態：已接受（Tier 0/1 為承諾工作項；Tier 2/3 另立後續 ADR）
- 日期：2026-07-23
- 相關文件：`PRD.md §6／§13`、`ARCHITECTURE.md §1／§5／§9`、ADR-0002（隱私元資料基線）、
  ADR-0057（NIP-42 AUTH）、ADR-0123（具名訂閱／反爬蟲）、ADR-0088（P2P presence 卸載）、
  ADR-0109（自適應心跳）、ADR-0120/0129（typing/nudge/status 封裝、無內容心跳）、
  ADR-0034（多中繼路由）、ADR-0235（濫用防護）、ADR-0236（前向保密）、
  `docs/relay-metadata-observability.md`

## 背景與問題

§1 誠實化（見前次提交）已把「中繼站無法重建社交圖譜」修正為「訊息層隱藏、presence 層洩漏
聯絡人集合」。但深入追下去發現：**兩者其實是同一個根因，而且這個根因也打穿了訊息層的
寄件人隱藏**——比原先以為的更廣。

### 根因：連線都綁在你的真實 npub 上

為了反爬蟲，中繼要求 NIP-42 AUTH（ADR-0057/0123）——**讀與寫都要先認證**
（`relay-core` 的 `handle`：REQ 與 EVENT 未認證皆拒）。而客戶端**每一條連線都以真實金鑰
認證**（`relay-backend.ts:615/744`：`authSigner = buildAuthEvent(challenge, url, this.sk)`），
且發布與訂閱都走這同一條已認證連線。於是一個會把「連線的認證身分 ↔ 該連線的收發」關聯起來
的中繼（它本來就握有 `authState`，這關聯是零成本的），看得到：

1. **你的 presence 訂閱** `{kinds:[20000], authors:[聯絡人]}` → **你的聯絡人集合**，綁在你的真名上。
2. **你發布的 Gift Wrap**（外層 `#p` = 收件人真名，供路由）→ **你的送出邊 A→B**，
   綁在你的真名上——**即使外層事件作者是一次性臨時金鑰**。

換句話說：**事件層的隱藏（Gift Wrap 的臨時作者、sealed presence）被連線層的歸屬打穿。**
Gift Wrap 的臨時作者保護的是「靜態資料與其他客戶端看不到寄件人」——但一個主動把 live 連線
認證身分與 live 發布關聯的中繼，仍然看得到寄件人。NIP-17 預設的威脅模型假設你可能以**未認證
或臨時身分**的連線發布；Cinderous 為反爬蟲加上的 requireAuth 恰好取消了這個前提。

> 附帶張力（ADR-0235 H1 速率限制）：Gift Wrap 洪水的修法正是「以**認證身分**（非臨時事件
> 作者）計數」——反濫用**需要**知道真實寄件人，而元資料隱私**想要**藏住它。同一個 `authState`
> 既是防濫用的依據，也是這條洩漏的來源。

### 為什麼 crypto 層單獨修不好：IP／連線關聯

「那就用臨時身分認證發布連線」——問題是：所有連線都來自**你的同一個 IP**。一個按 IP／連線
關聯的中繼會把「真名 A 的收件匣連線」與「臨時 Q 的發布連線」在同一 IP 上重新綁起來
→ 臨時身分白搭。對一個**託管**中繼（如 Cloudflare），跨連線的應用層關聯是零成本、IP 關聯是
低成本。**因此 crypto／連線層的混淆，在沒有傳輸層匿名（Tor/onion）之前，對託管中繼幾乎不付
出實質效益。** 這是本 ADR 最重要的結論。

### 已有的緩解（Tier 0，已上線）

- **P2P presence 卸載（ADR-0088e）**：資料通道已開時 presence 走 P2P、抑制 relay 信標；
  全聯絡人皆有 P2P 時完全不發 relay 信標。**這是目前最有效的槓桿**——它把中繼直接移出 presence。
- **最小信標（ADR-0129）**：裸心跳（20000）已無內容，狀態文字/音樂/typing/nudge 改走封裝定址
  （ADR-0120/0129），洩漏面已縮到「在線與否＋節奏」。
- **時序抖動（ADR-0006 jitter）**、**僅對雙向同意聯絡人廣播**。

## 考量的選項

| # | 選項 | 修什麼 | 成本 | 關鍵殘餘 |
| --- | --- | --- | --- | --- |
| 0 | 維持現狀＋文件化 | — | 零 | 全部 |
| 1 | **嚴格 presence 隱私（opt-in）**：關掉 `authors:[聯絡人]` 訂閱，presence 只走 P2P | presence 訂閱洩漏 | **低**（gate 一個訂閱） | 只看得到有 P2P 的聯絡人在線；發布邊仍歸屬 |
| 2 | 臨時身分認證發布/訂閱連線 | 連線歸屬（發布＋訂閱） | 中（多連線、re-AUTH） | **IP 關聯即打穿**；Sybil |
| 3 | 輪替每對金鑰 presence（§9 原議） | presence 作者可連結 | 高（**N× 信標**，撞 ADR-0109 容量） | IP 關聯；批次發布又重新聚類 |
| 4 | 共享輪替 presence 金鑰（1 信標變體） | presence 作者可連結 | 中（金鑰分發/輪替） | 惡意聯絡人可解匿；watcher 聚類；IP 關聯 |
| 5 | presence 改走封裝定址（如 typing/nudge） | presence 訂閱洩漏 | 高（N× 發布＋失去即時 catch-up） | 發布 `#p` 仍歸屬（同訊息層） |
| 6 | 傳輸層匿名（Tor/onion 中繼） | **連線＋IP 關聯** | 很高 | 延遲、可用性；生態工程 |
| — | **自架中繼**（隨時可用） | 全部（對手＝你自己的節點） | 使用者自理 | 需自行營運 |

關鍵觀察：**選項 2–5 全部被選項 6（或缺它）擋住**——沒有傳輸匿名，IP 關聯就把 crypto 層的
努力抹平。選項 3（§9 原本設想的「輪替金鑰 presence」）除了 IP 問題，還獨立撞上 **N× 信標**的
容量牆（PRD §13 已記）：presence 是 1-對-N，per-contact 金鑰＝一人發 N 顆不可合併的信標。

## 決策

採**分層路徑**，本 ADR 定方向並承諾 Tier 1：

- **Tier 0（已上線，持續強化）**：把 **P2P presence 卸載（ADR-0088e）做得更積極**——它是唯一
  能真正把中繼移出 presence 的機制，且效益隨 P2P 普及提升。列為主要槓桿。
- **Tier 1（近期承諾，選項 1）**：**opt-in「嚴格 presence 隱私」模式**——啟用後不對中繼發出
  `authors:[聯絡人]` 訂閱，presence 僅由 P2P 資料通道取得。**你以「看不到離線／無 P2P 聯絡人的
  在線狀態」換得「聯絡人集合永不上中繼」。** 成本極低（gate 一個訂閱＋一個設定），不撞容量、
  不破反爬蟲、對隱私極大化者是誠實且划算的取捨。
- **支援解答：自架**。對「需要完整元資料隱私」的使用者，**官方立場是自架中繼**（洩漏對象＝
  你自己的節點）。文件與官網已朝此導引（§1 誠實化、FAQ「自架連元資料都不外流」）。
- **Tier 2/3（延後，另立後續 ADR）**：臨時身分連線（選項 2）與傳輸匿名（選項 6）**綁在一起**
  評估——**在傳輸匿名到位前不單獨建臨時身分連線**，因為 IP 關聯會讓它白搭。選項 3/4/5（輪替
  金鑰 / 封裝 presence）僅在 Tier 3 之上、且容量許可時才重新評估。

**本階段明確不做**：輪替金鑰 presence（選項 3/4）、封裝定址 presence（選項 5）、傳輸匿名層
（選項 6）。理由見下。

## 理由

- **不建輪替金鑰 presence（回應 §9 原議）**：這是本 ADR 被要求評估的方案。結論是它**不划算**：
  (a) per-contact 版撞 N× 信標容量牆（ADR-0109）；(b) 共享金鑰版可被任一惡意聯絡人解匿、且
  watcher 仍聚類；(c) **兩版都被 IP 關聯打穿**——花大力氣做 crypto，對託管中繼卻幾乎零收益。
  與其做一個昂貴又有結構殘餘的半套，不如把力氣放在真正有效的 P2P 卸載與「自架＝真解答」。
- **為何 Tier 1（嚴格模式）值得**：它不試圖用 crypto 騙過中繼，而是**直接不把資料交出去**——
  這是唯一不被 IP 關聯打穿的客戶端側手段。代價（只看 P2P 聯絡人在線）對隱私極大化者可接受，
  且工程量極小。
- **為何自架是官方解答**：連線歸屬＋IP 關聯的根因，對**第三方託管**中繼才成立。自架把對手
  從「Cloudflare／可傳喚方」變成「你自己」，一次解決 presence 與發布兩條洩漏。這與整體
  「本地優先、可自架」定位一致，也是最誠實的答案。
- **與單一全域 DO 的關聯**：Tier 2 的「多連線」會加重單一全域 DO 的連線數壓力——這是**再一個
  該分片的理由**（見與 C 分片的討論；目前尚無 ADR）。反過來，分片＋自架也順帶稀釋本問題。

## 後果

- 正面：
  - 把「presence 洩漏」正確地重新框定為「連線歸屬打穿事件層」的**統一根因**，並涵蓋了原先
    遺漏的**發布邊歸屬**（訊息層寄件人對主動關聯的中繼其實可見）。
  - Tier 1（嚴格模式）給隱私極大化者一個不被 IP 關聯打穿、且今天就能做的選項。
  - 明確把「自架」定為元資料隱私的官方解答，避免對託管中繼做出做不到的承諾。
  - 存下對「輪替金鑰 presence」的嚴謹否決理由，省下未來重複評估的成本。
- 負面 / 已知殘餘風險：
  - **不啟用嚴格模式時，presence 訂閱與發布邊對主動關聯的中繼可見**（託管站）。緩解＝P2P 卸載、
    自架。
  - **嚴格模式的代價**：看不到離線／無 P2P 聯絡人的在線狀態。
  - **反濫用與隱私的內在張力**：ADR-0235 的速率限制以認證身分計數，本身即依賴連線歸屬。
  - **完整解需傳輸匿名（Tor/onion）**，延後；在此之前 crypto 層混淆對託管中繼不付出效益。
- 後續行動 / 待辦：
  1. 進一步校正 §1 與 `docs/SECURITY.md`：訊息層寄件人隱藏應加註「對**主動關聯連線認證與發布**
     的中繼可被歸屬」——目前措辭仍偏樂觀。
  2. 實作 Tier 1（嚴格 presence 隱私 opt-in）：`relay-backend` gate `authors:[聯絡人]` 訂閱
     ＋設定項＋UI 說明取捨；TDD。
  3. 強化 P2P presence 卸載的覆蓋率量測（多少比例聯絡人的 presence 已不經中繼）。
  4. Tier 2/3（臨時身分連線＋傳輸匿名）與**單一全域 DO 分片**一起立後續 ADR。
  5. 產品文案守則：不得宣稱對第三方中繼「完全隱藏社交圖譜」；元資料隱私的答案是自架。
