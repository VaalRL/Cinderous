# 0199. 「解鎖隱藏身分」🔒 入口收斂——閘在「有啟用密碼的身分」而非「有隱藏身分」

- 狀態：已接受
- 日期：2026-07-18
- 相關文件：`apps/desktop/src/App.tsx`、`apps/desktop/src/App.test.tsx`、ADR-0067（隱藏身分）

## 背景與問題

身分切換列的 🔒「解鎖隱藏身分」入口（`App.tsx`）原本只以 `isTauri()` 為條件——**桌面永遠顯示**，即使使用者從未設過任何密碼或隱藏身分。點下去只會得到「密碼不符任何隱藏身分」，對沒用過此功能的人是純困惑（多數使用者屬此情形）。

直覺的修法是「**有隱藏身分才顯示 🔒**」，但這會**反噬功能目的**：隱藏身分（ADR-0067）要保護的是**否認性**——旁人看不出這台裝置藏了帳號。若 🔒 只在「存在隱藏身分」時出現，則 **🔒 的出現本身就洩漏了「這裡藏了東西」**，等於自毀否認性。

## 決策

把顯示條件改為 `isTauri() && shouldOfferUnlockHidden(profiles)`，其中純函式

```ts
export function shouldOfferUnlockHidden(profiles: Profile[]): boolean {
  return profiles.some((p) => p.locked);
}
```

閘在「**本機存在任一已啟用密碼（`locked`）的身分**」，而**不是**「存在隱藏身分（`hidden`）」。

理由鏈：
- 隱藏身分的**必要前置**就是該身分已啟用本地密碼（停用密碼會一併解除隱藏，ADR-0067）；所以「有密碼」是「可能有隱藏」的最小可觀測前提。
- 「有密碼」**不等於**「有隱藏」（使用者可能只想要開機解鎖、沒藏任何東西）。因此 🔒 出現時，旁人**無法區分**「只是有密碼」與「真的藏了帳號」→ 否認性保住。
- 沒設過任何密碼的常見情形 → 看不到 🔒 → UI 乾淨、不再誤導。

依 `convoVisibleIn` 既有慣例把判斷抽成**匯出的純函式**，於 `App.test.tsx` 單元測試（TDD）。

## 後果

- 正面：三方平衡（乾淨 UI／否認性／可發現性）；行為可測、自我文件化。
- 中性：正在作用中的隱藏身分仍照 ADR-0067 顯示於切換器（本 ADR 不動可視性規則，只動 🔒 入口）。
- 負面 / 已知殘餘風險：極端情境——使用者啟用了密碼、後來又全部停用，則 🔒 消失屬預期（沒有可解鎖對象）。
- 需重建桌面（Rust 無關，但前端須重新嵌入）方能於安裝版生效。
