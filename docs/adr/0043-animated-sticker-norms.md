# 0043. 自製動態貼圖規範（借鏡 LINE）

- 狀態：已接受
- 日期：2026-07-04
- 相關文件：docs/adr/0031（動態貼圖：宣告式 SVG 動畫）、docs/adr/0032（自製貼圖）、docs/adr/0042（自製貼圖容量與規格限制）

## 背景與問題

自製貼圖（ADR-0032）的驗證器不封鎖 `<style>`/`@keyframes`/SMIL `<animate>`，因此**匯入含 CSS/SMIL 動畫的 SVG 已技術性可行**，經 `<img src=dataURI>` 播放（JS 已停用）。但這是「意外可行」——缺乏無障礙與濫用規範。研究 LINE 動態貼圖後，決定借鏡其「約束」（非其格式）。

## LINE 規範摘要（研究所得）

LINE 用 **APNG（點陣逐幀）**：尺寸 ≤ 320×270、每張 5–20 幀、播放 1/2/3/4 秒、loop ≤ 4 次、總長 ≤ 4 秒、檔案 < 300 KB（硬上限 500 KB）、RGB、背景透明、**首幀即靜止圖**。

## 決策

**不採 LINE 的 APNG 格式**（點陣逐幀在「內容隨訊息扇出」模型下頻寬爆炸，ADR-0027/0042），維持 **SVG 宣告式向量動畫**（ADR-0031）。但**借鏡其規範性約束**，本次落實可被靜態強制的部分：

1. **無障礙護欄（本次強制）**：新增 `withReducedMotionGuard(svg)`，於任何貼圖 SVG 的 `<svg>` 後注入
   `@media(prefers-reduced-motion:reduce){*{animation:none!important;transition:none!important}}`，
   並集中於 `svgToDataUri`（所有貼圖渲染的唯一出口）套用。用 `*` 選擇器不依賴使用者 class；
   已自帶該規則者（內建 `anim()`）不重複注入。→ 自製/匯入動態貼圖**預設尊重系統減動偏好**。
2. **首幀即靜態（借鏡 LINE）**：SVG 天然以初始狀態為靜態預覽；在 reduced-motion 下凍結於初始幀，
   等同 LINE「首幀靜止圖」語意，無需額外機制。
3. **容量上限（沿用 ADR-0042）**：動態貼圖同受 **SVG ≤ 32 KiB + 標籤 ≤ 48 字**約束；因扇出放大，
   維持此上限比 LINE 的 300 KB 更嚴，且 SVG 向量遠比 APNG 省——此為相對 LINE 的結構性優勢。

**列為規範文件（暫不硬強制，靜態分析不可靠）**：
- **時長 ≤ 4 秒、loop 有界**（LINE 準則）：作為作者指引；CSS/SMIL 的實際播放時長與循環次數難由字串靜態精準判定，故不硬擋（內建 motion 包目前為 `infinite` 循環，於 reduced-motion 下已可凍結）。
- **禁止快速閃爍**（光敏性癲癇風險）：列為規範，enforcement 待未來（需更強的動畫語意解析）。

## 理由

- **對症補洞**：先前三缺口（無障礙、濫用、規格）中，無障礙是最明確且可靠強制者，本次以 reduced-motion 護欄一次補平，且集中於單一渲染出口、涵蓋所有場景（訊息、選擇器、貼圖庫、編輯器底圖）。
- **格式選擇正確**：SVG 向量 + 宣告式動畫符合去中心化與扇出省頻寬；APNG 逐幀與之相悖。
- **保守可測**：純函式、字串注入、13 項 sticker-svg 測試（含護欄注入、不重複注入、非 SVG 原樣、data URI 套用）；全套 166 桌面測試綠。

## 後果

- 正面：自製/匯入動態貼圖**預設無障礙**（減動偏好下凍結首幀）；渲染出口集中、規範明確。
- 負面 / 後續：
  - 時長/loop/閃爍上限僅為文件規範，未硬強制；若出現濫用再評估更強的動畫解析或改採「渲染側逐幀取樣檢測」。
  - 編輯器（ADR-0033）仍僅靜態筆劃創作；「app 內製作動畫」為更大題目，另議。
  - `svgToDataUri` 現一律注入護欄；對靜態貼圖為無害的空規則（僅 reduced-motion 下生效）。
