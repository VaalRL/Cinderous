# 0013. 限時訊息（Disappearing / 閱後即焚，M6）

- 狀態：已接受
- 日期：2026-07-01
- 相關文件：docs/adr/0010（M6 路線圖）、docs/adr/0011（Reactions）、docs/adr/0012（Unsend）；ARCHITECTURE.md §5、PRD.md §6

## 背景與問題

M6 第三項功能：限時訊息（閱後即焚）。訊息在指定時間後於雙方客戶端自動消失。需沿用既有 1:1 加密通道與隱私機制，且讓收件端在解密後即可得知何時該讓訊息消失，不依賴中繼站是否仍持有該事件。

## 決策

- **協定**：延伸既有 `wrapMessage`（**Fix-First**，不另開路徑），新增選項 `disappearAt`（unix 秒）。設定時：
  - 於 **rumor 內層** 寫入 NIP-40 `expiration` tag —— 收件端解密後即知到期時間，與中繼是否留存無關。
  - 同步將 **外層 Gift Wrap 的 `expiration`** 縮短為同一時間，讓中繼站提早清除密文，縮小暴露窗口。
  - 核心以純函式 `messageExpiry(rumor)` 讀出到期時間。
- **UI（v1）**：對話視窗提供限時選單（關 / 1 分鐘 / 1 小時 / 1 天）；限時訊息旁顯示 ⏱ 標記；到期後以佔位字「訊息已到期」呈現（與 Unsend 的「訊息已收回」對稱），前端每秒掃描到期。
- **持久化**：`expiresAt`（毫秒）隨訊息存入本機並於重載回放；到期判定由客戶端依 `expiresAt` 重新計算。

## 後果

- 正面：沿用 NIP-17/59 隱私通道與 NIP-40 過期語意；核心可測（`disappearAt` → rumor/外層過期、`messageExpiry` 讀回），端到端經真實 relay 驗證。
- 負面 / 未來（與 Unsend 同類的固有限制）：v1 為客戶端「軟性」到期隱藏，無法阻止對方在到期前快取／截圖；本機仍保留明文至下次覆寫（後續可加到期即清除本機明文）。中繼上的密文於外層過期後由中繼清除，非即時。此為 E2E 加密「閱後即焚」的共通限制。
