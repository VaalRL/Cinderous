# 0028. 前向保密與後妥協安全：維持靜態金鑰、FS/PCS 統一交由未來 MLS（F2）

- 狀態：已接受
- 日期：2026-07-01
- 相關文件：PRD §6/§7；docs/adr/0002（隱私基線）、0007（NIP-44）、0009（多設備）、0027（群組加密）；ROADMAP F2
- 取代：ROADMAP「未決策 ADR：前向保密（F2）」

## 背景與問題

現況 1:1 與群組（ADR-0027）皆採 **NIP-44 靜態 ECDH**：單次私鑰外洩即可解密**全部
歷史與未來**訊息——無前向保密（FS）、無後妥協安全（PCS）。PRD §6 已將此列為「已知
殘餘風險」。本 ADR 決定是否導入棘輪（Double Ratchet）等機制，或維持現狀。

## 關鍵觀察

1. **即時 P2P 已有 PFS**：WebRTC 資料通道走 **DTLS**（每會話臨時金鑰、具前向保密）。
   因此「即時線上」訊息、Nudge、檔案、通話媒體**已具傳輸層 FS**。FS 缺口僅在**經中繼
   的離線 Gift Wrap 留言**。
2. **一致性**：1:1 與群組目前姿態一致（皆靜態）。若只為 1:1 加 Double Ratchet，會與
   群組（ADR-0027 扇出、無 FS）不一致，且需要**兩套**機制（1:1 棘輪 + 群組某方案）。
3. **MLS 可一統**：MLS（RFC 9420）對 **1:1（視為 2 人群組）與多人群組同時**提供
   FS+PCS。ADR-0027 已把 MLS 列為群組的未來升級路徑。若未來需要 FS/PCS，MLS 是
   **單一**、標準化、涵蓋兩者的答案。
4. **成本/稽核**：Double Ratchet 需 X3DH 式預金鑰（prekeys）發佈到中繼、每對/每設備
   會話狀態、與多設備（ADR-0009）及群組的整合——是本專案最大的新密碼子系統，稽核面大；
   且會**破壞 Nostr 靜態金鑰模型**（雖 Buddy 為封閉生態，DM 不需與任意 Nostr 客戶端互通，
   但仍增加複雜度）。

## 決策

- **v1 維持 NIP-44 靜態 ECDH**（1:1 與群組），如 PRD §6 明載為已接受殘餘風險。
- **明確以 WebRTC/DTLS 承擔即時 FS**：高敏感對話建議在雙方上線時走 P2P 即時通道
  （已具 PFS）；產品可於 UI 提示「此為即時加密通道」。此非新工程，僅明確化既有性質。
- **不導入獨立的 Double Ratchet**。若未來需要 FS/PCS（涵蓋離線留言與群組），
  **統一採用 MLS**（與 ADR-0027 的群組升級同一次決策/工程），而非為 1:1 另立棘輪。
- **觸發條件**（升級到 MLS 才重啟本議題）：威脅模型升級（需對抗設備/金鑰事後外洩）、
  第三方稽核建議（F4）、或明確的高敏感使用者需求。

### 為何不現在做

FS/PCS 的完整價值需**同時**覆蓋離線留言與群組，且與多設備金鑰管理深度耦合；分階段半套
（只做 1:1 棘輪）會留下群組缺口並製造兩套機制。以本專案規模與未稽核現況，現在導入的
複雜度/稽核成本，遠高於「靜態金鑰 + 即時走 DTLS」所留風險（且該風險已誠實揭露）。

## 後果

- 正面：維持精簡、Nostr 相容、與群組一致的加密姿態；即時通道本已具 FS；未來升級路徑
  單一明確（MLS 一統 1:1＋群組），避免疊床架屋。
- 負面 / 未來：經中繼的**離線留言仍無 FS/PCS**——私鑰外洩可解其歷史。緩解＝WebRTC 即時
  通道、以及未來 MLS。需在 PRD §6/§7 與 README 持續誠實揭露；F4 安全自評應把「離線留言
  無 FS」列為明確風險項。多設備下的 MLS 群組金鑰管理待 MLS 落地時一併設計（ADR-0009 延伸）。
