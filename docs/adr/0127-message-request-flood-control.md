# 0127. 訊息請求防洪——數量上限（FIFO 逐出）＋全部刪除

- 狀態：已接受（已實作）
- 日期：2026-07-15
- 相關文件：**ADR-0121（訊息請求：陌生人不進聯絡人清單）**、0111（封存）、0123（中繼具名訂閱）

## 背景與問題

ADR-0121 把陌生人的訊息導進「訊息請求區」，並在該 ADR 的已知限制裡明記：

> 請求區本身可以被灌爆（大量陌生 pubkey 各傳一則）。目前沒有數量上限、沒有速率限制、
> 沒有「全部刪除」。**這是下一步該做的。**

`ensureKnown()` 每收到一個新 pubkey 就 `addRequest`，**無界成長**。攻擊者用一萬個一次性
pubkey 各發一則，就能把請求區與本機儲存（每個請求的對話都落地）撐爆。

## 決策

### 1. 請求區**數量上限**，超過就 FIFO 逐出最舊

```ts
const MAX_REQUESTS = 100;
// ensureKnown()：加入後
while (this.requests.length > MAX_REQUESTS) {
  const oldest = this.requests[0];
  this.storage.removeRequest(oldest.pubkey);
  this.storage.removeContact(oldest.pubkey); // 連同其訊息／封存一起清（走 decline 的清理路徑）
  this.requests = this.storage.loadRequests();
}
```

**逐出要連訊息一起清**——否則清單封了頂，儲存仍在漲。取 100：真的同時有 100 個陌生人在等你
確認，那已經是被灌爆了；此時逐出最舊是合理的，讓它無界成長才是問題。

### 2.「全部刪除」——被灌爆時的一鍵出路

```ts
clearRequests(): void {
  for (const r of this.requests) {
    this.storage.removeRequest(r.pubkey);
    this.storage.removeContact(r.pubkey); // 訊息／封存一起清
  }
  …emitRequests();
}
```

桌面與行動端的請求區標頭都加「全部刪除」（**多於一筆才顯示**——只有一筆時單項刪除即可）。
**不封鎖**：被清掉的人還能再傳（會變成新的請求）——「全部刪除」是清雜訊，不是封殺。

### 3. 為什麼**不做**時間窗速率限制

計畫原本列了「速率限制」。實作時判斷它對這個威脅**加不了什麼**：

- 訊息請求**不通知、不點亮徽章、不能 nudge**（ADR-0121）——所以高速率本身**不消耗使用者的
  注意力**，不像一般 App 的推播轟炸。
- 剩下的唯一危害是「清單與儲存無界成長」，而那由**硬上限**（決策 1）**直接封住最壞情況**，
  與到達速率無關。
- 而時間窗速率限制需要一個**時鐘**。儲存層刻意不依賴時鐘（`Date.now()` 難測、跨裝置不可信）；
  為一個硬上限已涵蓋的威脅引入時鐘，得不償失。

所以：**硬上限＝速率限制的上位替代**。這是刻意的取捨，記在案。

## 理由

- 這是 ADR-0121 明講「下一步該做」的收尾。ADR-0121 把陌生人擋在注意力管道之外，
  但沒擋住「儲存被撐爆」——本 ADR 補上。
- FIFO 逐出有一個代價（見已知限制）：**理論上攻擊者可用洪水把某個真實請求擠出去**。但要湊到
  100 個並發陌生請求本身就是明顯的攻擊訊號，而使用者的自然反應是「全部刪除」——之後佇列清空、
  新的真實請求照進。所以這個代價在實務上被「全部刪除」吸收掉。

## 後果

- 正面：
  - 請求區與其儲存**有界**（≤ 100 個對話），不再能被無界灌爆。
  - 被灌爆時有一鍵「全部刪除」（連訊息／封存一起清；不封鎖）。
  - 測試 1014 → **1023**（engine +4：上限 FIFO 逐出、逐出連訊息清、全部刪除清空、
    全部刪除不封鎖；desktop UI +3、mobile UI +2：多於一筆才顯示「全部刪除」）。

- 已知限制：
  - **FIFO 逐出理論上可被用來擠掉舊的真實請求**（見理由）。緩解：湊滿 100 本身是明顯攻擊訊號，
    且「全部刪除」是自然出路。若要更強，可改「達上限即拒收新請求」，但那會讓真實的新請求在
    佇列滿時被靜默丟棄——兩害相權，選 FIFO。
  - **訊息仍會抵達中繼**（Nostr 擋不掉，ADR-0121 已記）——本 ADR 是客戶端的顯示層與儲存防禦。
  - 上限 100 是固定值，未開放設定（目前沒有理由讓使用者調它）。
