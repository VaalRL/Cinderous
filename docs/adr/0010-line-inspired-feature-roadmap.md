# 0010. 借鏡 LINE 的功能路線圖與範疇界線（M6–M9）

- 狀態：已接受
- 日期：2026-06-30
- 相關文件：PRD.md §12；ARCHITECTURE.md §5、§7、§8；docs/adr/0002、0008

## 背景與問題

MSN 經典體驗（M1–M5）之外，需評估現代即時通（以 LINE 為對照）的熱門功能，決定哪些納入路線圖、哪些明確排除，並確保新功能不違背本專案的隱私、去中心化與零中央資料庫原則。

## 決策

**納入路線圖（M6–M9）**，一律沿用既有隱私機制（NIP-44 加密 + NIP-17/59 Gift Wrap 隱藏元資料；WebRTC 走 P2P）：

- **M6 訊息互動**：訊息回應（NIP-25 kind 7）、收回訊息（NIP-09 kind 5）、限時訊息（NIP-40 較短過期）——皆以 Gift Wrap 包封。
- **M7 富媒體**：語音訊息與圖片/媒體相簿（複用 WebRTC 檔案分塊傳輸 + 本機 SQLite 媒體庫）、貼圖包（內建包以 `pack/id` 參照，客戶端渲染）。
- **M8 語音/視訊通話**：WebRTC media track；通話信令（offer/answer/candidate/hangup）複用既有 SDP 信令通道（NIP-59 包封的 ephemeral 21000-21999）。媒體全程 P2P、不經中繼。
- **M9 聯絡人與群組**：QR 加好友（交換 `npub` + 雙向同意流程）；群組聊天（需群組金鑰管理，見待決議）。

每項功能在實作時各自新增一份細節 ADR（事件契約、UI、隱私取捨）。

**明確排除（與本專案精神衝突）**：

- **LINE Pay／錢包／點數**：非金融 App，超出範疇。
- **官方帳號／企業帳號／廣告／貼圖商店金流**：中心化營利模式，違背去中心化與 AGPL 精神。
- **集中式雲端聊天備份**：違反「明文不上雲、設備全毀即帳號死亡」的核心設計（多設備同步 M4 已是折衷）。
- **Open Chat 公開大群／VOOM 社群動態牆**：會將產品從私密即時通拉成社群平台，範疇與定位不符。

## 理由

這些功能多能複用既有資產（WebRTC、檔案傳輸、Gift Wrap、NIP-40），落地成本低且不破壞隱私模型；排除項則會引入中心化、金流或明文上雲，與核心價值抵觸。

## 後果

- 正面：路線圖明確、範疇有界；新功能沿用同一套隱私機制。
- 負面 / 待決議：**群組聊天的群組加密方案（MLS vs sender-key）與多設備群組金鑰管理**尚未定案（另立 ADR）；語音訊息離線傳遞受中繼大小限制，需設計退回策略。
- 後續行動：更新 PRD §12 與 ARCHITECTURE §5/§7/§8；各功能實作時補細節 ADR。
