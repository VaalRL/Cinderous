# 0002. 隱私元資料保護與 Nostr 協定基線（NIP-44/17/59/42/13）

- 狀態：已接受
- 日期：2026-06-29
- 相關文件：PRD.md §5–§9、ARCHITECTURE.md §5（事件契約，待同步更新）

## 背景與問題

原始 PRD 以 Nostr `kind 4`（NIP-04）作為離線留言加密，並宣稱「中繼站只看得到密文」。但：

1. NIP-04 已被官方棄用，會洩漏訊息長度且缺乏現代 AEAD。
2. Nostr 事件的 `pubkey`（寄件者）與 `p` tag（收件者）預設為明文，使中繼站可重建完整社交圖譜——與「極致隱私」核心承諾直接矛盾。
3. 「Kind 4/44」標示混淆了「kind 編號」與「NIP-44 加密方案」。
4. 開放中繼站無存取控制與抗濫用，免費額度易被灌爆。

## 考量的選項

- 維持 NIP-04 + 明文元資料（最簡單，但違背隱私承諾）。
- 改用 NIP-44 內容加密，但保留明文收發欄位（解決內容，未解決社交圖譜）。
- 採 NIP-44 + NIP-17/59 Gift Wrap 隱藏收發雙方，並加上 NIP-42/13 防護。

## 決策

採第三案，定為協定基線：

- **內容加密：** NIP-44（取代 NIP-04）。
- **元資料隱藏：** 私訊用 NIP-17（kind 14 → kind 13 seal → kind 1059 Gift Wrap）。
- **中繼站防護：** NIP-42 AUTH + NIP-13 PoW + 每 pubkey 速率/大小上限。
- **Ephemeral kind 釐清：** 上線 20000、輸入中 20001、音樂 20002、SDP 信令 21000-21999（Gift Wrap 包封）。

## 理由

唯有隱藏收發雙方，社交圖譜才不外洩，「極致隱私」方能成立；NIP-44 為現行安全加密；NIP-42/13 支撐「永久免費且不被灌爆」。

## 後果

- 正面：內容與社交圖譜對中繼站皆不可見；中繼站具基本抗濫用能力。
- 負面 / 已知殘餘風險：上線心跳仍洩漏概略時序；靜態 ECDH 無前向保密（是否導入棘輪另立 ADR）；Gift Wrap 增加封裝運算與事件體積。
- 後續行動：同步更新 `ARCHITECTURE.md` §5 事件契約；於 M1 前完成 Ephemeral 心跳的 Worker 請求容量估算（另立 ADR）。
