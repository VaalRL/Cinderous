# 0072. 桌面配對克隆（D4a：一次性 P2P 全量搬家）

- 狀態：已接受
- 日期：2026-07-10
- 相關文件：docs/adr/0009（多設備同步——本 ADR 為其 D4 的桌面先行子集，並補其
  「互認」待辦）、0017/0029（WebRTC 通道與二進位分塊）、0045（多身分）、
  0048（企業政策開關）、0053（金鑰庫）、0055（隱藏身分）、0067（本地密碼／nsec 定位）、
  0070（備份碼）、0071（雲端快照）、packages/core/src/pairing.ts、
  apps/desktop/src/storage/types.ts（StorageSnapshot）

## 背景與問題

1. 換機還原已有兩條路（0070 備份碼＝身分＋relay；0071 雲端快照＝聯絡人/群組/封鎖/
   近期訊息），但仍缺：**完整歷史**（快照僅近期 500 則/180KB）、回應/收回/群組標籤/
   貼圖收藏等全量狀態，以及一條**內容完全不經 relay** 的搬家路（雲端快照關閉者的唯一全量選項）。
2. 地基已存在：`pairing.ts`（QR 載荷＝一次性 AES-256 金鑰＋AEAD 捆包）、
   `raceConnections`、A4 的 WebRTC 資料通道與分塊、`StorageSnapshot`（B2 全量狀態形狀）。
3. 兩個桌面特有的問題：（a）配對時**新裝置尚無身分**，不能以正式 pubkey 做 Nostr
   信令，同身分雙機互發信令也會自迴圈；（b）桌面無相機，「掃描」以**貼上字串**替代
   ——但 Windows 雲端剪貼簿/剪貼簿歷史會留存載荷（含一次性金鑰），有時間窗竊取風險。

## 考量的選項

- 傳輸：
  - 選項 A：**WebRTC＋拋棄式信令金鑰**（採用）——兩端各產僅用於 SDP 交換的一次性
    keypair（房間號＝載荷 room），資料走 P2P 通道；跨網路可用、重用 A4 已驗證機械、
    不需監聽埠；relay 只見兩把與身分不可連結的拋棄 pubkey。
  - 選項 B：LAN 直連（Tauri 開 TCP 監聽）——同網段最快，但需原生監聽器且瀏覽器
    不可用；留作後續 Happy Eyeballs 加速項，非 v1。
  - 選項 C：全量走雲端快照——否決：NIP-44 單包 64KB 限制、且違背「不經 relay」的定位。
- 互認：無互認（僅憑載荷）——否決（剪貼簿竊取窗）；**SAS 短碼確認**（採用）。

## 決策

1. **傳輸**：WebRTC 資料通道＋拋棄式信令金鑰（選項 A）；傳輸層抽象為
   `PairTransport`（雙工位元組流），記憶體對接供測試、WebRTC 為產線實作。
2. **互認（SAS）**：連線建立後兩端自握手雜湊導出 4 位短碼，**舊裝置顯示並由使用者
   確認相符後才送捆包**；配對載荷短時效（約 2 分鐘）＋房間一次性（首連即毀）。
   三件套合力封殺剪貼簿殘留載荷的利用窗。
3. **方向與 UX**：授權動作在資料持有方——舊裝置「設定→配對新裝置」產生載荷
   （QR 圖＋可複製字串）並等待 SAS 確認；新裝置「SignIn→從舊裝置匯入」貼上字串
   發起連線；收包後寫入（nsec→金鑰庫、`StorageSnapshot`→儲存、profile 登錄）→ 重載。
4. **範圍**：一次克隆**單一作用中身分**（多身分重複操作；隱藏身分不被順手帶走）；
   **企業身分 v1 排除**（多裝置克隆屬組織政策，未來由名冊政策 `allowPairing` 決定）；
   **本地密碼不克隆**（各裝置獨立包裹，ADR-0067）；裝置偏好（主題色等）不帶。
5. **捆包**：`StorageSnapshot` 全量（身分、聯絡人、全部訊息、回應、收回標記、群組、
   引導清單）以載荷的一次性金鑰 AEAD 加密（`encryptBundle`），走分塊通道（MB 級）。
6. **邊界**：D4a 只做一次性克隆；克隆後兩機各自獨立，持續收斂靠既有機制
   （7 天窗雙收＋0071 雲端快照，模式隨快照自動接續）；**即時 delta 通道留給 D4b**（另立 ADR）。

## 理由

- 拋棄式信令解掉「無身分不能信令／同身分自迴圈」兩個結構問題，且不增加元資料暴露。
- SAS 把「載荷即權限」升級為「載荷＋人在現場確認」，成本一個確認框，收益是封掉
  桌面貼上流程引入的最大攻擊面。
- 捆包直接採用既有 `StorageSnapshot`＝零新格式；傳輸重用 A4 分塊＝零新原語。

## 後果

- 正面：全量搬家不經 relay；完整歷史與全量狀態可攜；補齊 ADR-0009 的互認待辦。
- 負面／已知殘餘風險：剪貼簿仍會殘留已失效載荷（時效與一次性房間使其無利用價值，
  文案建議用後清剪貼簿）；企業身分暫無多裝置克隆路徑；兩機需同時在線（本設計的
  本質，離線橋接屬 0071）。
- 後續行動 / 待辦：依 ROADMAP D4a 施工（core 配對協定＋SAS → 傳輸抽象與 WebRTC
  實作 → 兩端 UI → E2E）；完成後更新 ARCHITECTURE.md；D4b（delta 通道）另立 ADR。
