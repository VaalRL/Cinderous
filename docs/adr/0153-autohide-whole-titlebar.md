# 0153. 平時隱藏＝整條標題列滑出（頂端熱區喚回；修正 0151 的 autoHide 語意）

- 狀態：已接受（已實作）
- 日期：2026-07-16
- 相關文件：**ADR-0151（原 autoHide＝只藏按鈕，本 ADR 升級其語意）**、0152（pointer 拖曳）、
  0150（自繪外框）

## 背景與問題

ADR-0151 的「平時隱藏」只把**按鈕帶**設為 opacity 0，標題列本體（32px 高的條）仍佔著版面。
使用者補充：啟用時應該是**整個視窗自繪外框都隱藏**——內容拿回整個視窗，滑鼠碰到才出現。

## 決策

`autoHide` 開啟時（設定鍵/模型不變，只改行為）：

- 標題列改為 **fixed 覆蓋層**（z-index 100，蓋過 app 內最高的通話遮罩 70），平時
  `translateY(-100%)` 滑出畫面頂端；**視窗頂端 6px 熱區**（z-index 101）hover、或標題列
  本身 hover/focus-within 時滑入（0.15s；`prefers-reduced-motion` 下關動畫）。
- 內容區拿回整個視窗：`.window-chrome--autohide` 把 `--viewport-h` 覆寫回 `100vh`
  （0150 的版面變數機制直接受益，無需動任何版面）。
- 外框本體抽成 `ChromeFrame`（`WindowChrome` 在 Tauri 下渲染它）——SSR 可直接測
  autoHide 殼層類與熱區，不再卡在 `isTauri()` 分流之後。
- `TitleBar` 不再感知 autoHide（0151 的 `titlebar__controls--autohide` 移除）；隱藏是殼層
  （ChromeFrame）的責任。

## 後果

- 正面：極簡模式名副其實——外框整條讓位給內容；喚回手勢直覺（滑到頂端）；拖曳移動、
  雙擊最大化在滑入後照常可用。
- 已知限制／取捨：
  - 標題列隱藏期間無法拖動視窗（必先滑到頂端喚回）——這是「整條隱藏」的固有代價。
  - 頂端 6px 熱區永遠吃 hover：貼齊頂端的 app 內元素在那 6px 內點不到（實務上該區只有
    標題列自己的位置，無既有互動元素）。
  - 滑入以 hover 驅動：觸控裝置等同一直隱藏（桌面版無此情境，僅記錄，同 0151）。
