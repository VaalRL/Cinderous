# 0131. 早到的群訊不該被丟掉——未知群組的訊息緩存、加入後重放

- 狀態：已接受（已實作）
- 日期：2026-07-15
- 相關文件：**ADR-0107（早到的回條：緩存＋重放，本 ADR 的樣板）**、0027（群組扇出）、
  0124（群組傳檔）、0119（健檢：本問題於該次記錄）

## 背景與問題

`receiveGroup()` 收到指向**本地還不存在的群組**的訊息時，**靜默丟棄**：

```ts
const g = this.groups.find((gr) => gr.id === groupId);
if (!g) return; // 未知群組（尚未被加入）
```

而群訊與「群組建立控制」是**兩個獨立的 Gift Wrap 事件**。NIP-59 的 `jitteredPast()` 會把外層
`created_at` 隨機往前推最多 2 天（隱私設計）——**所以中繼的重播順序是隨機的**。於是「你被加入
群組那一刻的頭幾則訊息」很可能**比 group-create 控制先到**：

```
T+0.0s  收到 kind 1059 → 內層是群訊（g=某群）    → 本地沒這個群 → **丟棄**
T+2.3s  收到 kind 1059 → 內層是 group-create      → 群組實例化
```

那幾則先到的訊息**永遠不會回來**——群訊是 ephemeral 之外的 Gift Wrap，中繼只留 7 天且不重送，
而客戶端這端直接丟了。**使用者一被加進群組，就漏掉了開頭幾則訊息，且毫無跡象。**

## 決策

### 1. 未知群組的訊息**緩存**，不丟棄（比照 ADR-0107 的早到回條）

`if (!g)` 時改為 `deferGroupMsg(groupId, sender, rumor)`——把 `{sender, rumor}` 存進一個
**有界**的 `Map<groupId, …[]>`，而不是丟掉。這與 ADR-0107 的 `deferReceipt`（回條比目標訊息先到
→ 緩存 → 目標到了再套用）是同一個形狀。

### 2. 群組實例化時**重放**緩存

`applyControl()` 走 group-create／group-snapshot 分支、`saveGroup` 讓群組在本地存在後，
`drainPendingGroup(id)`：取出該群的緩存、依**送出時間**排序、逐一重新過 `receiveGroup`
（此時群組已存在 → 正常入庫、emit、未讀）。先刪緩存再重放，避免重放時又 defer 回去。

### 3. 有界，且去重

- **每群上限 64 則、總群數上限 32**（FIFO 逐出最舊）——防止惡意寄件人用大量假 `g` tag 撐爆記憶體
  （那些假群永遠不會有 group-create → 緩存自然被逐出，從不重放、從不入庫）。
- **以 rumor.id 去重**：跨中繼的重複副本（外層 id 不同、內層 rumor.id 相同）不會重複緩存，
  避免重放時 `bumpUnread` 重複計數。

## 理由

- 這與 ADR-0107 完全同源：**分散式系統裡「相關的兩個事件」不保證按序抵達**，而 NIP-59 的 jitter
  讓它更不保證。凡是「B 依賴 A 先到」的地方，都需要「A 沒到就緩存 B、A 到了再重放」。ADR-0107 對
  回條做了，本 ADR 對群訊做——同一個缺一不可的模式。
- 有界緩存 ＋ 只重放「真的被建立的群」＝ 惡意假群訊只會佔一小段有界記憶體後被逐出，**永不處理**。

## 後果

- 正面：
  - **被加入群組時不再漏掉開頭幾則訊息**（早到的群訊會在群組實例化後補進來，依時間排序）。
  - 惡意假 `g` tag 撐不爆記憶體（有界 ＋ 去重 ＋ 只重放真群）。
  - 測試 1045 → **1048**（engine +3：群訊比 group-create 先到仍收得到、假群訊被逐出不入庫、
    跨中繼重複不重複計未讀）。

- 已知限制：
  - 緩存是**記憶體內、不落盤**——重放前重開 App 會丟掉緩存（極短窗口內的邊界，且對方仍在 7 天
    TTL 內，換機/雲端快照另有其路）。
  - 上限外的極端洪流仍會逐出最舊——但那已是「被 Gift Wrap 灌爆」的情境，與一般使用無關。
