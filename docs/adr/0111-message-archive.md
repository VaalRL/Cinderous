# 0111. 訊息封存（冷熱分離）：把 N 綁住，並修好 ADR-0094 的資料遺失

- 狀態：已接受（已實作）
- 日期：2026-07-14
- 相關文件：ADR-0054（加密儲存基質）、0072（配對搬家）、0094（保留上限／明文導出）、
  0102（縮圖）、0110（長對話熱路徑）

## 背景與問題

ADR-0110 把每一條熱路徑對「歷史長度」的**常數**壓了下來（已讀水位 3,531ms → 18ms、
建置 10 萬則 逾時 → 68ms）。但它**沒有把 N 綁住**——歷史長度本身仍然無上界：

- **LocalStorage 配額 5–10MB** → 約 5 萬則後撞牆，而 `write()` 只 `console.warn`
  → **靜默丟失寫入**（訊息在記憶體看得到，重載就沒了）。
- 記憶體占用與單次序列化成本仍隨總量線性成長。

而 **ADR-0094 的保留上限是 `list.splice()`——直接刪掉舊訊息**（永久遺失）。
預設雖是「無上限」（不刪），但一旦使用者為了避開配額而設上限，代價就是**真的失去歷史**。

## 決策

**冷熱分離。** 熱區（`AppStorage`，同步）只保留最近 **`HOT_CAP = 5,000`** 則；更舊的
**移入封存**（非同步、分塊 **`ARCHIVE_CHUNK = 1,000`**、獨立基質）。主視窗只讀熱區
→ **每一條熱路徑的成本被結構性地綁住**，與總歷史長度無關。

**這同時把 ADR-0094 的「刪除」改成「封存」——資料不再遺失。**

### 1. 封存不能放 localStorage——這是會讓設計直接失敗的陷阱

localStorage 的配額是**每個 origin 的總量**。把舊訊息從 `msgs.bob` 搬到 `archive.bob`
**完全不解決問題**——同一個額度。**封存必須換基質。**

### 2. 桌面走**加密檔案**——而且這是唯一不會讓安全性倒退的選擇

桌面的儲存**本來就是加密的**（`encstore`，AES-256-GCM，金鑰在 OS 金鑰庫，ADR-0054/0067）。

**封存若改走 IndexedDB 會是明文**——那是靜態加密的**默默降級**。

走檔案則直接沿用同一把 db 金鑰與同一套加密，**零新機制**：ADR-0110 剛加的
`store_save_part` 已經在寫加密檔，封存只是多幾個塊檔。

- 落地：`<store>/<ns>/archive/<convo>.<seq>.enc`
- 新增 Rust 命令：`archive_append` / `archive_count` / `archive_load` / `archive_remove`
- `archive/` 是 `<ns>/` 下的**子目錄**，而 `store_load_parts` 只收 `.enc` **檔案**
  （目錄無副檔名 → 自然跳過）——**開機不會把封存一起載入**，這正是重點。
- 沿用 ADR-0110 的部位名白名單（防路徑穿越）；單一塊毀損不拖垮其餘塊。

### 3. 行動／網頁走 **OPFS**

webview 裡**沒有**檔案系統（`react-native-web` 跑在 DOM 上，不能寫任意檔案）。
**OPFS（Origin Private File System）** 是最接近「檔案」的東西：真的檔案（有 file handle、
可分塊讀寫），只是沙箱化、使用者看不到。

關鍵：**OPFS 的配額與 localStorage 是完全不同的池子**——localStorage 是 5–10MB 的小額度，
OPFS 走 Storage API 配額（通常是可用磁碟的一大部分）。

- **加密**：OPFS 是明文。這與本平台**現況一致**（localStorage 本來就是明文），
  故不構成新的暴露面；但這是既有的不一致，應另立 ADR 處理。
- 移植到真 RN 時換掉 `opfs-archive.ts` 即可（`expo-file-system`），介面不變
  ——沿用專案既有的平台縫模式（`native/files.ts` vs `native/save-file.ts`）。

### 4. 搬移順序是**正確性紅線**：先寫封存 → 成功後才裁切熱區

```ts
await this.archive.append(convo, chunk);   // ← 先寫封存
this.mem.trimOldest(convo, chunk.length);  // ← 成功後才裁切
```

若中間當機，訊息會**同時**存在於封存與熱區（重複）——讀取端以 id 去重即可。
**寧可重複，絕不遺失。** 反過來做（先裁切再寫封存）一旦當機就是**永久遺失**。

**封存寫入失敗 → 熱區不裁切**（訊息仍在，只是熱區暫時超過上限）。已有測試釘住。

### 5. **沒有封存就不裁切**

`attachArchive()` 未被呼叫時（OPFS 不支援、私密模式、配額拒絕），熱區**完全不裁切**
——退回舊行為（無上限），但**資料完好**。

`openOpfsArchive()` 在不支援時回 **`null`，而不是一個什麼都不做的替身**：
若封存是靜默的 no-op，熱區仍會被裁切 → **永久遺失**。這是刻意的安全設計。

### 6. 同步／非同步的邊界（本設計最漂亮的一點）

`AppStorage` 是**同步**介面；封存是**非同步**的。兩者不衝突，因為**封存只被兩個地方讀**：

- **歷史紀錄 UI**（`HistoryWindow`）——分頁，一次載一塊
- **匯出**（ADR-0094）

兩者**天生可以非同步**。主視窗（熱區）完全不碰封存 → **零改動、零延遲**。

### 7. 匯出**必須**含封存——正確性紅線

`exportRecords()` 原本只讀 `storage.loadMessages()`（熱區）→ 會**靜默漏掉所有被封存的舊訊息**
——那正是使用者最想留下來的部分。故改為**非同步**並併入封存（以 id 去重）。

### 8. 刪好友／封鎖／退群 → **封存也要清**

否則會出現「刪了好友，歷史卻還在」。

## 理由

- ADR-0110 解的是**常數**，本 ADR 解的是 **N**。兩者都需要——但只有本 ADR 能讓成本
  **與總歷史長度脫鉤**。
- 「封存而非刪除」不只是新功能，它是把 **ADR-0094 原本就該有的語意**補上。
- 檔案基質在桌面是**免費**的（機制已存在）**且是唯一不降級加密的選擇**。

## 後果

- 正面：
  - 熱路徑成本**與總歷史長度無關**（熱區固定 5,000 則）；記憶體亦被綁住。
  - LocalStorage 的 5–10MB 配額不再是天花板（封存另走 OPFS 池子）。
  - **ADR-0094 的資料遺失被修好**：超出上限的訊息**被封存，不再被刪**。
  - 桌面封存**加密落地**（沿用 AES-256-GCM，零新機制）。
  - 測試 +11（含「封存失敗絕不裁切」「未掛封存絕不裁切」「匯出必含封存」等不變式）。
    全 repo **859 通過**（＋13 Rust）。

- 負面 / 已知限制：
  - **行動端尚無歷史紀錄 UI**（桌面已有 `HistoryWindow`）。行動端的封存**會照常寫入**
    （資料不遺失），但**目前讀不到**——這是明確的缺口，應盡快補上。
  - **OPFS 是明文**（與該平台的 localStorage 現況一致，非新增暴露面）。
  - **配對搬家（ADR-0072）只搬熱區**：`StorageSnapshot` 不含封存——把幾百 MB 塞進 P2P
    信封不可行。封存留在舊機；使用者若要帶走，走 ADR-0094 的明文匯出。
  - **身分輪替（`remapContact`）不會改寫封存**裡的 `contact`／`sender`。封存是唯讀歷史，
    影響有限，但這是已知的不一致。
  - 加密檔**不是可攜備份**：金鑰在 OS 金鑰庫，把 `store/` 複製到另一台機器**打不開**。
    換機請走 ADR-0072；要看得懂的紀錄請走 ADR-0094 明文匯出。
