# 0120. typing／nudge 不得廣播「已簽章的社交圖譜邊」——以 NIP-59 封裝

- 狀態：已接受（已實作）
- 日期：2026-07-14
- 相關文件：**ADR-0119（全面健檢，本問題於該次發現）**、0004（NIP-17/44/59 Gift Wrap）、
  0025（WebRTC 信令以 NIP-59 封裝——本 ADR 的樣板）、0109（自適應心跳）

## 背景與問題

健檢（ADR-0119）發現：「正在輸入中」與「敲一下」是**用真實金鑰簽名、收件人明文寫在 tag 裡**
的廣播事件。

```ts
createEphemeralEvent(sk, KIND.TYPING, { tags: [["p", recipientPk]] })
//                   ↑ 真的私鑰簽名              ↑ 收件人 pubkey，明文
```

中繼站看到的是：

```json
{ "kind": 20001, "pubkey": "<Alice 的真實 pubkey>", "tags": [["p", "<Bob 的 pubkey>"]], "sig": "…" }
```

**這是一條有簽章、有時間戳、有方向的社交圖譜邊。** 而且它是不可否認的——簽章由 Alice 的
金鑰產生，Alice 沒辦法事後說「那不是我發的」。

### 🔴 真正致命的不是圖譜，是它**反推得出 Gift Wrap 的寄件人**

NIP-59（Gift Wrap）存在的**唯一**理由，是讓中繼站看不出「誰寄了這則訊息給誰」：外層
wrap 由**一次性的臨時金鑰**簽名，只有 `["p", 收件人]` 是明文。中繼知道「有人寄東西給 Bob」，
但不知道是誰。

而 typing 把這件事直接拆穿：

| 時間 | 中繼站看到的事件 | 中繼站學到 |
|---|---|---|
| T+0.0s | `kind 20001`，**author = Alice**，`p = Bob` | Alice 正在對 Bob 打字 |
| T+2.3s | `kind 1059`，author = 臨時金鑰，`p = Bob` | 有人寄了一則加密訊息給 Bob |

**兩者關聯起來，那個「有人」就是 Alice。** 打字幾秒後送出訊息是人類的必然行為，這個時間
相關性極強、幾乎不會誤判。

**我們花了整套 NIP-59 去隱藏寄件人，然後在它前面兩秒鐘，用真名廣播了同一件事。**

### 樣板早就在專案裡

`signaling.ts`（ADR-0025，WebRTC 信令）**已經**是正確做法——它同樣是 ephemeral kind，
但用 `sealAndWrap`：

```ts
return sealAndWrap(
  { kind: SDP_SIGNAL_KIND, created_at: nowSec, tags: [], content: JSON.stringify(signal) },
  senderSk, recipientPk,
  { kind: SDP_SIGNAL_KIND, tags: [["p", recipientPk]] },
);
```

那時我們就想清楚了「中繼不該知道誰在呼叫誰」。**typing／nudge 只是漏掉了。**

## 決策

### 1. typing 與 nudge 一律以 NIP-59 封裝（sealAndWrap）

外層仍是同一個 ephemeral kind（20001 / 20100 → 中繼不落盤），但：

- **外層由一次性臨時金鑰簽名** → 中繼看不到寄件人。
- 真實寄件人在 **seal** 裡（以 NIP-44 加密給收件人）→ 只有收件人解得開。
- `["p", 收件人]` 仍為明文——**與 kind 1059 完全一致**。中繼必須知道要轉發給誰。

於是中繼看到的 typing 事件，和它看到的 Gift Wrap 訊息事件**長得一模一樣**（臨時作者、
指名收件人）。時間相關性還在，但它關聯的是「兩個都不知道寄件人的事件」——**沒有真名可以錨定**。

### 2. 訂閱端 `authors:` 過濾器**必須拿掉**，改為只靠 `#p`

封裝後外層作者是臨時金鑰，`authors: [聯絡人們]` 永遠不會命中。改成
`{ kinds: [TYPING], "#p": [me] }`——與 Gift Wrap 收件箱同一種形狀。

順帶一提：**這也修掉了 ADR-0119 提到的「訂閱過濾器本身洩漏社交圖譜」的一半**——
過去 `authors: [聯絡人們]` 這個 REQ 等於把整份聯絡人清單交給中繼。現在 typing/nudge 的
訂閱不再帶名單。（心跳的 `authors:` 仍在——見「已知限制」。）

### 3. 🔴 拿掉 `authors:` 之後，**必須在程式碼裡補上「只收聯絡人」的把關**

過去是 `authors: [聯絡人們]` 這個過濾器在擋陌生人。拿掉它以後，**任何人都能對你發 nudge**
——而 nudge 會**震動裝置、跳通知**。那是現成的騷擾管道。

所以解開封裝後，一律檢查：

```ts
if (this.isBlocked(sender)) return;                              // 封鎖者
if (!this.contacts.some((c) => c.pubkey === sender)) return;     // 陌生人
```

**這條把關比封裝本身更容易被漏掉**（因為漏了不會有任何錯誤，只是變得可被騷擾），所以有測試釘住。

寫測試時發現一件相關的事：`receiveDm()` 對**每一則**收到的 gift wrap 都會 `ensureContact(sender)`
——也就是說**任何人只要先傳一則訊息給你，就自動變成你的聯絡人**（ADR-0061 的自動建檔，
「像簡訊一樣，誰傳給你誰就進清單」）。所以這道把關擋得住「掃到 pubkey 就開炮」的騷擾者，
擋不住「先傳一則訊息再狂 nudge」的。

**但那是既有行為，不是本 ADR 引入的**（封裝前，對方一送訊息就進了 `authors:` 名單，
同樣可以 nudge）。要不要讓陌生人能私訊你，是產品層級的決策——留待專門的 ADR。

### 4. `NUDGE_KIND` 收進 core 的 `KIND` 常數（SSOT）

它原本在 `relay-backend.ts` 與 `browser-backend.ts` **各定義一次** `const NUDGE_KIND = 20100`，
而其他所有 kind 都在 `core/constants.ts`。趁這次改動收斂。

### 5. 舊版明文格式：**只收不發**（過渡期）

版本落差是必然的（web 一更新就生效，桌面 App 得等使用者裝）。若新版完全不收舊格式，
使用者會看到「打字指示燈壞了幾週」。

- **送出**：一律封裝，**永不再發明文**。（洩漏發生在送出端——這裡守住，就守住了自己。）
- **接收**：解封裝失敗 → 退回舊版明文格式（`event.pubkey`），但**同樣要過第 3 條的把關**。

接收舊格式不會讓自己洩漏任何東西；它只是讓還沒更新的對方能繼續顯示打字狀態。
**所有客戶端更新後即可移除**（標記為過渡）。

## 理由

- 這個 bug 的本質是**「安全機制被自己旁邊的功能繞過」**。Gift Wrap 本身沒有錯，
  NIP-59 也實作正確——但只要旁邊有一個用真名、指名、帶時間戳的廣播，整套匿名性就歸零。
  **隱私不是單點屬性，是系統屬性**：只要有一條路徑洩漏，其餘路徑的努力都白費。
- 而正確做法早就在 `signaling.ts` 裡了。這不是「我們不知道該怎麼做」，是**沒有把已知的做法
  套用到所有同類事件上**。所以本 ADR 的真正決策是：**凡是「指名某人」的 ephemeral 事件，
  一律封裝，沒有例外。**

## 後果

- 正面：
  - 中繼站再也拿不到「Alice → Bob，T 時刻」這種**已簽章、不可否認**的有向邊。
  - **Gift Wrap 的寄件人匿名性回來了**——不再有真名事件可以拿來做時間相關性錨定。
  - typing/nudge 的訂閱不再把聯絡人清單交給中繼。
  - 惡意中繼／任何訂閱者對 typing、nudge 的收割一併失效（外層只剩臨時作者與收件人）。
  - **中繼成本不變**：仍是每則 1 個事件（封裝是端上的加密，不增加發佈數）。
  - `NUDGE_KIND = 20100` 原本在 `relay-backend.ts`、`browser-backend.ts`、`demo/main.ts`
    **各定義一份**，現收進 core 的 `KIND`（SSOT）。
  - 測試 953 → **961**（core +4：外層作者 ≠ 真實作者、收件人解得開、他人解不開、
    每則的外層作者都不同；engine +4：端到端 typing/nudge、**線上流量不含真實 pubkey**、
    **陌生人的 nudge 被丟棄**、**封鎖者的被丟棄**）。

- 已知限制：
  - **收件人仍是明文**（`["p", …]`）。這與 kind 1059 相同，是 NIP-59 的既定取捨——中繼必須
    知道要轉發給誰。**我們隱藏的是寄件人，不是收件人。**
  - **心跳（kind 20000）仍以真名廣播**，且其訂閱仍帶 `authors: [聯絡人們]` →
    **中繼仍可從訂閱過濾器重建聯絡人清單**。這是 Nostr 的固有性質（REQ 過濾器對中繼可見），
    要修得改成「向每位聯絡人各發一則封裝心跳」——那會讓發佈數乘以聯絡人數，
    直接摧毀 ADR-0109 的配額節省（心跳本來就佔 92% 的請求）。**本 ADR 不處理**，
    留待專門的 ADR 權衡。
  - 心跳內容（狀態訊息、正在聽什麼）同樣是明文廣播——同上。
  - 過渡期的「接收舊版明文」是**暫時**的程式碼路徑，待所有客戶端更新後移除。
