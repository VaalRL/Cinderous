# 0160. 企業訊息保留天數——名冊政策 `messageTtlDays`＋中繼站可設上限

- 狀態：已接受（已實作）
- 日期：2026-07-16
- 相關文件：ADR-0065（離線留言 TTL 上限 7 天——本 ADR 使其可依部署放寬）、
  0047/0048（名冊與政策——保留天數為新政策欄）、0044（企業自架封閉節點）

## 背景與問題

離線留言的壽命由**發送端**蓋的 NIP-40 `expiration`（預設 7 天）與**中繼站**的
TTL 上限（`DEFAULT_MAX_TTL_SECONDS`＝7 天，超長截斷）共同決定。企業自架自己的
relay（ADR-0044），儲存是自己的成本與合規範圍——應可設定訊息保留更久。

## 決策

兩層都要動，缺一不可（發送端不蓋長章＝白配；中繼不放寬＝被截斷）：

### 1. 名冊政策 `policy.messageTtlDays`（1–365 整數）

管理者在名冊管理視窗設定「訊息保留天數」（留空＝預設 7 天）。隨簽章名冊分發，
全組織一致。`parsePolicy` 僅接受 1–365 整數（壞值視為未設）；
`policyTtlSeconds(policy)` 純函式供發送端換算。

### 2. 發送端蓋章：聊天／檔案 metadata 的外層 `expiration`

`lastRoster.policy.messageTtlDays` 存在時（成員＝採用的名冊；企業主＝自己發佈的
名冊），`wrapMessage`／`wrapFileMessage`／`wrapGroupMessage`／`wrapGroupFile` 的外層
過期改蓋 `now + N 天`。**限時訊息（閱後即焚）不受影響**——`disappearAt` 語意優先，
不被組織保留期拉長。回條/個人檔/入職等控制事件維持 7 天（無久存價值）。

### 3. 中繼站可設上限：`MAX_TTL_DAYS`

- CF Worker（`worker.ts`）：wrangler 環境變數 `MAX_TTL_DAYS` → store `maxTtlSeconds`。
- Node relay（`node-relay.ts`）：環境變數 `MAX_TTL_DAYS`（比照既有 `MAX_PER_RECIPIENT`）。
- 未設＝維持 7 天（公共站行為不變）。超過站方上限的章仍被 `effectiveExpiration`
  截斷——**站方上限恆為權威**，client 蓋不出超過部署者意願的壽命。

## 理由

- 「保留多久」本質是**組織政策 × 部署者資源**的交集：政策走名冊（0047 的單一真實
  來源，全員一致、離職者自然被撤銷），資源上限走部署設定（0044 的自架語意）。
- 兩端解耦而防呆：舊客戶端在新 relay 照舊 7 天；新客戶端在舊/公共 relay 被截回 7 天
  ——任何組合都不壞，只是取兩者較小值。

## 後果

- 正面：企業可依合規/習慣設 30/90/365 天離線信箱；公共站零影響。
- 已知限制／取捨：
  - 保留期是**離線信箱**壽命（未取件的密文），不是聊天記錄——已取件的訊息本來就
    永存於兩端本機；這只影響「離線多久內回來還收得到」。
  - 每收件人 500 則配額（PRD §8）不變——保留變長時滿溢仍以 FIFO 逐出最舊。
  - 全組織一份天數（無分群組/成員）；動態調整後僅影響**新發**的訊息（舊章不改）。
- 測試：core（policy 解析防禦、policyTtlSeconds、群訊/群檔 expiration 覆寫）＋
  relay（放寬上限後 30 天章不被截；未設仍截 7 天）＋desktop（名冊視窗欄位/預填）。
