# 0045. 單一 App 多身分並存與切換（工作＋個人）

- 狀態：已接受
- 日期：2026-07-05
- 相關文件：docs/adr/0044（企業封閉 allowlist）、docs/adr/0002（隱私基線）、docs/adr/0009（多設備同步）；apps/desktop 儲存與 App 殼

## 背景與問題

企業模式（ADR-0044）讓「工作身分」限用公司節點。使用者仍應保有「用個人身分對外」的能力，且希望**在同一個 App 內一鍵切換**工作/個人身分（本 ADR 的產品決策，經與使用者確認）。

現況：`AppStorage` 與 `RelayChatBackend` 皆為**單一身分/單一 relay 情境**（`nb.identity` 一把、一個 home relay）。要支援多身分需解決：身分登錄、**每身分資料隔離**、每身分各自的 relay 情境、切換時的拆/建。

## 決策

**採「多身分設定檔（profile）+ 一次一個作用中身分 + 每身分資料以 pubkey 命名空間隔離」。**

1. **身分設定檔**：`Profile = { pubkey, nsec, name, relayUrl, enterprise }`。全域登錄於 `nb.profiles`，作用中身分於 `nb.activeProfile`。`enterprise=true` 者為工作身分（連公司座、鎖定）。
2. **資料隔離（關鍵）**：每身分的聯絡人/訊息/群組/回應/貼圖庫等，一律以 **pubkey 命名空間前綴**存放（`nb.<pubkey>.<key>`），**互不交雜**——這對工作/個人分離與合規至關重要。既有單一身分維持舊鍵（命名空間為空字串），**新身分才用前綴**，避免資料遷移風險。
3. **每身分 relay 情境**：
   - 工作身分（`enterprise`）：`relayUrl` 鎖公司座，**關閉 relay pool（ADR-0034）/ 錨點 / hint 漫遊（ADR-0035/0036）/ home 遞補（ADR-0039）**——不得漫遊出公司節點。
   - 個人身分：維持開放模式（pool/錨點/漫遊照常）。
4. **一次一個作用中身分**：切換＝`stop()` 目前 backend、清空 UI 狀態、以選定 profile 的儲存命名空間與 relay 設定 `start()` 新 backend。**不同時多連線**（避免跨身分事件交叉、狀態簡單）；同時線上為後續可選項。
5. **UI**：聯絡人視窗標題列的身分切換器（下拉）＋「新增身分」（產生或匯入 nsec、設定 relay、標記是否企業）。切換與新增為前端殼職責，通訊層不變。

## 理由

- **尊重自主又能受控**：個人身分走開放、工作身分受 allowlist 且鎖定；兩者在同一 App 但**資料以命名空間硬隔離**。
- **最小侵入、可測**：profile 登錄與命名空間為純邏輯（可 TDD）；`LocalStorage` 加一個命名空間建構參數即可，既有鍵不動（空命名空間）→ 無資料遷移、向後相容。
- **一次一作用中**：重用既有 `start()/stop()` 生命週期，避免重寫 backend 為多實例；跨身分不交叉。
- **符合 SSOT**：身分仍是自主金鑰、relay 仍零狀態；本 ADR 只在**客戶端**新增身分容器與切換。

## 後果

- 正面：同一 App 內工作/個人並存、一鍵切換；資料不交雜；工作身分被鎖在公司節點、個人身分自由對外。
- 負面 / 已知限制：
  - 一次只有一個身分在線 → 另一身分的即時訊息在切換回去前不會即時收到（離線留言仍會在重連時取回）。同時在線為後續。
  - 多身分放同一裝置：合規上若企業要求工作資料完全不與個人同機，仍須搭配裝置管理（本系統之外）。
  - nsec 目前存於本機儲存（多把）；原生版應改存 OS 金鑰庫（沿用既有 TODO）。
- 後續行動：`profiles.ts` 純登錄 + `LocalStorage` 命名空間（本批）；App 切換器 UI 與 backend 切換接線；工作身分的「鎖定 relay 情境」旗標接入 backend；離職/移除身分的清理。
