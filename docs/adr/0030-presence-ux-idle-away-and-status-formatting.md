# 0030. 在線狀態 UX：閒置自動離開與狀態列表情/格式

- 狀態：已接受
- 日期：2026-07-02
- 相關文件：ARCHITECTURE.md（在線狀態與心跳）、docs/adr/0006（心跳容量）、packages/core/src/presence-status.ts、apps/desktop/src/ui/idle-status.ts、apps/desktop/src/ui/emoticons.ts

## 背景與問題

自訂狀態盤點後，相對 MSN 仍缺兩個經典行為：

1. **閒置自動「離開」**：使用者離開電腦時，好友仍看到「線上」，造成期待落差。
2. **狀態列表情/格式**：MSN 打 `:)` 會變笑臉、個人訊息可帶格式；目前狀態列為純文字。

兩者皆屬純前端呈現/行為，不應牽動 Nostr 事件契約或隱私模型。需決定：門檻與覆蓋語意、表情資產來源、以及與現有心跳/Markdown 的整合方式。

## 考量的選項

- **閒置偵測門檻**：(A) 固定 5 分鐘；(B) 使用者可設定。
- **覆蓋語意**：(A) 僅在「線上」時自動離開、手動狀態一律優先；(B) 任何狀態逾時都轉離開。
- **表情資產**：(A) Unicode emoji 短碼映射；(B) 內建圖片貼圖組。
- **狀態格式**：(A) 重用既有行內 Markdown 渲染；(B) 另設狀態專用語法。

## 決策

- 閒置門檻採 **(A) 固定 5 分鐘**（`DEFAULT_IDLE_MS`，程式碼常數，日後要開設定再說）；tick 以 30 秒輪詢，監聽 `mousemove/mousedown/keydown/touchstart/wheel`。
- 覆蓋語意採 **(A)**：僅當「手動狀態＝線上」時，閒置逾時才自動切 `away`；一有活動立即還原手動狀態。使用者手動選定的 `busy/offline/away` **永遠優先**、不被自動邏輯覆蓋。狀態機為純函式 `reduceIdle`（`idle-status.ts`），套用走既有 `setStatus` 心跳路徑。
- 表情採 **(A) Unicode emoji 短碼映射**（`emoticons.ts`），刻意避開任何專有貼圖/商標資產。
- 狀態格式採 **(A)**：重用訊息用的行內 Markdown（`markdown.tsx`），`status-text.tsx` 先做短碼替換再交給 Markdown 渲染；自我狀態提供即時預覽。

## 理由

- 固定門檻與純函式狀態機把「難測的邏輯」與「DOM/計時器副作用」分離，可在 node 環境完整單元測試，符合 Fix-First 與最小依賴。
- 「手動優先」避免自動邏輯把使用者刻意設的忙碌/離線洗掉，是最小驚訝原則。
- 走既有心跳路徑代表**協定零變更**（`PresenceState` 已含 `away`），中繼站與加密模型不受影響。
- Unicode emoji 免資產、免商標風險，跨平台一致；重用 Markdown 避免新語法與重複程式碼。

## 後果

- 正面：更貼近 MSN 體感（自動離開、`:)`→🙂、粗體/斜體狀態）；無協定/隱私/依賴變更；核心邏輯純函式、具回歸測試（idle 8、emoticons 6）。
- 負面 / 已知殘餘風險：門檻寫死、暫不可由使用者調整；表情短碼為全文字面替換，狀態列中若出現如 `8)` 等序列可能被誤轉（狀態列情境下影響輕微）。
- 後續行動 / 待辦：若使用者反映需求，再把閒置門檻與「更細狀態預設值（BRB/通話中/外出）」納入設定與 `PresenceState` 擴充（後者屬協定層，須另立 ADR）。
