# 0140. 修復切換身分→掉登入頁→建重複身分（帶著舊聯絡人）

- 狀態：已接受（已實作）
- 日期：2026-07-15
- 相關文件：**ADR-0045（多身分設定檔）**、0053（OS 金鑰庫）、0067（本地密碼）、0112（web 靜態加密）、
  0122（重載不得靜默換人），apps/desktop/src/App.tsx

## 背景與問題

從身分下拉選單選一個過去身分後：**跳回登入頁 → 輸入相同名稱登入 → 又多一個身分，且新身分裡有舊聯絡人。**

追查（`App.tsx`）發現是**兩個疊加缺陷**把「取不到金鑰」這件本該可救援的小事放大成災難：

1. **觸發點**：切換＝`switchProfile()` 設 active＋`location.reload()`。重載時要重新載入該身分的私鑰；
   Tauri 走 `loadNsecFromVault()`（OS 金鑰庫）。一旦**取不到**（換機/金鑰庫遺失/舊狀態），
   `buildBackend()` 會因 `expectPubkey` 守衛拋 `IDENTITY_UNAVAILABLE`（ADR-0122）→ 後端 null →
   `if (!backend) return <SignIn>` → **掉通用登入頁**。
2. **缺陷 A（不對稱）**：瀏覽器取不到金鑰時會 `setNeedNsec(active)` 引導**救回同一身分**；
   **Tauri 路徑漏了這一步**，直接掉通用登入。
3. **缺陷 B（命名空間重用）**：登入處理 `signIn()` **不論何時都用空命名空間 `""` ＋新產生的金鑰**。
   於是輸入相同名稱登入 → 新金鑰＝新 pubkey＝**多一個身分**；命名空間又是 `""`，而第一個身分的
   聯絡人正存在 `""` → 新身分**讀到舊聯絡人**。

## 決策

### A — Tauri 取不到金鑰 → 救援分流（與瀏覽器對稱）

開機流程 Tauri 分支：`loadNsecFromVault(active)` 回 undefined 且身分已知（有 pubkey）時，
`setNeedNsec(active); return;`——引導使用者**貼回這個身分的 nsec**（`onEnterNsec` 會驗證 pubkey 相符），
救回**同一身分**（同命名空間、聯絡人完好、不建重複），而非掉通用登入頁。

### B — 命名空間隔離（純函式 `pickSignInNamespace`）

`signIn()` 只有在**登錄裡還沒有任何身分佔用 `""`**（真正第一個身分，向後相容）時才用 `""`；
否則新產生的身分用 `namespace: <新pubkey>`。抽成純函式 `pickSignInNamespace(profiles, pubkey)` 上單元
測試。這樣即使使用者仍走了新建路徑，也**不會讀到第一個身分存在 `""` 的資料**。

### C — 救回即補金鑰庫（治本觸發點）

`enterWithNsec()` 在 Tauri 分支救回後 `getKeyVault().setKey(p.pubkey, nsec)`——若這次是因金鑰庫遺失
才走救援，補寫後**下次重載就能自動載入、不再掉登入頁**。以 pubkey 為鍵、冪等。

## 理由

- A 讓兩平台對稱：取不到金鑰是「請證明你是這個身分」（救回），不是「請重新建一個身分」。
- B 是最後一道防線：命名空間是資料隔離的根（ADR-0045）；空命名空間屬於第一個身分，重用它就是把
  兩個身分的資料混在一起。純函式化便於測試與推理。
- C 治本：把「一次成功的救回」變成「永久修好」，避免每次重載重蹈覆轍。

## 後果

- 正面：
  - 切換到取不到金鑰的身分 → **引導救回同一身分**（不再掉通用登入、不再建重複、不再汙染舊聯絡人）。
  - 即使走新建路徑，新身分**用自己的命名空間**，資料乾淨隔離。
  - Tauri 救回後補寫金鑰庫，觸發點治本。
  - 測試 314 → **316**（`pickSignInNamespace`：第一個身分用 `""`、已有 `""` 佔用時新身分用 pubkey 命名空間）。

- 已知限制：
  - `needNsec` 畫面（`SignIn`）目前仍**同時**提供「貼 nsec 救回」與「輸入名稱新建」兩條路——B 已讓
    新建不再汙染資料，但「把新建入口在救援情境下降級/隱藏」的 UI 細修留作後續（非資料安全問題）。
  - 瀏覽器端救回後的**持久化**需使用者設密碼（Argon2id 包裹，ADR-0112）——與 C 的 Tauri 免密碼補寫
    不同側，屬既有設計，非本 ADR 改動。
