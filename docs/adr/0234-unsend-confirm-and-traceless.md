# 0234. 收回確認彈窗與無痕收回

- 狀態：已接受
- 日期：2026-07-22
- 相關文件：ADR-0012（收回 v1：佔位「訊息已收回」）、ADR-0233（收回擁有者驗證）、ADR-0107（自封副本）、ADR-0136（行動端長按選單）

## 背景與問題

兩個收回體驗缺口：(1) 收回**沒有二次確認**——hover 誤點就直接收回、不可逆；(2) 收回一律留下「訊息已收回」佔位，**沒有「無痕」選項**——有時使用者不想讓對方看到「這裡曾有一則被收回的訊息」。

## 決策

1. **確認彈窗（兩端）**：
   - 桌面：收回鈕改開確認彈窗（標題＋說明＋「無痕收回」勾選＋收回/取消）；主列表、串面板、詳情面板三個入口共用。
   - 行動端：長按選單的「收回」改開**確認列**（inline，含 ☐ 無痕收回切換＋收回/取消）——維持長按選單的操作習慣、不引入新 Modal。
   - 勾選每次開啟重置（預設不無痕），避免上次選擇殘留造成誤無痕。
2. **無痕收回契約**：kind 5 刪除 rumor 加 `["purge"]` tag（`wrapDeletion({traceless})`／`deletionTraceless`）。攜於加密內層、中繼不可見；**舊版客戶端不認得 → 退回佔位顯示（無害降級）**。
3. **儲存語意分流**：`AppStorage` 新增 `markPurged`/`loadPurged`（與 `deleted` 分開的集合；snapshot 加 `purged` 欄位、舊快照容忍缺欄）。**purged＝顯示層整行剔除**（主列表、串、相簿、回覆數、開機回放一致）；deleted＝佔位。原文仍在本機儲存（與 ADR-0012 同為軟隱藏架構）。
4. **匯出也無痕**：明文匯出（ADR-0094）將 purged 訊息整行剔除（一般收回維持「（已收回）」佔位）。
5. **驗證沿用 ADR-0233**：無痕收回同樣只接受原寄件人；等待區（pendingDeletes）連同 traceless 旗標暫存。
6. `unsendMessage(to, id, traceless?)`／`onUnsend(id, traceless?)` 契約擴充；1:1 與群組、自封副本（多裝置）皆支援。

## 理由

- 確認彈窗補上「不可逆操作要確認」的一致性（接管/刪除託管等既有先例）；把無痕選項放進同一個彈窗＝一次決策、零多餘步驟。
- 無痕與佔位分兩個集合而非布林欄位：與既有 `deleted` 持久化管道同構、舊快照/舊客戶端自然相容。
- 行動端 inline 確認列而非 Modal：長按選單本來就是列，樣式與手勢一致。

## 後果

- **正面**：誤觸收回有攔截；使用者可選擇是否留痕；兩端（桌面/行動）與多裝置行為一致。
- **負面／已知殘餘**：
  - 「無痕」只是**顯示層**無痕：對方可能已讀過/截圖；原密文仍在中繼至過期；本機儲存仍有原文（僅顯示與匯出剔除）。與 ADR-0012 的固有限制相同。
  - 舊版客戶端顯示佔位而非無痕（契約降級，已明示於 hint 文案）。
- 測試：core purge tag 往返、engine 兩端 purged/deleted 分流、desktop purged 整行剔除渲染；全 repo 綠。
