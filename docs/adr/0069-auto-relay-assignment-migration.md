# 0069. Relay 自動分配與自動搬家（簽章清單驅動）

- 狀態：已接受
- 日期：2026-07-09
- 相關文件：docs/adr/0006（容量估算）、0034（多中繼路由）、0036（離線回退）、
  0039（混合式引導路由／home 遞補）、0059（休眠化與免費層量級）、0065（TTL）、
  0066（搬家三階段）、apps/desktop/src/bootstrap-config.ts、.github/workflows/relay-health.yml

## 背景與問題

1. **新帳號需手填 relay 網址**——去中心化的必要之惡？其實不是：ADR-0039 已有
   維護者簽章 relay 清單＋每小時健康檢查，只差「從清單自動選座」這一步。
2. **home 遞補只有 session 級**（ADR-0039 缺口）：`onHomeSwitched` 只更新
   `nb.relayUrl`，不更新 `nb.profiles` 的 `relayUrl`——每次重啟都連回死掉的舊站、
   空等 5 分鐘再重新遞補，且無排水、無主動廣播。
3. **額度與退役無官方遷移路徑**：Workers 免費層額度吃緊或維護者要下架某座時，
   沒有讓用戶優雅撤離的機制。
4. 營運現實：`ANCHOR_RELAYS` 目前為空、生產僅一座 relay——本 ADR 的機制先行，
   實效待錨點立起（見前提）。

## 考量的選項

- 選項 A：維持現況（手填網址＋session 遞補）。
- 選項 B：**簽章清單擴充＋分級觸發**（採用）——見決策。
- 選項 C：「接近額度即自動搬家」——否決：免費層額度**每日重置**，接近額度多為暫時
  現象（既有 ADR-0036/0039 離線回退已撐住）；且「搬到下一個」的羊群效應會骨牌式
  壓垮清單後座。
- 選項 D：relay 端指派/遷移帳號——否決：違反零伺服器狀態。

## 決策

1. **清單 schema 擴充**：簽章清單每座 relay 增
   `{ url, accepting: boolean, weight: number, status: "ok"|"draining"|"retired" }`；
   維護者簽章與帶內傳播機制不變（ADR-0039）。relay 以 DO 內粗略計數自估用量，
   健康檢查探測後寫入 `accepting`。
2. **新帳號自動分配**：SignIn relay 欄改為選填（保留手填）；留空→自 `accepting`
   座**加權隨機**選一座（無伺服器協調的統計式均衡）＋連線健康探測。
3. **自動搬家分級觸發**：
   - T1（現況保留）：home 離線 >5 分 → session 遞補（可反悔、零副作用）。
   - T2（新）：遞補後持續離線超過門檻（建議 24h）或開機時 home 已死且引導座健康
     → **durable 搬家**：走 ADR-0066 機械（`changeProfileRelay` 保命名空間＋H3 排水
     ＋H1 廣播改道）＋事後通知使用者。此項同時修正背景 2 的持久化缺口。
   - T3（新）：清單標記 `draining`/`retired` → 既有用戶**分批隨機延遲**自動搬家
     （防羊群）；`accepting: false` 僅擋新分配、既有用戶不動。
4. **護欄**：搬家目標＝清單健康座加權隨機；企業身分排除（鎖定漫遊）；遲滯防抖
   （只有連續離線超長才 durable）；多裝置 split-brain 記 ADR-0009 backlog
   （緩解：決定性選座——照清單序取首個健康座）。

## 理由

- 全部機械已存在（0039 清單＋遞補、0066 三階段搬家），本 ADR 只補 schema、觸發
  策略與接線；隱私模型不變（清單公開但只含 relay 資訊，用戶路由仍走帶內加密 hint）。
- 分級把「暫時壅塞」與「永久遷移」分開：前者靠既有回退零成本吸收，後者才動用搬家。

## 後果

- 正面：新用戶免填網址；relay 死亡/退役對用戶透明；遞補從「每次重啟重演」變一次到位。
- 負面／已知殘餘風險：維護者承擔錨點成本與清單治理（收錄第三方 relay＝信任其
  元資料視野，需收錄準則）；自動搬家改變分享字串（既有聯絡人 hint 自癒，新朋友用
  舊字串時靠 ADR-0036/0039 冗餘廣播兜底）；多裝置各自搬家的收斂待 ADR-0009。
- 前提（營運）：立起 ≥2 座錨點 relay、填入 `ANCHOR_RELAYS` 與 `MAINTAINER_PUBKEY`。
- 後續行動 / 待辦：依 ROADMAP Phase I 施工（I1 清單 schema → I2 T2 持久化搬家 →
  I3 T3 退役撤離 → I4 SignIn 自動選座）；完成後更新 ARCHITECTURE.md。
