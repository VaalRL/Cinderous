# 0243. WebRTC TURN 保底：缺口、影響界定與明確標示

- 狀態：已接受（明確標示限制＋上一個「有上限」的公共 TURN；成本經評估 trivial、修 ~10–20% 通話失敗）
- 日期：2026-07-23
- 相關文件：ADR-0048（企業強制 TURN／`forceTurn`）、ADR-0162（檔案塊經 relay 退路）、
  ADR-0025／0026（M8 通話）、ADR-0017（M3 檔案傳輸信令）、`packages/engine/src/backend/rtc-config.ts`、
  `ARCHITECTURE.md §7`、`docs/SECURITY.md`

## 背景與問題

WebRTC 直連要穿越 NAT：**STUN**（照鏡子，告訴你自己的公網位址；`rtc-config.ts` 預設
`stun.cloudflare.com`）＋ ICE 打洞。但**對稱式 NAT／嚴格防火牆**（常見於公司網路、部分電信 CGN）
對每個目的地給不同對外 port，STUN 位址對方用不上 → **兩邊都是這種網路時，沒有直接路徑**。

**TURN** 是打不穿時的中繼保底：兩邊都連上它、由它轉發封包（仍 E2E 加密，只轉密文）。但它**搬運
全部影音、吃頻寬**（不像 STUN 只回一次位址）。

**現況**：程式碼支援 TURN（企業可設 `turnServers`＋`forceTurn`，ADR-0048），但**沒有部署給一般
使用者的公共 TURN**。故一般使用者：**STUN 打不穿時沒有退路。**

### 影響界定（關鍵：不同功能退路不同）

| 功能 | 直連打不穿且無 TURN 時 |
| --- | --- |
| **語音/視訊通話（M8）** | 🔴 **失敗**——即時影音**只能**走 P2P 或 TURN，**無「經文字中繼」退路**。兩個對稱 NAT 的人通話接不通。 |
| 檔案傳輸（M3） | ✅ 有退路——切塊走 relay 信箱（ADR-0162）。 |
| 文字/回應/收回/群訊 | ✅ 本來就走 relay。 |
| Nudge/輸入中/在線 | ✅ 退回 relay。 |

**唯一真正的痛點：限制網路下的通話接不通。** 其餘皆有退路。

## 考量的選項

- **選項 0：現況＋不說（靜默失敗）。** 使用者只看到「通話連不上」卻不知為何——最差。
- **選項 1（採用）：明確標示限制＋企業/自架可配 TURN＋公共 TURN 延後。** 誠實文件化＋（後續）通話
  失敗的 UI 提示；企業/自架者用既有 `turnServers`/`forceTurn`；公共 TURN 當成本決策另議。
- **選項 2：現在部署無上限公共 TURN。** 解決最徹底，但**頻寬成本**對免費/捐款制是負擔。
- **選項 3：有流量上限的公共 TURN。** 保底但不無限燒錢；上限用完仍退回「通話可能失敗」。

## 成本與必要性評估（2026-07-23 補）

原本把「公共 TURN」當成本負擔而延後——**重估後發現這個顧慮被高估了。**

- **多久真的需要 TURN？** 業界經驗值：約 **8–20% 的 WebRTC 連線**需退回 TURN（其餘靠直連/STUN 就通）。
  Cinderous 的使用者偏「家用網路、個人」（多為對 STUN 友善的 cone NAT）→ 大概落在**低段（~10% 或更低）**；
  但**行動端電信 CGN 與企業網路**是高失敗段（企業已可自配 TURN）。結論：**真實需要 TURN 的是「少數通話」，
  非全部流量。**
- **成本其實很小。** 只有「P2P 失敗的通話」用 TURN，而**語音頻寬很低**（雙向 ~80 kbps → 10 分鐘語音
  ≈ 6 MB）。視訊較高（~150 MB/10 分鐘）但同樣只在失敗時。故實際 TURN 流量 = 少數通話 × 小頻寬 = **很小**。
- **現規模下可近乎免費：** (a) **自架 Coturn** 於便宜 VPS（固定 ~$5/月、含足量 egress）——對捐款制而言等同
  免費；(b) 供應商**免費額度**（metered.ca 等）；(c) **Cloudflare TURN**（relay 已在 CF、最低摩擦，按 GB 計、
  流量小＝帳單小、可設上限）。

**修正結論：TURN 的成本在目前規模是 trivial，而它修的是 ~10–20% 通話的接不通——所以「延後」的理由不成立，
值得直接上一個「有上限的」公共 TURN。**

## 決策

**採選項 1（明確標示）＋修正為「上一個有上限的公共 TURN」（成本 trivial、修真實 ~10–20% 通話失敗）。**

- **一般使用者**：預設 STUN ＋ **一個有上限的公共 TURN 保底**（選項 3）。上限用完仍退回「通話可能失敗」，
  但覆蓋絕大多數需求。候選：**Cloudflare TURN**（最低摩擦、可設上限）或**便宜 VPS 上的 Coturn**（固定成本）。
- **企業/自架**：以既有 `turnServers`＋`forceTurn`（ADR-0048）自配（避第三方元資料、或更高上限）。
- **明確標示**（本 ADR 核心，已落地）：ARCHITECTURE §7（M3/M8）、SECURITY 殘餘、以及**（後續）通話失敗的
  使用者提示**：「限制網路下通話可能接不通；文字/檔案/nudge 不受影響；企業/自架可配 TURN」——不靜默失敗。
- **不做**：無上限的公共 TURN（防視訊在大規模下失控燒頻寬）——用「上限＋退回提示」保底即可。

## 理由

- **為何上「有上限的」而非「無上限」或「延後」。** 延後的理由（成本負擔）**經評估不成立**——只有失敗
  P2P 的少數通話用 TURN、語音頻寬又小，實際流量很小，現規模下近乎免費（Coturn ~$5/月固定、或 Cloudflare
  TURN 小帳單）。但也**不上無上限**：視訊在未來大規模下仍可能燒頻寬 → 設「上限＋退回提示」既保底又封頂。
- **為何仍要明確標示（即使有 TURN）。** 上限用完、或某些極端網路 TURN 也不通時，仍會失敗——靜默失敗
  最傷信任。誠實標示＋指路（企業/自架配更高上限）是最誠信的處置。
- **為何 Cloudflare TURN 是首選候選。** relay 已在 Cloudflare；用其 TURN 摩擦最低、按 GB 計、流量小＝帳單
  小、可設上限。次選：便宜 VPS 上的 Coturn（固定成本、可控元資料）。
- **隱私標示。** TURN 一旦使用，會看到「這兩個 IP 在通話」的元資料（時間/流量），但**看不到內容**
  （E2E 密文）——與 relay 同級。自架 TURN 可避第三方。這點併入 SECURITY 殘餘。

## 後果

- 正面：
  - **修真實 ~10–20% 通話失敗**，且現規模下成本 trivial（近乎免費）。
  - 上限封頂，避免視訊在未來大規模下失控燒頻寬。
  - 不再靜默失敗——限制受**明確標示**；企業/自架可配更高上限或避第三方。
- 負面 / 已知殘餘風險：
  - **上限用完 / 極端網路 TURN 也不通時，通話仍失敗**——明確標示、非靜默。
  - **TURN 使用時是元資料可見方**（見兩端 IP、通話時序/流量，不見內容）；公共 TURN 即多一個第三方可見方，
    自架可避。需選「可設上限、且信任的」TURN（Cloudflare／自架）。
- 後續行動 / 待辦：
  1. ✅ 明確標示：ARCHITECTURE §7（M3/M8）、SECURITY 殘餘（本次一併）。
  2. **上一個有上限的公共 TURN**：評估 Cloudflare TURN（首選、設流量上限）vs 便宜 VPS Coturn；接進
     `rtc-config` 的預設 `iceServers`（一般使用者即享保底）。
  3. **通話失敗的使用者提示**（後續實作）：連線失敗時提示「限制網路可能導致，可試 Wi-Fi/其他網路」，
     而非只顯示「連不上」。
