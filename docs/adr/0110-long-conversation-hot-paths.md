# 0110. 長對話的熱路徑成本：O(n)／O(n²)／O(k×n) 全面收斂

- 狀態：已接受（已實作）
- 日期：2026-07-14
- 相關文件：ADR-0054（加密儲存基質）、0058（送達/已讀回條）、0094（保留上限）、
  0095（群組回條水位）、0107（自封副本）、0108（已讀水位）

## 背景與問題

盤點「單一對話歷史非常長」的風險。用**真的儲存層**跑基準（不是理論推估）：

**LocalStorage（手機版與網頁版實際使用）**

| 對話長度 | 收一則訊息 | 單則狀態更新 | **已讀水位回條（50 則）** |
|---|---:|---:|---:|
| 1,000（0.1MB） | 1.1ms | 0.8ms | 43ms |
| 10,000（1.0MB） | 10ms | 14ms | 487ms |
| **50,000（4.9MB）** | **47ms** | **53ms** | **3,531ms** |

**MemoryStorage 建置 10 萬則歷史**：**逾時（>120 秒）**——因為 `appendMessage` 的線性去重
讓整批匯入退化成 **O(n²)**。

**TauriStorage（桌面）**：每次寫入把**整個**儲存序列化＋**AES-256-GCM 加密**＋寫檔，
每 250ms 一次。10 萬則＝每次 35ms 序列化 ＋ ~10MB 加密 ＋ ~10MB 寫檔——**只因為改了一則
訊息的狀態**。

而 **ADR-0094 的保留上限預設是「無上限」**——這條路是預設開著的。

### 四個具體缺陷

- **R1（ADR-0108 引入）**：`unreadCounts()` 在**每一則收訊**時重掃「**所有對話的所有訊息**」。
  在 LocalStorage 下＝收一則訊息就重新 `JSON.parse` 每個對話的完整歷史。
- **R2（平方級）**：已讀水位回條對**每一則**未讀訊息各呼叫一次 `setMessageStatus`，
  而持久化層每次都是「載入整個對話 → 改一則 → 寫回整個對話」→ **O(k×n)**。群組回條同理。
- **R3（平方級）**：`appendMessage` 以 `list.some()` 線性去重 → 整批匯入 **O(n²)**。
  這條路是**配對搬家還原／雲端快照合併／身分輪替**在走的。
- **R5（全量持久化）**：TauriStorage 每次都重寫整包；LocalStorage 每個方法都
  parse＋stringify 整個對話。

## 決策

### 1. 每個對話持有 `id → 訊息` 索引（R3）

`MemoryStorage` 的 `Map<convo, StoredMessage[]>` 改為 `Map<convo, { list, byId }>`。
兩者共用**同一份訊息物件參考**，就地改狀態即可，不需同步兩份資料。

不只 `appendMessage` 的去重變 O(1)——`setMessageStatus`／`setFileSavedPath`／`setFileThumb`／
`setMessageReceipt` 的 `.find()` 也全變 O(1)，而它們都在回條的熱路徑上。

**逐出（ADR-0094 保留上限）必須同步清索引**，否則會留下指向已逐出訊息的幽靈條目
（狀態改到一個不在 list 裡的物件、或讓被逐出的 id 無法重新加入）。已有測試釘住。

### 2. 批次介面：`setMessageStatusBulk` / `setMessageReceiptBulk`（R2）

整個已讀水位**一次**改完、**一次**寫回。回傳實際有前進的 id（供 UI 渲染）。

### 3. 未讀數改為增量維護（R1）

後端持有 `unread: Map<convo, number>` 與 `lastIncomingAt: Map<convo, number>`：
- **開機**：全量重算一次（`recountAllUnread`）。
- **收訊**：`bumpUnread(convo, at)` → +1，**O(1)**。
- **清未讀**：水位取自 `lastIncomingAt`，免掃全對話 → **O(1)**。
- **快照合併**：只重算被注入訊息的對話。

（`bumpUnread` 仍會檢查水位——**晚到的舊訊息**（時間在水位之下）不計未讀，與 ADR-0108
的水位語意一致。）

### 4. LocalStorage 改為「狀態常駐記憶體 ＋ 逐鍵寫回」（R5a）

委派 `MemoryStorage`（複用其索引），開機一次把 localStorage 讀進記憶體，之後**只把受影響的
鍵**寫回。讀取零解析、查找 O(1)，只剩寫回時的一次序列化。

**不引入平行實作**：LocalStorage 與 TauriStorage 現在是同一個模式（記憶體層 ＋ 持久化），
共用同一份索引邏輯。

### 5. TauriStorage 改為分部位持久化（R5b）

檔案格式由 `<ns>.enc`（整包）改為 `<ns>/<part>.enc`：
- `meta`：身分/聯絡人/群組/封鎖/回應/收回/已讀水位/引導清單（都很小）
- `msgs.<convo>`：每個對話一個檔

只重寫**變動的**部位 → 成本與總歷史長度無關。新增 Rust 命令
`store_load_parts` / `store_save_part` / `store_remove_part`。

- **路徑穿越防護**：部位名白名單（`[A-Za-z0-9._-]`、拒絕 `..`）——對話鍵是 hex，不該出現路徑字元。
- **單一部位毀損不拖垮其餘**：解密失敗的部位跳過，其他仍載入。
- **舊格式遷移**：開機時若無部位，讀舊的整包快照並就地拆成部位（**舊檔保留當備份**）。
  已有測試釘住——不能因為換格式而丟資料。
- **移除對話 → 刪掉它的部位檔**，不留孤兒密文。

## 理由

- 這些全是**常數因子與複雜度**的問題，不改任何語意——但它們決定 App 在真實使用下會不會凍結。
- R1 是 ADR-0108 自己引入的缺陷；**該修的是我**，不是等使用者踩到。
- R3 的 O(n²) 特別惡劣：它不在收訊路徑上，而在**換機還原**路徑上——使用者最不能忍受失敗的時刻。

## 後果

- 正面（實測）：

| | 修正前 | 修正後 | 倍數 |
|---|---:|---:|---:|
| 收一則訊息（5 萬則對話，LocalStorage） | 47ms | **12.6ms** | 3.7× |
| **已讀水位 50 則（5 萬則對話）** | **3,531ms** | **18ms** | **196×** |
| **建置 10 萬則歷史** | **逾時 >120s** | **68ms** | **>1,700×** |
| 桌面每次持久化 | 全量（10 萬則＝35ms＋10MB 加密寫檔） | **只寫變動的對話** | — |

  - 未讀重算由「每收一則訊息掃全部對話」變成 **O(1)**。
  - 測試 +25（storage 17 ＋ desktop 5 ＋ 既有回歸）。全 repo **848 通過**（＋13 Rust）。

- 負面 / 已知限制：
  - **記憶體**：LocalStorage 現在把全部歷史常駐記憶體（TauriStorage 本來就如此）。
    10 萬則 ≈ 10MB 字串 → 尚可，但**這不是無限的**。
  - **仍未解決的是「總量」**：`appendMessage` 的寫回仍是 O(該對話)（序列化那一次），
    且 **LocalStorage 的 5–10MB 配額**依舊會在約 5 萬則時撞牆並**靜默丟失寫入**。
    本 ADR 把**常數壓下來**，但沒有把 **N 綁住**。
  - → **N 的上界必須另解**（封存/冷熱分離）。見下。

## 後續：這只解了一半

本 ADR 讓每一條熱路徑對「歷史長度」的敏感度降到最低，但**歷史長度本身仍無上界**：

- LocalStorage 配額（5–10MB）→ 約 5 萬則後**靜默丟失寫入**（`write()` 只 `console.warn`）。
- 記憶體與單次序列化成本仍隨總量線性成長。

真正的解法是**把 N 綁住**：熱區固定上限、超出者**封存**（而非像 ADR-0094 現在的
`splice()` **直接刪掉**），另開歷史紀錄功能讀封存。那需要一個新的持久化基質
（localStorage 換不掉配額——同一個 origin 額度）與非同步的封存介面。**另立 ADR。**
