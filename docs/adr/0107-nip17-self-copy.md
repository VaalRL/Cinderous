# 0107. NIP-17 自封副本：讓自己的其他裝置看得到自己發出的訊息（訊息 id 統一為 rumor.id）

- 狀態：已接受（已實作）
- 日期：2026-07-14
- 相關文件：ADR-0009（多設備收斂語意）、0056（離線留言）、0058（送達/已讀回條）、
  0071（加密雲端快照）、0093（檔案投遞）、0095（群組回條與 rumor.id）；NIP-17 / NIP-59

## 背景與問題

盤點「多設備即時同步（ADR-0009 D4b）」時，查證出一個**比同步更根本的破損**。

`sendMessage()` 只發**一份** Gift Wrap，外層定址給**收件人**：

```ts
const evt = wrapMessage(text, this.sk, to, {...});  // 外層 ["p", Bob]
this.publishReliable(evt);                          // 就這一份
```

而每台裝置的收件箱訂閱是 `{ kinds: [1059], "#p": [自己的 pubkey] }`（`relay-backend.ts:605`）。

**於是「我發出的訊息」永遠不會進入我自己的收件箱**——它定址給 Bob，不是給我。發送裝置之所以
看得到，純粹是因為它**本機存了一份**。

### 破損是對稱的——這才是最糟的地方

| | Bob 傳來的訊息 | 我在手機發的 | 我在電腦發的 |
|---|:---:|:---:|:---:|
| **手機** | ✅ | ✅ | ❌ |
| **電腦** | ✅ | ❌ | ✅ |

兩台都拿到 Bob 的**全部**訊息（那些定址給我），但**各自只有自己發出的那半**。

只要交替使用兩台裝置，**沒有任何一台擁有完整對話**，而且兩台的紀錄**互相矛盾**。這比單向缺失
更難察覺，也更難事後修復。

一個旁證說明了問題的本質：Bob 回的**已送達回條是定址給我的** → **我的每一台裝置都收得到**，
然後對一則它根本沒有的訊息呼叫 `setMessageStatus` → 找不到，靜默丟棄。
**另一台裝置看得到「這件事發生過」的證據，卻看不到訊息本身。**

### 現有機制補不了這個洞

ADR-0071 的加密雲端快照**確實會持續合併**（`receiveSnapshot` → `mergeSnapshotContent`），
但它是**備份／最終一致**，不是同步：

- 預設 **`cloudSync = "off"`**（桌面 `p.cloudSync ?? "off"`、手機 `readCloudSync()` 同樣 fallback）
- 就算開到 `full`：**每日至多發一次**、且**只在開機時檢查** → 延遲最長 24 小時且需重啟
- 只帶最近 **500 則**，整包還要塞進 relay 的 **256KB** 單顆上限

**這是 NIP-17 早就解決的問題**：標準做法是送訊息時發**兩份** Gift Wrap——一份給對方、
**一份給自己**——正是為了讓自己的其他裝置看得到自己發的訊息。Cinder 沒有做這件事。

## 考量的選項

- **A. 把 `cloudSync` 預設改成 `full`**：不採。快照是**每日、有上限（500 則／256KB）**的備份，
  當同步用會遲到一整天，且大量歷史根本塞不進去。而且它把**訊息明文**（加密後）常駐在中繼上，
  是比自封副本大得多的暴露面——只為了修一個 NIP-17 本來就該做的事，不值得。
- **B. 自己的裝置之間走 P2P 直連補齊**：不採。要求兩台**同時在線**，而多設備最需要同步的場景
  恰恰是「另一台關著」。
- **C（採用）. 實作 NIP-17 的自封副本**：送訊息時，把**同一個 rumor** 另外包一份給自己的 pubkey。
  所有裝置本來就訂閱自己的收件箱 → **自動扇出到自己的每一台裝置，零額外機制**；
  離線的裝置靠 relay 的 7 天 TTL 補收。

## 決策

### 1. 送出「對話內容」時，額外對自己的 pubkey 再包一份 Gift Wrap（自封副本）

範圍：**1:1 文字訊息、檔案 metadata、群訊、回應（reaction）、收回（unsend）**。

**收回（unsend）必須納入**：若只同步訊息而不同步收回，會出現「手機收回了、電腦還顯示」
——使用者以為訊息已撤回，但它其實還躺在自己另一台裝置上。**那是隱私破損，不是不便**，
所以不列入後續，本次就做。

**不納入**：回條（`wrapReceipt`）、個人檔、輸入中／敲一下。
回條本來就定址給對方，而**對方回給我的回條會自然抵達我的所有裝置**——自封自己的回條是純浪費。

### 1b. 群組控制訊息（create/add/remove/leave）也必須自封——否則群訊自封形同無效

**這一項是實作時才發現的**（新增的群訊自封測試直接掛掉），但它不是可選的：

群控（`wrapGroupControl`）一向只發給**成員**，從不含自己。於是我的另一台裝置**根本不知道
這個群存在** → 群訊的自封副本抵達時，`receiveGroup()` 會以「未知群組」`if (!g) return` 丟棄。
**只做群訊自封而不做群控自封，等於什麼都沒做。**

另一台裝置由 `applyControl(from = 自己)` 處理，語意自然正確且冪等：
建群時 admin 就是自己；在一台裝置離群 → 另一台也離群；增/刪成員 → 另一台同步收斂。

例外：`addGroupMember` 發給**新成員**的那顆 `group-create` 不含自己（我已經有這個群了）。

### 2. 訊息 id 統一為 `rumor.id`（把 ADR-0095 的群組做法推廣到 1:1）

這是自封副本的**前提**，不是順手改的東西。

自封副本與給對方的那份是**兩顆不同的外層 wrap**（外層 id 必然不同），但**內層 rumor 完全相同**。
今天 1:1 用**外層 wrap id** 當訊息 id——那個 id 對「我的另一台裝置」毫無意義。

**唯有 `rumor.id` 能讓三方指向同一則訊息**：發送裝置、我的其他裝置、收件人。

連帶：**回條／回應／收回的 target 一律改用 `rumor.id`**（收件人回條從 `event.id` 改為 `rumor.id`）。

群組**早已**如此（ADR-0095 `wrapGroupMessage` 回傳共享的 `rumor.id`）——本決策是把 1:1 拉齊，
消除「1:1 用外層 id、群組用 rumor.id」這個既有的不一致。

### 3. rumor 內以 `["to", <收件人>]` 標記收件人——**不可用 `p` tag**

自封副本抵達我的另一台裝置時，它必須知道「這則訊息是發給誰的」才能歸檔到正確對話。

**不能用 `p` tag**：`mentionedPubkeys()` 會把 rumor 內**所有** `p` tag 當成 @提及
（`mention.ts:70`）——加了會讓每則訊息都「提及」收件人，污染提及與紅點。

故另立 `to` tag。它在**加密內層**，中繼看不到（與訊息內文同等保護）。

### 4. 收端以 `sender === self.pubkey` 分流自封副本

- `outgoing: true`、`contact` 取自 `to` tag（1:1）或 `g` tag（群訊）
- **不回送已送達回條**（不能對自己回條）
- **不 `ensureContact(自己)`**；改為 `ensureContact(收件人)` ——附帶效果：**我的另一台裝置會自動
  學到我發過訊息的聯絡人**，順帶補上 ADR-0009 提到的部分聯絡人同步缺口
- 回應的 `mine` 旗標取 `sender === self`
- **發送裝置自己也會收到自封副本** → 以 `seenMsg`（改鍵為 `rumor.id`）天然去重，無需特例
- **回應／收回／回條不帶 `to`**（它們以 `e` tag 指定目標訊息，與對話無關）→ 收端必須在
  「解析對話」**之前**處理它們；只有真正要建立訊息的分支（文字／檔案）才強制要有對話

### 5. 緩衝「早到的回條」——自封副本引入的新競態

NIP-59 的 `jitteredPast()` 把外層 wrap 的 `created_at` **隨機往前推**（隱私設計），
所以 relay 回放的順序**是亂的**：**Bob 的回條可能比我的自封副本更早抵達我的另一台裝置**。

若不處理，回條會套用到一則「還不存在」的訊息 → 靜默丟棄 → 該訊息永遠卡在 `sent`，
即使它早已送達／已讀。

故新增 `pendingStatus` 緩衝：找不到目標訊息的回條先暫存，待該訊息（自封副本）抵達時補套用。

### 6. 送出狀態只追蹤「給對方」的 wrap；自封副本為 best-effort

`sent` 的語意是「**已進中繼、在往對方的路上**」。若把自封副本也算進去，會出現
「只有自封副本成功、給對方的那份失敗」卻標成 `sent` 的謊報。

順帶把 1:1 併入既有的 `trackFanout()`（群訊在用），**刪掉 `awaitingSent` 這條平行機制**——
兩者語意本來就相同（任一 wrap OK → `sent`；全部放棄 → `failed`）。

## 理由

- 這是 **NIP-17 的標準行為**，不是 Cinder 的發明。我們一直是**偏離**標準的那一方。
- 傳輸成本極小：**每則外送訊息 +1 個事件**。以每天送 50 則的使用者估算，
  相對於**心跳**（30 秒一次 → **2,880 requests/日**）只增加約 **1.7%**。心跳大它 50 倍以上；
  真要省中繼額度，該動的是心跳，不是放棄「另一台看得到自己發的訊息」這種根本修復。
- **元數據不增加**：Gift Wrap 外層是**一次性金鑰**簽的 → 中繼看不出是誰發的。
  多一份定址給自己的密文，在中繼眼中就跟「又有人傳訊息給我」沒兩樣。
- **狀態自動收斂（附帶紅利）**：對方的回條本來就定址給我 → 我的**所有**裝置都收得到。
  只要 id 統一為 `rumor.id`，已送達／已讀就會**自動**在所有裝置一致，不必再為狀態另建同步機制。

## 後果

- 正面：
  - 對話在**所有自己的裝置**上**即時且完整**——這是 ADR-0009「多設備收斂」真正該有的樣子，
    而且不必打開雲端快照（不必把訊息常駐中繼）。
  - 訊息 id 語意統一（1:1 與群組都用 `rumor.id`），消除既有不一致。
  - 刪掉 `awaitingSent` 平行機制（1:1 併入既有的 `trackFanout`）。
  - 另一台裝置自動學到**聯絡人**與**群組**（後者是群控自封的附帶效果，見決策 1b）。
  - 測試：新增 19 個（core 238→248、engine 126→135）。engine 端以「**同一把 nsec ＋
    兩個 storage**」模擬手機/電腦兩台裝置，直接驗證本 ADR 的破損與修復。全 repo **809 通過**。

- 負面 / 代價：
  - **破壞性變更**：舊版用外層 wrap id、新版用 `rumor.id` → **跨版本的回條／回應／收回會對不上**。
    專案尚無外部使用者，接受；不做相容層（Fix First，不留平行路徑）。
  - 每則外送訊息**多一顆事件**：收件箱事件數約翻倍 → 重連時 `dm` 訂閱要多拉一倍
    （**啟動延遲**成本，非額度成本）。
  - 發送裝置會把自己的自封副本**下載回來再丟掉**（靠 `seenMsg` 去重）——小幅浪費，
    可日後以 `since` 收斂，現在不值得為此增加複雜度。

- **實作時才發現、設計階段沒想到的兩件事**（記於此，供日後判斷本 ADR 的完整性）：
  1. **群控不自封 → 群訊自封形同無效**（見決策 1b）。設計時只想到「訊息」，忘了另一台裝置
     連**群組本身都不知道**。是新增的測試把它抓出來的。
  2. **回應／收回不帶 `to`**，會被「解析對話」那一步的 `if (!convo) return` 吞掉。
     同樣由測試抓出。教訓：`to` 是**訊息**專用的歸檔標記，不是所有自封副本的通用欄位。

- 已知殘餘（**不在本 ADR 範圍**）：
  - **檔案位元組仍只到一台**（ADR-0093 的紅線：中繼不備份位元組）。自封副本讓另一台裝置
    看得到**檔案 metadata**（名稱/大小/類型），但**沒有位元組、也沒有縮圖**（縮圖是本機產物）。
    這是刻意的取捨，不是遺漏。
  - **已讀水位仍不同步**——而且它**連本機都還沒持久化**（`unread` 只是 React `useState`，
    `StoredMessage` 無已讀欄位）。跨裝置同步已讀的前置條件尚不存在，且它是**最吵、最洩漏
    元數據**的一項（每開一個對話就一則事件），與 ADR-0088（在線元數據最小化）方向相衝。
    應另立 ADR 專門處理，先從「本機持久化」做起。
  - 聯絡人／封鎖／群組的**完整** delta 同步（ADR-0009 D4b）仍未做。本 ADR 只順帶補上
    「發過訊息的聯絡人」。
