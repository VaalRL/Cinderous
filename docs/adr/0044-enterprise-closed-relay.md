# 0044. 企業模式：封閉 allowlist 中繼 + 自架單節點

- 狀態：已接受
- 日期：2026-07-05
- 相關文件：docs/adr/0002（隱私基線）、docs/adr/0005（自建中繼）、docs/adr/0034/0039（多中繼/引導路由）；PRD §7 威脅模型

## 背景與問題

企業客戶希望：**同一企業只用同一節點、可自架於本地伺服器/電腦（不限 Cloudflare Worker）、且外部客戶不進此系統（僅內部成員）**。需在不破壞既有隱私鐵則（明文不上雲、E2E、中繼零狀態）下提供封閉部署。

評估兩種成員管制：NIP-42 AUTH 與靜態 allowlist。使用者偏好 **allowlist**，且明確「外部客戶不進系統」。

## 決策

**採「發布 allowlist（內容層）＋ 自架私網（讀取層）」的封閉單節點模式。**

1. **中繼核心 allowlist（本次實作）**：`RelayCoreOptions.allowedAuthors`（hex pubkey 集合）。設定後 `handleEvent` 在驗簽後、寫庫/扇出前檢查 `event.pubkey ∈ allowlist`，非成員的**任何事件（含心跳）**回 `OK false "blocked:"`（永久拒絕，客戶端 Outbox 不重試）。未設＝維持開放中繼（現況不變）。
2. **自架單節點**：`RelayCore` 本就傳輸無關（`worker.ts` 僅注入 Cloudflare WS 外殼；測試/開發已在 Worker 外以 in-memory / dev-server 執行）。企業自架＝以 Node/Deno/Bun/Docker 的 WS server 包 `RelayCore`，跑在內網伺服器或單機。
3. **客戶端鎖單座（企業建置旗標，後續）**：企業版關閉 relay pool（ADR-0034）、錨點、hint 漫遊（ADR-0035/0036）、home 遞補（ADR-0039），只連被鎖定的公司 relay。
4. **讀取層由私網把關**：allowlist 只擋「發布」；由於無 AUTH 無法在協定層辨識訂閱者身分。封閉節點的「不被外部連上」由企業**自架於私網/VPN/防火牆後**達成（企業本就掌控可達性）。且即使被連上，取到的皆為 **E2E 密文**（無金鑰無法解），洩漏面僅限元資料。
5. **範疇**：**內部成員限定、刻意不聯邦化**。外部客戶非使用者；跨企業互通不在此模式範圍（那需開放中繼或多座，與「單節點」互斥）。

## 理由

- **最小、隱私中立**：allowlist 純粹「是否為成員」的內容層閘門，不需伺服器讀明文、不新增狀態、不引入 AUTH 交握；集中於 `relay-core.ts` 一處，20 項 relay 測試綠。
- **符合既有哲學**：拒絕語意與「移除成員即下次不放行」一致；`blocked:` 前綴讓 Outbox 歸類為永久失敗。
- **自架幾乎免費**：核心已與 Cloudflare 解耦，只差一個 WS 外殼。
- **偏好 allowlist 而非 NIP-42**：實作與維運更簡單；讀取層改由企業掌控的網路邊界把關，對「內部限定」場景足夠。

## 後果

- 正面：企業可在自有硬體跑封閉節點，只有名冊成員能發訊/上線，外部客戶天然不在系統內；資料落地與保留期完全內控。
- 負面 / 已知限制：
  - allowlist 只擋發布；**讀取層依賴私網部署**（非協定強制）。若未來需協定層讀取管制，再評估 NIP-42 AUTH（另立 ADR）。
  - 成員名冊的**佈建/更新**（新增/離職）需管理工具與名冊分發機制——本 ADR 只定核心閘門，佈建為後續。
  - 封閉即不聯邦：跨企業通訊不支援（設計如此）。
  - **法遵歸檔/eDiscovery/DLP** 仍與 E2E 根本衝突，**不在本模式**；若客戶要求須另立 ADR 並很可能是獨立「受監督版」（涉及金鑰託管/端點擷取的隱私取捨）。
- 後續行動：客戶端企業建置旗標（鎖單座、關漫遊）；成員名冊佈建工具（產生/指派 npub、分發 allowlist）；自架 WS 外殼與部署文件；評估企業強制 TURN（P2P IP 揭露）。
