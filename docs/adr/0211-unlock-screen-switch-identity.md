# 0211. 解鎖畫面加「用其他身分登入」入口

- 狀態：已接受
- 日期：2026-07-20
- 相關文件：ADR-0045（多身分登錄）、ADR-0067（本地密碼／隱藏身分）、ADR-0112/0122（瀏覽器本地密碼）、ADR-0146（登入畫面以名稱解析身分）、ADR-0201（軟登出 clearActive）

## 背景與問題

使用者問：「瀏覽器版是否可以比照桌面版，根據輸入的顯示名稱與密碼來切換要使用的私鑰？」

先釐清現況，避免平行發明新機制（SSOT / Fix First）：

- **依「名稱」選私鑰的邏輯早已存在，且桌面與瀏覽器共用同一份程式碼**——`packages/engine/src/storage/profiles.ts` 的 `resolveSignIn(name)`：顯示名稱只是「選哪個既有身分（pubkey）」的查找鍵（非加密鍵），命中既有＝切為作用中並解鎖，無命中＝建新身分（ADR-0146）。
- **密碼不選鍵，只解鎖。** 瀏覽器以 Argon2id 把「那一把 nsec」包成密文（`browserPassEnable(pubkey, nsec, password)`，ADR-0112）；密碼只能解開**已選定**的那把，本身無法分辨要哪一把。唯一「用密碼試出身分」的是隱藏身分的「解鎖隱藏身分」逐一試密碼（O(n)，且目前僅接桌面 `passUnlock`）。

真正的落差在**開機閘門的落點**：瀏覽器一旦記住「作用中身分」，開機直接落在**那一個**身分的 `UnlockScreen`（名稱＋密碼欄），畫面上**沒有**「改用其他身分登入」的入口——桌面亦然（`UnlockScreen` 為共用元件，僅 `onUnlock`/`onRescue`）。要在解鎖前換一把私鑰，現行唯一路徑是先「登出」(`clearActive`) 回登入頁、打另一個名稱、再解鎖。此外還衍生一個 UX 死角：只有單一鎖定身分的裝置，在解鎖前**無法**新建第二個身分（解鎖畫面沒有任何往登入頁的出口）。

## 考量的選項

- **選項 A（採用）：在 `UnlockScreen` 加「用其他身分登入」入口。** 點擊＝`clearActive`（軟登出、保留所有身分、不重載）回 `SignIn` 的名稱欄，讓使用者以顯示名稱挑另一把私鑰。維持「名稱選、密碼解」模型，桌面／瀏覽器一致。
- 選項 B：把隱藏身分那套「單一密碼欄、逐一試出是哪把」擴到一般身分。缺點：O(n)、密碼可能誤中多把、且會洩漏「有某身分用此密碼」；與「名稱與密碼」的訴求不符（那是名稱＋密碼，不是密碼獨選）。僅保留給隱藏身分。
- 選項 C：僅補齊瀏覽器版「解鎖隱藏身分」。屬另一議題，非本次訴求（使用者明確要「名稱＋密碼」）。

## 決策

採用選項 A。

- `UnlockScreen` 新增可選 `onSwitch?: () => void`；提供時渲染 `data-testid="unlock-switch"` 的「用其他身分登入」逃生口（沿用 `settings__reveal` 樣式，比照既有「忘記密碼？」）。
- `App.switchIdentity()`＝`clearActive(profilesState)` → `saveProfiles` → `setProfilesState` → `setLockedProfile(null)`。等同軟登出（ADR-0201）但**不重載**，直接回落 `SignIn`（因作用中為 null、`lockedProfile` 已清）。
- i18n 新增 `unlock_switch`（繁中「用其他身分登入」／英文「Sign in as someone else」）。
- 桌面與瀏覽器共用此入口（`UnlockScreen` 為共用元件），行為對稱。

## 理由

- **Fix First / SSOT**：不新增私鑰選擇機制——名稱選鍵的 `resolveSignIn` 與軟登出的 `clearActive` 都已存在，本 ADR 只補一個把兩者接起來的 UI 逃生口。
- **安全模型不變**：密碼仍只解鎖已選定的那把 nsec；名稱仍是選擇器。不引入「密碼獨選」的誤中與資訊洩漏風險。
- **零伺服器狀態**：全程純本機——`clearActive` 只改本機登錄的 `active` 欄，不刪任何身分、不與中繼互動。
- **順帶解死角**：單一鎖定身分的裝置，現在可在解鎖前經此入口回登入頁新建第二個身分。

## 後果

- 正面：瀏覽器（與桌面）可在解鎖畫面直接切換要用的私鑰，符合「依名稱＋密碼切換」的心智模型；消除單一鎖定身分無法新建身分的 UX 死角。
- 負面 / 已知殘餘風險：此入口回登入頁後，若使用者只是想「重新輸入密碼」，多了一個可能誤按的按鈕——以文案「用其他身分登入」與次要樣式（`settings__reveal`）降低誤解。
- 後續行動 / 待辦：瀏覽器版「解鎖隱藏身分」（目前僅桌面 `passUnlock`）仍待補（選項 C），與本 ADR 無耦合，另案處理。
