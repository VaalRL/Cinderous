# 0126. 保留上限＝封存而非刪除——統一兩個各說各話的「上限」

- 狀態：已接受（已實作）
- 日期：2026-07-15
- 相關文件：**ADR-0094（每對話保留上限）**、**ADR-0111（冷熱分離／封存）**、
  0119（健檢：本問題於該次發現）、0110（狀態常駐記憶體）

## 背景與問題

系統裡有**兩個**「每對話上限」，語意打架：

| | 誰設 | 觸發 | 行為 |
|---|---|---|---|
| **`maxPerConvo`**（ADR-0094） | 使用者（設定 UI，presets 0/1000/5000/10000） | `cap()` | **`splice()` 刪除** |
| **`HOT_CAP`**（ADR-0111） | 內部固定 5000 | `ArchiveWriter` | **封存**（移入 OPFS/塊檔，不刪） |

ADR-0111 的檔頭寫著「這同時修好一個既有的資料遺失：ADR-0094 的保留上限…現在是『封存而非刪除』」。
**那句話是錯的。** ADR-0111 只讓**內部的 HOT_CAP 溢出**走封存；使用者在設定裡設的 `maxPerConvo`
**仍然是真的 `splice()` 刪除**（ADR-0119 抓到並記錄了這個過度宣稱）。

而且更糟：如果使用者把上限設成 1000（< HOT_CAP），`cap()` 會先把對話刪到只剩 1000，
於是**永遠不會達到 HOT_CAP 的封存門檻**——封存機制根本沒被觸發過，全程是純刪除。

### UI 對使用者說的是「刪除」

```
每個對話在本機最多保留幾則；**超過即刪最舊**。
⚠ 本機儲存空間已滿…建議設一個保留上限。
```

問題是那個「儲存空間已滿」是 **localStorage 配額**爆掉（`QuotaExceededError`）。而 localStorage
在 web/mobile 只有 ~5–10MB，OPFS 封存卻常有 GB 級空間。**把溢出移進 OPFS 封存，正好解決那個
配額問題**——不需要刪任何東西。

## 決策

### 1. 保留上限＝**熱區大小**；溢出**封存**，永不刪除

統一兩個上限：使用者的 `maxPerConvo` 就是**封存的有效熱區門檻**。

```ts
// ArchiveWriter.run()
const hotCap = this.mem.retentionCap() > 0 ? this.mem.retentionCap() : this.hotCap; // 使用者設就用他的，否則 HOT_CAP
while (this.mem.loadMessages(convo).length >= hotCap + ARCHIVE_CHUNK) {
  … 先寫封存 → 成功後才裁切熱區（順序是正確性紅線，ADR-0111 不變）
}
```

- 設 1000 → 熱區保留 ~1000–2000，其餘進封存（**歷史紀錄仍可讀**）。
- 設 10000 → 熱區保留 ~10000，其餘進封存。
- 設 0（無上限）→ 沿用 HOT_CAP=5000 的內部封存（行為不變）。

`cap()` 的 `splice()` **刪除移除**。保留上限不再刪任何一則訊息。

### 2. web/mobile 的真正好處：**紓解 localStorage 配額**

熱區在 localStorage（同步、小），封存在 OPFS（非同步、大）。把溢出從 localStorage 移到 OPFS，
**localStorage 佔用直接下降**——這正是「儲存空間已滿」警告想解決的事。總位元組沒變少，
但**擠爆的那個池子**（localStorage）鬆了。桌面則是熱區 blob 檔縮短、封存另存加密塊檔。

### 3. 🔴 沒有封存就**不裁切**（ADR-0111 的紅線，本 ADR 延續）

若封存基質不可用（CLI、或瀏覽器無 OPFS），`writer` 是 undefined → 保留上限**不生效**
（訊息累積，不刪）。**絕不讓「封存不可用」變成「訊息被刪掉」。** 桌面／web／mobile 都會掛上封存，
所以實務上上限一律生效；CLI 沒有這個設定 UI。

### 4. 調整上限要**立即**重新封存

把上限從 5000 調到 1000，不能等下一則訊息才生效。`setMaxPerConvo` 之後對所有對話
`schedule()` 一次封存搬移——溢出即刻進封存。

### 5. 更新 UI 文案：不再說「刪除」

```
每個對話在快取區保留幾則；更舊的**移到封存**（歷史紀錄仍可讀），不刪除。
網頁版空間有限，設一個上限可把舊訊息移出 localStorage、紓解配額。
```

## 理由

- 這是**誠實兌現 ADR-0111 檔頭那句話**——它宣稱「封存而非刪除」，但只對內部 HOT_CAP 成立；
  使用者摸得到的那個上限一直在刪。ADR-0119 抓到了，本 ADR 修掉。
- 而兩個上限本來就該是**同一個概念**：「快取區保留多少」。把 `maxPerConvo` 接成 ArchiveWriter
  的有效門檻，HOT_CAP 退化為「使用者沒設時的預設」——沒有平行機制。
- 「省空間」的訴求在 web/mobile 由「移出 localStorage」達成；真要回收總空間是另一件事
  （明確的「清除歷史」動作），不該用一個看起來像「保留設定」的東西偷偷刪掉歷史。

## 後果

- 正面：
  - 使用者設保留上限**不再靜默刪掉歷史**——舊訊息進封存，歷史紀錄讀得到。
  - localStorage 配額壓力真的被紓解（移進 OPFS），且**符合**「儲存空間已滿」警告的本意。
  - 兩個上限收斂成一個概念，HOT_CAP 成為預設值；無平行路徑。
  - 測試 1011 → **1014**（archive.test +4：小上限溢出**進封存不刪除**、調低上限即刻重新封存、
    封存搬移清 id 索引、無上限沿用 HOT_CAP；memory/local 的舊「刪除」測試改寫為「無封存不裁切」）。

- 已知限制：
  - **總位元組不因設上限而變少**（封存仍持有）。要真正回收空間需另做「清除歷史／清空封存」
    的明確動作——本 ADR 不含。
  - 上限是**近似**的：熱區在 `[cap, cap+ARCHIVE_CHUNK)` 間浮動（封存以固定 1000 則為一塊，
    換取分頁單純，ADR-0111）。使用者設 1000 實際熱區可能到 2000。
  - 自訂上限若非 1000 的倍數，封存塊界線與它不對齊——行為正確（仍以 1000 為塊），只是熱區
    停在最接近的塊界。
