# 0128. 檔案 IPC 縱深防禦——收檔檔名消毒 ＋ 讀檔路徑白名單

- 狀態：已接受（已實作）
- 日期：2026-07-15
- 相關文件：**ADR-0093（收檔另存）**、0102（讀回原檔）、0103（原生選檔）、0119（健檢：本問題於該次發現）、
  0122（安全網：別做危險的預設）

## 背景與問題

健檢（ADR-0119）在桌面原生橋（Rust IPC）發現兩個縱深防禦缺口。**目前都沒有已知的可觸發
XSS**——但這兩個 IPC 是「一旦頁面內有惡意 JS 就會被放大成嚴重後果」的原語，該在破口出現前補上。

### 🟠 1. 收到的檔名**未消毒**就進原生另存對話框（遠端可控）

```rust
fn save_file(name: String, bytes: Vec<u8>) -> … {
    rfd::FileDialog::new().set_file_name(&name).save_file()  // ← name 來自對方
```

`name` 一路從**對方傳來的檔案 metadata**（`file` tag：名稱/大小/類型）流到這裡。對方是遠端、
不可信。惡意檔名可以是 `../../../../evil`、帶控制字元、或幾 KB 長——`set_file_name` 會把它
當預填檔名。使用者仍會看到對話框並確認，但預填一個穿越路徑或詭異檔名是不該發生的。

### 🟠 2. `read_saved_file` 是**無限制的任意檔案讀取**原語

```rust
fn read_saved_file(path: String) -> Result<Option<Vec<u8>>, String> {
    std::fs::read(p)  // ← path 是什麼就讀什麼
```

目前三個呼叫端（選檔送出、讀回原檔、重新指定位置）的路徑**都來自原生對話框或已存的
savedPath**（本地控制）——所以**沒有已知的遠端注入**。但這個 IPC 對整個 webview 開放：
**一旦有 XSS**，惡意 JS 就能 `invoke("read_saved_file", { path: "…/.ssh/id_rsa" })` 讀走行程
能讀的**任何檔案**並外傳。這是「安靜地做危險的事」（ADR-0122 的主題）的一個未爆彈。

## 決策

### 1. 檔名消毒：`sanitize_filename()`（純函式，放 lib，有測試）

進原生對話框前，把檔名收斂成**乾淨的 basename**：

- 只取最後一段（丟掉所有 `/`、`\` 之前的東西）→ 杜絕路徑穿越；
- 移除控制字元與 Windows 保留字元（`< > : " | ? *`）；
- 去掉開頭的 `.`（避免 `..`／隱藏檔）與前後空白；
- 長度上限（255）；空字串退回 `"file"`。

放進 `partfile.rs`（ADR-0119 已把檔案安全原語搬進 lib，因為 bin target 的 `cargo test`
編不到——這裡延續同一個原則：**安全關鍵的純函式一律可測**）。`save_file` 與 `pick_existing_file`
都先消毒 `name` 再 `set_file_name`。

### 2. 讀檔路徑白名單：只讀「原生對話框授權過的路徑」

`read_saved_file` 不再讀任意路徑，只讀**經由原生對話框授權過**的路徑：

- `save_file`（使用者選了存檔位置）與 `pick_existing_file`（使用者選了要讀的檔）**在成功後把
  該路徑加入白名單**。
- `read_saved_file` 只在路徑於白名單內時才讀，否則回 `Ok(None)`（＝「檔案不存在」的既有語意，
  UI 自然走「重新指定位置」，不給攻擊者可用的訊號）。

白名單**持久化**（存 app_data），所以 ADR-0102 的「跨 session 點縮圖讀回原檔」照常可用——
不因這道防禦而每次重開都要重新指定。白名單存的是**路徑的 SHA-256 雜湊**，不是路徑本身：
避免在一個明文檔案裡洩漏使用者的檔名（那些路徑在訊息儲存裡本來是加密的）。

於是：**使用者透過原生對話框授權過的檔案讀得到；XSS 憑空給的任意路徑讀不到。**

## 理由

- 這兩個都是**縱深防禦**（無已知 XSS）。但它們的形狀正是 ADR-0119/0122 反覆講的：
  一個原語安靜地做了它不該做的事（讀任意檔、預填任意檔名），平常沒事，破口一出現就致命。
- 白名單用**授權過的路徑**而非「限定某個目錄」——因為合法用途本來就是「讀使用者選的任意檔」，
  限定目錄會擋掉真實用法。關鍵不是「檔案在哪」，而是「**這個路徑是不是使用者透過原生對話框
  親自選的**」。
- 雜湊而非明文存路徑：這道防禦不該自己變成一個新的資訊洩漏（明文檔名清單）。

## 後果

- 正面：
  - 對方無法用惡意檔名預填你的另存對話框（穿越／控制字元／超長）。
  - `read_saved_file` 從「任意檔案讀取」收斂為「只讀你親自授權過的檔」——XSS 讀不到系統檔案。
  - 白名單持久化 → ADR-0102 的跨 session 讀原檔 UX 不受影響。
  - `sanitize_filename` 是純函式、在 lib、有 Rust 測試（延續 ADR-0119）；瀏覽器側對應的 JS `sanitizeFilename` 亦有測試。
  - 測試 1023 → **1029**（desktop JS +6：路徑穿越只留 basename、控制/保留字元、開頭點、清光退預設、超長截斷）；Rust 16 → **17**（`sanitize_filename` 同上）。

- 已知限制：
  - **升級前已存的 savedPath 不在白名單裡** → 升級後第一次讀那些舊原檔會回「missing」，走一次
    「重新指定位置」重新授權（之後就記住）。一次性成本。
  - 白名單是 app 範圍、append-only 的雜湊清單（載入時去重）；不隨身分命名空間隔離——它只證明
    「這個路徑被原生對話框授權過」，不含任何內容或身分資訊。
  - 瀏覽器路徑（`<a download>`）由瀏覽器自行消毒檔名；另加一道輕量 JS 消毒以求一致。
