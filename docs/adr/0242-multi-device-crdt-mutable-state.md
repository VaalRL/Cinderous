# 0242. 多設備可變狀態的 CRDT 化：把臨時合併規則系統化

- 狀態：已接受（方向與分階段計畫；實作待排、逐階段觸發）
- 日期：2026-07-23
- 相關文件：ADR-0009／0107（多設備自封同步）、ADR-0071（加密雲端快照）、ADR-0108（已讀水位 LWW）、
  ADR-0224（資產庫 LWW＋墓碑）、ADR-0217（每對話靜音）、`packages/engine/src/storage/cloud-snapshot.ts`、
  `ARCHITECTURE.md §4／§9`

## 背景與問題

同一身分可在多台裝置使用，各裝置**可能離線各自編輯**、之後才碰面同步（自封 NIP-17 事件 ADR-0107、
加密雲端快照 ADR-0071）。難點是：**兩邊改的東西矛盾時，合併成什麼、且所有裝置能否收斂到同一結果。**

現況是**臨時、add-biased 的合併**（`mergeSnapshotContent`）：

| 狀態 | 現行合併 | 型別 | 缺口 |
| --- | --- | --- | --- |
| 訊息 | id 去重 | G-Set ✅ | 無（append-only，天生無衝突） |
| 已讀水位 | 取最新（LWW） | LWW-max ✅ | 無（ADR-0108） |
| 自訂資產 | LWW＋墓碑 | OR-Set ✅ | 無（ADR-0224，刪除傳播、重匯復活） |
| **聯絡人** | **補缺不覆蓋** | add-only ❌ | **刪除復活**（一台移除→另一台補回）；**欄位編輯不傳播**（暱稱/標籤/hint 改了，對方若已有此聯絡人就跳過→收不到） |
| **群組** | **補缺** | add-only ❌ | **離群復活**（一台離開→另一台補回） |
| **封鎖清單** | **聯集** | G-Set ❌ | **解封不傳播**（聯集只增不減） |
| **設定/偏好** | **未同步** | — ❌ | 每對話靜音（ADR-0217）、狀態文字、主題、通知開關等**根本不跨裝置**，各台各自一份 |

**根因**：`add-biased`（補缺/聯集）只處理「新增」，不處理「刪除」與「並發欄位編輯」。每個新功能各自
發明合併規則——資產踩到「emoji 刪了又復活」才補墓碑（ADR-0224），聯絡人/群組/封鎖/設定仍是坑。

## 考量的選項

- **選項 0：維持現狀（臨時打補丁）。** 每功能各自處理，坑一個個冒（刪除復活、解封失效、欄位不同步）。
- **選項 1（推薦）：輕量、系統化的 CRDT 規則。** 把 ADR-0224 已驗證的「LWW＋墓碑」模式**推廣**到所有
  可變狀態，用最小一組 CRDT 積木（G-Set／LWW-Register／OR-Set-with-tombstone），不引入重框架。
- **選項 2：重量級 CRDT 框架（Automerge／Yjs 等）。** 完整因果排序與自動合併，但**撞快照 180KB 明文
  預算**（ADR-0071）、需帶版本向量與大量 metadata、與現有自封同步/快照格式不相容——過度工程。

## 決策

**採選項 1：以最小一組 CRDT 積木統一多設備合併。** 每種狀態對應一個型別：

- **G-Set（只增集合）**：訊息（id 去重）——**已完成**。
- **LWW-max（單調暫存器）**：已讀水位——**已完成**（ADR-0108）。
- **OR-Set with tombstone（可增可刪集合）**：**聯絡人、群組、封鎖清單**。加入即成員、移除留**墓碑**
  （帶時間）使刪除**傳得出去且蓋過過期的新增**；重加自動復活（同 ADR-0224 資產）。→ 修「刪除/離群
  復活、解封不傳播」。
- **LWW-Register per-field（逐欄位取最新）**：聯絡人的**暱稱／標籤／relay hint**、以及**設定/偏好**
  （每對話靜音、狀態文字、主題、通知開關）。每筆帶一個更新時間戳，並發編輯取較新。→ 修「欄位編輯
  不傳播」與「設定不跨裝置」。

**分階段（逐坑觸發，不一次全做）：**
1. **聯絡人/群組/封鎖 → OR-Set＋墓碑**（複用 ADR-0224 的墓碑機制與交換律測試）。修最痛的「刪除復活/
   解封失效」。
2. **聯絡人欄位 → per-field LWW**（暱稱/標籤/hint）。
3. **設定/偏好納入同步 → LWW**（先每對話靜音與狀態文字，其餘漸進）。

## 理由

- **為何輕量、不上重框架。** 快照有 180KB 明文預算（ADR-0071）；Automerge/Yjs 的版本向量與 op-log
  metadata 會吃爆它，且要重寫同步格式。而 Cinderous 的狀態多是「集合＋少數欄位」，用 G-Set／
  LWW-Register／OR-Set 三塊積木就夠——**只帶墓碑與 per-record 時間戳**，metadata 增量可控。
- **為何推廣 ADR-0224 而非重造。** 資產的「LWW＋墓碑＋交換律合併＋重匯復活」已實作、已測、已在產線；
  把同一模式套到聯絡人/群組/封鎖，是延伸不是新發明（符合 Fix-First）。
- **為何分階段、逐坑觸發。** 這是**正確性/品質**問題，非阻斷性——現況大致能動。先修「刪除復活/解封」
  （使用者最可能踩、且是資料正確性），欄位與設定同步漸進。與 ADR-0224 由 emoji 逼出同一節奏。
- **墓碑要能回收（GC）。** 墓碑無限累積會撐大快照。以「墓碑帶時間、超過保留窗（如對齊快照/離線
  留言的天數）即可 GC」收斂大小——但 GC 窗要 ≥ 裝置可能離線的最長期，否則久未上線的裝置會漏掉刪除。

## 後果

- 正面：
  - 多設備合併**收斂有保證**：刪除傳播、解封生效、欄位/設定跨裝置一致；不再每功能各踩一次坑。
  - 複用 ADR-0224 已驗證的墓碑/交換律，延伸而非重造。
  - 分階段：先修最痛的正確性坑，其餘漸進，風險可控。
- 負面 / 已知殘餘風險：
  - **metadata 成本**：墓碑與 per-field 時間戳增大快照，需 GC；GC 窗與「裝置最長離線期」的張力
    （窗太短→久離線裝置漏刪除；太長→墓碑膨脹）。
  - **設定納入同步**改變現況（目前設定是裝置本地）——要小心「某些設定本就該裝置本地」（如該裝置的
    通知音量），別一律同步。逐項判定。
  - 仍非因果一致（無版本向量）：極端的「同一欄位在兩台以毫秒級並發改」只能 LWW 取一，會丟另一個
    ——對聯絡人暱稱/設定可接受。
- 後續行動 / 待辦：
  1. 階段 1：`cloud-snapshot` 的聯絡人/群組/封鎖改 OR-Set＋墓碑（複用 `mergeAssetLibrary` 模式）；TDD
     測「刪除傳播、重加復活、解封生效、交換律（合併順序無關）」。
  2. 階段 2：聯絡人欄位 per-field LWW（暱稱/標籤/hint 帶時間戳）。
  3. 階段 3：設定/偏好逐項納入 LWW 同步（先每對話靜音、狀態文字；排除「該裝置本地」項）。
  4. 墓碑 GC：帶時間、保留窗 ≥ 合理最長離線期，超窗回收。
