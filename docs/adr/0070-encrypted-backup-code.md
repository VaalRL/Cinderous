# 0070. 加密備份碼（使用者自持的密碼加密身分備份）

- 狀態：提議中
- 日期：2026-07-09
- 相關文件：docs/adr/0015（身分備份）、0053（OS 金鑰庫）、0067（本地密碼／nsec 定位）、
  0066（搬家）、apps/desktop/src-tauri/src/passlock.rs、apps/desktop/src/ui/SettingsPanel.tsx

## 背景與問題

1. 備份現況是**抄明文 nsec**（ADR-0015）：使用者最後一道防線是一串明文，
   常見下場是躺在記事本／截圖／剪貼簿歷史裡。
2. 跨裝置討論的結論（記錄於此）：「名稱＋密碼」不可能在新裝置還原身分——
   - **腦錢包**（`私鑰 = KDF(名稱+密碼)`）否決：離線暴力枚舉可直接偷身分、
     同名同密碼互撞。
   - **伺服器存可查詢加密備份**否決：以 `hash(名稱+密碼)` 索引＝把 relay 變成
     密碼猜測神諭，弱密碼＝身分被偷，且引入偽帳號命名空間。
3. 正確的跨裝置素材是「nsec＋relay 網址」（名稱僅顯示標籤、本地密碼不跨裝置）——
   值得把這組素材包成一份**使用者自持**的加密備份。

## 考量的選項

- 選項 A：維持明文 nsec 備份。
- 選項 B：**加密備份碼**（建議採用）——密碼加密的 nsec＋relay 網址，使用者自持。
- 選項 C：伺服器存放可查詢備份——否決（見背景 2）。
- 選項 D：腦錢包——否決（見背景 2）。

## 決策（提議）

1. **備份碼**＝以使用者自訂密碼（Argon2id，重用 passlock 的 KDF 與 AES-256-GCM
   包裹格式）加密的 `{ nsec, relayUrl }`，序列化為單一字串與 QR 圖。
2. **產生**：設定面板「身分備份」區塊升級——除現有明文顯示外，新增「產生加密
   備份碼」（輸入備份密碼→輸出字串/QR 供列印、存檔到使用者自選位置）。
3. **還原**：SignIn／新增身分支援「貼上備份碼（或掃 QR）＋輸入備份密碼」→
   還原 nsec 與 home relay，走既有匯入流程。
4. **密文使用者自持**：不上雲、不發佈到 relay、不進 repo——沒有可公開查詢的
   密文就沒有神諭問題；弱密碼的暴露面僅限「同時拿到備份碼的人」。
5. 開放問題（實作前定案）：格式是否對齊 **NIP-49（ncryptsec）**以取得跨客戶端
   互通（scrypt＋XChaCha20），或沿用自家 passlock 格式（程式碼重用、含 relay 欄）。
   傾向：nsec 部分用 NIP-49、外層再包 relay 欄位，兼得互通與完整還原。

## 理由

- 「名稱＋密碼＋relay＋私鑰」的使用者直覺，最安全的實作形態就是備份碼：
  密碼真正參與加密（同 ADR-0067 原則）、密文不落任何可查詢位置。
- 重用 passlock 的 Argon2id/AES-GCM 與 QR 產生（ADR-0024）——新增面積小。

## 後果

- 正面：最後一道防線不再是明文；換機救援體驗變成「掃碼＋密碼」。
- 負面／已知殘餘風險：忘記備份密碼＝該份備份失效（明文 nsec 備份仍為可選後備，
  文案須並陳）；備份碼與密碼同時外洩＝身分被偷（與明文 nsec 外洩同級，但多一道
  Argon2id 暴力成本）。
- 後續行動 / 待辦：接受後施工——Rust 端重用 passlock、前端備份/還原 UI、QR 產生
  重用 qr.ts、i18n 文案；ADR-0015 屆時標註被本 ADR 部分取代（明文備份降為後備）。
