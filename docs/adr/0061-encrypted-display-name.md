# 0061. 顯示名稱以加密個人檔廣播給聯絡人（非公開 kind 0）

- 狀態：已接受
- 日期：2026-07-08
- 相關文件：PRD §4/§6（隱私、本地優先）；ADR-0002（隱私基線）、0034（多中繼路由）；ARCHITECTURE §9
- 觸發：聯絡人清單目前顯示 `npub`／本地 shortNpub，非對方自選暱稱。

## 背景與問題

要讓聯絡人看到對方**自選的顯示名稱**。Nostr 標準做法是公開 **kind 0 個人檔**——但那會把「名字 ↔ npub」的對映**公開給中繼站與任何人**，違反 Cinder「明文/元資料最小化」的隱私鐵則（Gift Wrap 的用意就是連社交圖譜都藏）。

## 決策

**顯示名稱以「加密 Gift Wrap」只廣播給聯絡人，而非公開 kind 0。**

1. **傳輸：** `wrapProfile(name, sk, recipientPk)`——rumor 為 kind 0、content `{name}`（NIP-01 子集），包進 kind 1059 Gift Wrap（藏寄件人、E2E）。中繼站只見密文。
2. **何時送：** 加好友時送給對方（`addContact` → `sendProfileTo`）＋開機廣播給所有聯絡人（讓既有聯絡人也學到）。
3. **互惠（單向加即雙向得名）：** 收到對方個人檔時，若我**尚未送過**個人檔給對方（例如對方單向加我），回送一次。以 `profileSentTo` 集合**每對象至多一次**防止來回反覆（ping-pong）。
4. **接收：** 解出 `parseProfile` → `updateContactName` 更新聯絡人顯示名稱（僅在變動時 emit）。收到陌生人個人檔會先 `ensureContact` 再更新。
5. **不做公開 kind 0：** 犧牲與一般 Nostr 客戶端的互通，換取隱私一致性（Cinder 為自足生態）。

## 理由

- **隱私一致**：名字只有聯絡人看得到，中繼看密文——與訊息同級保護。
- **符合既有機制**：完全重用 Gift Wrap／`sealAndWrap`／收訊分派，與 reactions/receipts 同源。
- **單向加即得名**：互惠回送讓「A 加 B」就能讓雙方都顯示對方暱稱，符合直覺。

## 後果

- 正面：聯絡人顯示自選暱稱、隱私不破。
- 負面 / 已知限制：
  - 與公開 kind 0 生態不互通（刻意）。
  - 開機廣播給所有聯絡人有少量流量（gift wrap，7 天過期＋每收件人上限保護）；名字很少變，未來可優化為「僅變更時送」。
  - 目前沒有 in-app 改名；名稱固定於登入。日後加改名時，改名即重新廣播（`profileSentTo` 需清除以便重送）。
- 測試：core `profile`（wrap/parse round-trip、壞輸入）＋後端「單向加好友即互換暱稱」整合測試。
