# 0032. 自製貼圖：內容隨訊息、SVG 統一表示、點擊即擁有

- 狀態：已接受
- 日期：2026-07-02
- 相關文件：docs/adr/0021（貼圖）、0031（動態貼圖）、apps/desktop/src/stickers.ts

## 背景與問題

在內建貼圖（0021/0031）之上加入「自製貼圖」，需求：

1. 允許匯入 SVG（不只點陣圖）。
2. **「被展示過且點擊過＝擁有」**：收到他人自製貼圖，點擊即可收藏到自己的貼圖庫。
3. 管理功能：刪除、新增自製、從現有貼圖 fork。
4. 貼圖「編輯與製作」是否限制桌面版（見文末平台政策）。

需求 2 是設計支點：收端要能收藏，**貼圖內容必須隨訊息送達**，僅送參照
（先前評估傾向的 P2P 檔案通道方案）無法滿足——收端沒有位元組可收藏，
且 P2P 需雙方在線、群組扇出困難。

## 考量的選項

- **送達**：(a) P2P 檔案通道送參照＋位元組；(b) **內容內嵌於加密訊息**（Gift Wrap 原文即帶 SVG）；(c) 內容定址＋快取層。
- **內部表示**：(i) SVG 與 raster 並存；(ii) **一律 SVG**（raster 匯入時包進 `<image href="data:image/...">`）。
- **SVG 安全**：(x) 清洗（strip）；(y) **驗證拒收（reject）**——含危險樣式即整張拒絕。
- **識別**：(p) 送端自報 id；(q) **收端以內容雜湊自算 id**（sha256，取自 core）。

## 決策

- **送達採 (b)**：新增線格式 `nb-sticker:v2:<JSON>`，JSON 為 `{"label","svg"}`，
  隨既有 NIP-17/59 加密訊息送出。SVG 原始碼上限 **32 KB**（NIP-44 明文上限
  65535 bytes 的安全餘裕內）；離線送達、群組扇出、持久化/收回/限時全部自然沿用。
  v1 內建參照格式不變；舊客戶端看到 v2 顯示原文（與 0021 相同的優雅退化）。
- **表示採 (ii) 一律 SVG**：raster 匯入時由 canvas 縮至 256px、重編碼後包進 SVG
  `<image>`。fork、收藏、渲染共用同一資料形。
- **安全採 (y) 驗證拒收**（`validateStickerSvg`，純函式）：必須以 `<svg` 開頭、
  ≤32KB，且**不得**含 `<script`、事件屬性（`on*=`）、`javascript:`、
  `<foreignObject`、外部參照（`http(s)`/`//` 的 href、`url(http…)`）；`data:` href
  僅允許 `data:image/*`。套用時機：匯入、fork、**收端渲染前**、點擊收藏時。
  渲染一律 `<img src=dataURI>`（`<img>` 內 SVG 的 JS 與外部載入本已停用）——
  驗證為縱深防禦第二層；**永不** `dangerouslySetInnerHTML`。
- **識別採 (q)**：線上不傳 id；收端與本地一律以 `contentHash(svg)`（sha256 hex，
  core 出口）自算，天然去重、fork 改內容即新 id，杜絕「自報 id 污染」。
- **擁有語意**：對話中渲染出的自製貼圖（＝被展示過）＋使用者點擊 → 寫入本地
  貼圖庫（localStorage `nb.stickers.custom`，上限 **32 張**、逐張 ≤32KB）。
  自己的貼圖庫可刪除、可自 SVG 檔匯入、可自 raster 匯入、可 fork 任何
  內建或已擁有貼圖。庫內貼圖以虛擬包 `__custom` 融入最近使用/我的最愛。

## 平台政策：貼圖「編輯與製作」是否限桌面版（需求 4 研究結論）

- 匯入/刪除/fork/收藏等**管理功能為純 Web 技術**（檔案選取、canvas、localStorage），
  web 與 Tauri 桌面共用同一前端，**技術上無差異、不設人為限制**。
- 未來「編輯器」（畫布繪製、路徑編修）同樣是純 Web 技術，但**精細指標（滑鼠/手寫筆）
  與螢幕空間**是體驗前提；行動版（Phase D 規劃為 React Native）為**另一份 codebase**，
  移植編輯器是獨立工程且小螢幕體驗差。
- **結論：採「桌面優先」而非「桌面限定」**——共用前端內不做建置層閹割（無收益）；
  行動版首發不移植編輯器（僅保留收藏/使用/刪除）；若需在共用前端內降級，以能力偵測
  （`pointer: fine`、視窗尺寸）而非平台判斷。

## 理由

- 內容隨訊息是唯一同時滿足「離線、群組、點擊即擁有」的路徑；32KB 上限使其
  安全地落在既有加密事件內，零新協定通道。
- 拒收比清洗簡單且不易被繞過（無「清洗後仍殘留」的變形空間）；配合 `<img>`
  的天然沙箱構成兩層防禦。
- 內容雜湊識別去除信任問題，並讓 fork/去重語意自然成立。

## 後果

- 正面：零新依賴、零協定通道新增；管理/擁有全屬本地；群組與離線自然支援。
- 負面 / 已知殘餘風險：32KB 上限擋掉複雜插畫；validate 為樣式比對，可能誤殺
  含特殊字樣的合法 SVG（可接受，拒收有明確錯誤訊息）；貼圖內容隨每則訊息重送
  （無快取層），高頻重送同貼圖時流量非最優——留待未來內容定址快取（選項 c）。
- 後續行動：編輯器（畫布製作）另案；行動版範疇依平台政策執行。
