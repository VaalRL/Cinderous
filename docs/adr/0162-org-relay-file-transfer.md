# 0162. 組織檔案經 relay 暫存——加密分塊、企業限定、預設關、小上限

- 狀態：已接受（已實作）
- 日期：2026-07-16
- 相關文件：ADR-0093（檔案 P2P——本 ADR 為組織小檔案提供 relay 後備路徑）、
  0160（保留天數/`MAX_TTL_DAYS`——同一「企業站自己付成本」哲學）、0161（儲存槽）

## 背景與問題

P2P 檔案傳輸要求雙方同時在線；企業自架 relay 願意以自己的儲存換「離線也送得到檔」。
明文不上雲不是障礙（訊息本來就是密文走 relay），真正的牆：NIP-44 單則明文 64KB
（必須分塊）、密文膨脹 ×2、以及**離線信箱配額互踐**（檔案塊絕不能把聊天訊息擠出
500 則 FIFO）。使用者裁示：可開關、**預設關**、較小的檔案上限、只有企業方案可用。

## 決策

1. **獨立外層 kind `FILE_WRAP = 1060`**：檔案塊仍是 Gift Wrap 密文（中繼只見密文），
   但外層 kind 與聊天（1059）分開——中繼站因此能（a）**整類拒收**（未啟用時）、
   （b）給檔案塊**獨立配額桶**（不與聊天訊息互踐）。內層 rumor kind `FILE_CHUNK=43`，
   content＝`{tid, seq, total, name, mime, data(base64)}`，每塊明文 ≤48KB。
2. **中繼站雙閘（企業限定的真正執行點）**：
   - `MAX_FILE_MB` 環境變數（CF Worker wrangler var／node env）：未設＝**整類拒收
     1060**（公共站零影響）；≥1 才收。單顆 1060 事件另有 200KB 大小 sanity 上限。
   - 檔案塊獨立配額：每收件人 `filePerRecipient`（預設 4000 顆 ≈ 500MB 密文）FIFO，
     與聊天的 500 則完全分桶。塊事件的過期章走 ADR-0160 的保留政策/站方上限。
3. **名冊政策 `relayFilesMaxMb`（1–16，預設未設＝關）**：管理者設定後，成員與企業主
   對**名冊在世成員的 1:1** 送檔且大小 ≤ 上限時自動改走 relay 分塊（metadata 訊息
   照舊、跳過 P2P）；超過上限或非組織對話維持 P2P。群組檔案 v1 維持 P2P
   （×成員數的儲存膨脹另議）。
4. **收端重組**：僅接受**聯絡人**的塊（陌生人塊直接丟棄——relay 暫存是組織內功能）；
   併發重組 ≤8 個 tid、單檔 ≤16MB＋單塊 ≤48KB 防禦；收齊 → 走既有 `onFileBytes`
   收檔路徑（另存/縮圖流程照舊）。

## 理由

- 獨立 kind 讓「企業限定」由**部署者**執行（不是靠客戶端自律），也讓配額分桶不需
  破解密文——最小 metadata 揭露（中繼只多知道「這是檔案塊」，本來就從體積看得出）。
- 上限 16MB：NIP-44 分塊與 ×2 膨脹下，16MB ≈ 350 塊 ≈ 44MB 密文——夠日常文件，
  不足以把 relay 當網盤；大檔仍走 P2P（免費且無上限）。

## 後果

- 正面：組織內離線送檔成立（收件人 ≤ 保留期內上線即收）；公共站行為零變化。
- 已知限制：relay 路徑無傳輸進度條（外送匣節流背景送、送端直接標完成）；
  自己的其他裝置仍拿不到位元組（與 P2P 一致，metadata-only）；群組檔案未走 relay。
- 測試：core（分塊 wrap/parse/防禦、政策解析）＋relay（預設拒收 1060、啟用後收、
  檔案桶不踐踏聊天配額）＋engine（離線收件人上線後收齊檔案的端到端流）＋desktop
  （名冊欄位/預填）。
