# 0006. 心跳容量估算與免費額度策略

- 狀態：已接受
- 日期：2026-06-30
- 相關文件：PRD.md §8；ARCHITECTURE.md §6、§8；docs/adr/0005

## 背景與問題

ADR-0005 與 PRD §8 要求在 M1 前估算 Ephemeral 心跳對 Cloudflare 免費額度的消耗，據以決定心跳間隔與扇出策略。原 PRD 曾稱 Ephemeral「零消耗」，但實際仍消耗 Worker/Durable Object 的請求數。

## 估算（明確標示假設）

模型：單一 Durable Object 房間以 Hibernation WebSocket 保存連線；**每筆進入的 WebSocket 訊息約當一次計費請求**，扇出送出則為既有連線上的訊息。主要成本因子為「進入訊息數」。

- 心跳間隔 30 秒 → 每位上線使用者 `2 筆/分 × 1440 = 2,880 筆/日` 進入訊息。
- 免費層基準約 **100,000 請求/日**。
- 並行上線人數上限 ≈ `100,000 / 2,880 ≈ 34 人`（尚未計入「輸入中／音樂」事件，實際更低）。

> 結論：以 30 秒心跳，免費額度的天花板約在**數十位並行上線使用者**。扇出本身不是請求數瓶頸（同連線送出），瓶頸在「進入訊息數 = 上線人數 × 心跳頻率」。

## 決策

1. **M1 維持 30 秒心跳 / 60 秒判離線**：體驗優先，且免費額度足以支撐早期小規模。
2. **送出加入隨機抖動（jitter）**：心跳時間 `30s ± 數秒`，避免整點尖峰（於 T9/T11 客戶端實作）。
3. **明確記錄天花板**：~數十並行使用者；超過即需下列其一。
4. **擴充路徑（M3 後）**：
   - 心跳間隔可調為 60 秒（成本減半，離線判定改 120 秒）。
   - 合併上線/音樂/輸入中為單一彙整事件，降低筆數。
   - 已建立 WebRTC P2P 的好友改走 Data Channel 傳狀態，從中繼卸載。
   - 規模化時升級 Workers Paid（DO 額度大幅提高）。

## 後果

- 正面：M1 有明確、誠實的容量界線與可調旋鈕，不再宣稱「零消耗」。
- 負面 / 待驗證：上述 CF 計費換算為保守估計，需在 M1 實測校準；DO 在免費方案的實際可用性與限制需以部署時官方條款為準。
- 後續行動：T9/T11 心跳送出實作 jitter；M1 上線後實測請求數並回填校準本 ADR。
