# 0165. 依使用習慣的心跳時段降載（研究記錄）

## 背景與問題

能否依使用者使用習慣，在特定時段對心跳（presence 存活信標）進一步降載，以省電（手機
喚醒次數）與省中繼站免費額度？

## 現況（基準，ADR-0109/0129）

心跳已是自適應，降載接入點只有一個函式 `beatInterval()`：

- 有任一聯絡人在線 → **60 秒**（ACTIVE）；沒人在線 → **5 分鐘**（IDLE）。
- 全聯絡人皆有活的 P2P 通道 → **完全不經中繼**（`beat()` 直接 return）。
- 收心跳免費（中繼→客戶端不計費），成本只在「送」；ADR-0109 已把「92% 請求對空氣廣播」
  這個最大浪費用 IDLE-5 分鐘消除。
- 節奏**自報**（封包帶 `hb`，對方以 2.5×`hb` 判離線）→ **把心跳拉到 15/30 分鐘協定本來
  就支援**，只需改 `beatInterval()` 回傳值，零協定改動、零新事件。
- IDLE→ACTIVE **即時喚醒**：對方一上線立刻補一拍，故再慢/暫停也能一個 RTT 內恢復。

**降載的「插座」現成、協定零改動、恢復機制現成。**

## 兩種設計方案

### 方案 A：學習時段直方圖（原始提問的形態）

本機記每個小時桶的 EWMA：該時段歷史上（a）有無聯絡人在線、（b）我自己有無活動；兩者
趨近 0 的時段＝死區 → `beatInterval()` 於該時段回更長值（20–30 分鐘）或暫停。

- 純本機、零隱私面（只用既有本機訊號，不外送、不上中繼）。
- 代價：冷啟動（新用戶無歷史）、時區/出差漂移、平日 vs 週末、DST 等邊界；而它比 IDLE-5
  分鐘多省的，只是「沒人在線**且**歷史死時段」時再 ×4–6——在本就很小的 IDLE 負載上再打折。

### 方案 B：漸進式閒置退避（推薦，不用學習）

用**已有訊號**：「無任一聯絡人在線 > X 分鐘 **且** 自己閒置/上鎖 > Y 分鐘」→ IDLE 節奏
**漸進拉長**（5 → 10 → 20 分鐘，設上限），一有活動或有人上線立刻回 60 秒。

- 「裝置閒置＋沒人在線」本身就是死區訊號——狀態機直接讀得到，不需學出「凌晨三點」。
- 確定性、無冷啟動、無時區問題、無新持久化、無隱私面；實作量比 A 小一個量級，卻拿到
  絕大部分降幅。

## 評估結論（為何暫緩）

- **可行性：高。** 單一 chokepoint `beatInterval()`；`hb` 自報已能承載任意節奏；即時喚醒現成。
- **邊際效益：增量而非關鍵。** 最大一塊餅 ADR-0109 已吃掉。再降載對**手機電池**是明確
  好處、對**公共中繼額度**是加分；但目前無數據指出某特定時段的負載值得專門處理。
- **決策：暫緩。** 待有量測（電池/中繼請求分佈）支撐再做，且**先做方案 B**（簡單穩健），
  方案 A 的學習模型僅在數據顯示特定窗口值得時才上。

## 若日後實作，必守的紅線

- IDLE→ACTIVE **即時喚醒**不得砍除。
- `hb` 自報須跟著拉長的節奏誠實回報（否則對方誤判離線）。
- 「深度降載」＝該時段**顯示離線/反應慢**——可接受：訊息仍經離線信箱（Gift Wrap）送達，
  功能不壞，只是綠點即時性讓位給省電/省額度；隱身（ADR-0088）與離線語意不受影響。
- 任何學習資料**純本機**、依身分命名空間、不外送、不上中繼。
