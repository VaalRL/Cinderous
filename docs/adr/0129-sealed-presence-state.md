# 0129. 在線狀態內容改走「封裝＋只在改變時」——心跳降為無內容存活信標

- 狀態：已接受（已實作）
- 日期：2026-07-15
- 相關文件：**ADR-0088（presence 元資料最小化：jitter/隱身/P2P 卸載）**、0109（自適應心跳）、
  0120（typing/nudge 封裝——本 ADR 的樣板）、0061（顯示名稱加密同步）、0123（中繼具名訂閱）

## 背景與問題

在線心跳（kind 20000）一則事件同時做兩件事，而且是**明文**：

```ts
createHeartbeat(sk, { status: encodePresence({ s, m, np }) })  // 真名簽章、明文 JSON
```

1. **存活信標**：收端靠到達時間判定上線/離線。
2. **狀態內容**：`s`（上線/離開/忙碌）、`m`（自訂狀態文字，如「我在發呆」）、`np`（正在聽的音樂）。

於是 home relay **每 60 秒**就明文看到一次「這個 pubkey 在聽某首歌、狀態寫著某句話」。顯示名稱
早就加密了（ADR-0061），唯獨這三項狀態內容一直在明文廣播。

研究（前次對話）比較過幾條路：天真封裝（扇給全部聯絡人）會把發佈數乘以聯絡人數、炸穿免費層；
「改變時臨時建 P2P 傳」則因為建連線要 offer＋answer＋ICE 約 4 則信令，**比直接送還貴**，還會把
真實 IP 暴露給每位聯絡人。本 ADR 採**選項 3**：拆開兩個用途，內容改走封裝、且只在改變時送。

## 決策

### 1. 心跳降為**無內容存活信標**

`beat()` 發到 relay 的心跳**內容清空**（只留 `hb` 節奏 tag，供 2.5× 容忍窗）。relay 從此只看得到
「pubkey X 在 T 時刻在線」（時序，本來就藏不住），**再也讀不到 `m`／`np`**。

### 2. 狀態內容（`{s,m,np}`）改走兩條私密通道

- **有 P2P 通道的聯絡人**：走資料通道直送（ADR-0088，本來就是）。
- **在線但沒 P2P 的聯絡人**：送一則 **NIP-59 封裝**的 presence 事件（新 kind 21004，
  外層一次性臨時金鑰、`#p` 定址、ephemeral）。relay 看到的和一則 Gift Wrap 一樣——臨時作者、
  指名收件人、**內容全密文**。樣板同 ADR-0120 的 typing/nudge。

### 3. 🔴 只在**改變時**與**對方剛上線時**送——不是每則心跳

這是讓封裝可負擔的關鍵。天真封裝的成本是「每 60 秒 × 聯絡人數」；本 ADR 是：

- **狀態改變時**（`setStatus`／`setNowPlaying`）：送給每位**在線且無 P2P** 的聯絡人。
- **某位聯絡人剛上線時**（收到他的存活信標、且他先前不在線）：把我當下的狀態送給他一份（**補送**）。

因為改變稀疏、且只扇給「在線✕無P2P」這個小集合（離線的收不到 ephemeral、有 P2P 的走 P2P），
封裝的實際發佈量遠小於「N×每 60 秒」。穩態下（沒改變、沒人上下線）**封裝通道完全靜默**，
relay 只剩無內容信標。

### 4. 訂閱以 `#p` 收（不洩漏聯絡人）

新 kind 的訂閱是 `{ kinds:[21004], "#p":[我] }`——與 Gift Wrap 收件箱同形，不列聯絡人
（與 ADR-0120/0123 一致）。

### 5. 只接受**聯絡人**的封裝狀態

解開後檢查寄件人是聯絡人才採用——陌生人不得往你的介面注入在線狀態（比照 ADR-0121 的把關）。

## 理由

- 這是把 ADR-0120（typing/nudge 封裝）的同一招套到最後一個明文廣播的內容上。顯示名稱（0061）、
  typing/nudge（0120）都已封裝，presence 內容是遺漏的一塊。
- 「只在改變時 ＋ 只給在線✕無P2P」把封裝從「不可負擔」變成「幾乎免費」：它不是每則心跳的固定
  成本，而是與**狀態改變頻率 × 沒在跟你即時互動的在線好友數**成正比——兩者都小。
- 存活信標與狀態內容**本來就是兩件事**，硬綁在一則明文事件裡才造成洩漏。拆開後各自用最合適的
  通道：信標要頻繁、廉價、可容忍明文（只有時序）；內容要私密、稀疏。

## 後果

- 正面：
  - **relay 再也讀不到你的狀態文字與正在聽的音樂**（只剩「何時在線」的時序）。
  - 保留全部功能——沒有 P2P 的聯絡人仍看得到你的狀態文字/音樂（透過稀疏的封裝訊息），
    不像「純 P2P」方案會讓 NAT 打不通的對永久看不到。
  - 不暴露 IP（封裝走 relay，不建 P2P）；不增加中繼成本（穩態靜默，改變時才 1 則）。
  - 中繼成本：存活信標數量不變（ADR-0109 的節省全保留）；封裝只在改變/上線時發，量小。
  - 測試 1029 → **1037**（core +4：封裝往返、外層不含寄件人/內容、他人解不開、缺欄位安全預設；
    engine +4：**心跳無內容且明文找不到狀態/音樂**、狀態仍送達聯絡人、剛上線者被補送〔catch-up〕、
    陌生人不能注入）。

- 已知限制：
  - **時序與社交圖譜仍洩漏**：relay 仍看得到「你何時在線」，也仍能從心跳訂閱的 `authors:[聯絡人]`
    推出聯絡人清單。本 ADR 只處理**內容**（`m`/`np`/`s`）；時序/圖譜是廣播式 presence 的固有代價，
    留待威脅模型記錄（見前次研究的選項 F）。
  - **封裝狀態是 best-effort ephemeral**：改變時那一則若丟包，對方會停在舊狀態，直到你下次改變
    或他重新上線補送。now-playing 每首歌自癒；狀態文字可能短暫過時。這是拿「不重播明文」換來的
    取捨（現況是每 60 秒重播完整狀態，靠冪等自癒）。
  - **`s`（上線/離開/忙碌）也移進封裝**：收到只有信標、還沒收到封裝狀態的聯絡人，暫時顯示為
    「上線」（預設），待封裝狀態到達才細分離開/忙碌。
