# 0121. 訊息請求：陌生人不能直接進你的聯絡人清單

- 狀態：已接受（已實作）
- 日期：2026-07-14
- 相關文件：**ADR-0120（typing/nudge 封裝——本問題在其測試中被揭露）**、0061（顯示名稱個人檔）、
  0058（送達/已讀回條）、0108（未讀水位）、0035/0066（relay hint 自動學習）

## 背景與問題

使用者問了一個看似普通的問題：

> 如果對方把你加為好友，在你確認前他可以傳訊息給你嗎？

答案比問題更極端：**根本沒有「確認」這回事。**

實測（陌生人從未被加過，直接對著 pubkey 發訊息）：

```
我收到的訊息 : [ '哈囉，我是陌生人' ]
我的聯絡人   : [ 'npub1y892lze…' ]     ← 他自動出現在我的聯絡人清單裡
落地的訊息   : [ '哈囉，我是陌生人' ]   ← 已經寫進我的儲存
```

- 「好友請求／待確認／接受」這個概念在整個 codebase 裡**不存在**（grep 全空）。
- 加好友是**純本地、單方面**的動作：`addContact()` 只寫自己的儲存，然後把暱稱送過去（ADR-0061）。
- DM 訂閱是 `{ kinds: [1059], "#p": 我 }`——**沒有 `authors:` 過濾**。
- `receiveDm()` 唯一的把關是 `isBlocked(sender)`。不是聯絡人？照收。
- 而且 `ensureContact(sender)` 讓他**自動成為聯絡人** → 跳通知、能 nudge（震動）、看得到你的上線狀態。

### 這也讓 ADR-0120 昨天才加的把關形同虛設

ADR-0120 擋的是「只收聯絡人的 typing/nudge」。但**只要先傳一則訊息就自動變成聯絡人**——
騷擾者送一則訊息，然後就能無限震動你的手機。我在 ADR-0120 的已知限制裡記下了這件事，
說它是「產品層級的決策」。現在使用者裁示了。

## 決策

### 1. 陌生人的訊息進「訊息請求」，**不是**聯絡人清單

`receiveDm()` 的 `ensureContact(sender)` 改為 `ensureKnown(sender)`：

```ts
if (this.isBlocked(pubkey) || this.isContact(pubkey)) return;
this.storage.addRequest({ pubkey, name: shortNpub(npubEncode(pubkey)) });
```

儲存層新增 `requests: StoredContact[]`，**完全比照既有的 `blocked`**（同樣的 API 形狀、
同樣進 `StorageSnapshot`、同樣落 localStorage / Tauri 的 meta 部位）——不發明新機制。

### 2. 在你接受之前，請求者**碰不到你的任何注意力管道**

這是本 ADR 的核心。「不是聯絡人」必須真的有意義：

| 管道 | 請求者 | 為什麼 |
|---|---|---|
| 桌面／行動端**通知** | ❌ | 讓陌生人推播到你的鎖定畫面，那就是騷擾 |
| **未讀徽章** | ❌ | 徽章和通知是同一條注意力管道 |
| **自動開窗**（桌面） | ❌ | 等於讓陌生人強制彈出視窗 |
| **nudge**（震動） | ❌ | ADR-0120 的把關現在真的有效了 |
| **typing** 指示 | ❌ | 同上 |
| 看到你的**上線狀態** | ❌ | 不訂閱他的心跳，也不對他廣播 |
| 收到你的**個人檔** | ❌ | 見決策 3 |
| 收到**已讀回條** | ❌ | 見決策 3 |
| 訊息本身送達你 | ✅ | Nostr 上**擋不掉**——見「已知限制」 |

### 3. 🔴 **不給垃圾訊息發送者任何回饋**

兩處要特別擋，因為它們都是「系統自動做的事」，很容易漏掉：

**（a）不回送個人檔。** 原本 `receiveDm` 收到對方個人檔時會自動回送自己的：

```ts
if (!this.profileSentTo.has(sender)) this.sendProfileTo(sender);   // 舊
if (this.isContact(sender) && !this.profileSentTo.has(sender)) …   // 新
```

對還沒接受的陌生人回送，等於向他確認「**這把金鑰是活的、有人在線上**」——那正是
垃圾訊息發送者最想要的東西。

**（b）不送已讀回條。** 使用者點開請求「看一眼」時，`markRead()` 會發出已讀回條：

```ts
if (this.requests.some((r) => r.pubkey === convo)) return;   // ADR-0121
```

**看，但不要回話。**

### 4. 接受＝變成聯絡人；**主動回覆也算接受**

```ts
sendMessage(to, …) {
  if (this.requests.some((r) => r.pubkey === to)) this.acceptRequest(to);
  …
}
```

不接受就能回訊息會很怪：對方在你的清單裡永遠是「請求」，你卻在跟他聊天。
**你主動聯絡的人就是聯絡人。**

「刪除」＝清掉請求與他的訊息，但**不封鎖**（他還能再傳）。要永久擋掉請按「封鎖」。

### 5. 請求期間仍要學會的兩件事

雖然他不是聯絡人，但這兩樣**錯過就沒有第二次**（它們只在已解密的 rumor 裡）：

- **顯示名稱**（ADR-0061）：否則請求區只看得到 `npub1abc…`，使用者無從判斷要不要接受。
- **relay hint**（ADR-0035/0066）：否則你一按接受，回信只能走 home relay，而他可能在別座。

所以 `updateContactName` / `updateContactRelay` / `learnRelayHint` 都要**同時涵蓋請求區**，
且 `acceptRequest()` 要把學到的東西一起帶進聯絡人。

### 6.「對方把你加為好友」**本身就是一則請求**

`addContact()` 會送出一份個人檔（ADR-0061）。那份個人檔抵達你這裡時，就會建立一則請求
——帶著他自選的暱稱。**那就是好友請求，不需要新的事件類型。**

## 理由

- 這不是「加一個功能」，是**把一個一直存在的洞補起來**。專案從一開始就是簡訊模型
  （你的 npub 是公開的收件地址），但**沒有人決定過**要這樣——它只是 `ensureContact()`
  的一個副作用。
- 而「不是聯絡人」如果不真的切斷所有注意力管道，就只是**換個地方顯示同一批人**。
  所以本 ADR 的重點不在那個清單，而在上面那張表：**通知、徽章、開窗、震動、上線狀態、
  個人檔、已讀回條——一個都不給。**
- 這個模型是 Signal／Instagram 的做法：保留「陌生人能找到你」（那是去中心化的前提），
  但拿掉騷擾管道。

## 後果

- 正面：
  - 陌生人不再自動成為聯絡人，也碰不到任何注意力管道。
  - **ADR-0120 的把關現在真的有效**（過去傳一則訊息就能繞過）。
  - 測試 961 → **981**（engine +12：請求不進聯絡人、不能 nudge/typing、不送個人檔、
    不送已讀回條、不點亮徽章、接受/刪除/封鎖、主動回覆＝接受、重載後仍在、hint 照學；
    desktop UI +4、mobile UI +4）。
  - **8 個既有測試因此壞掉**——它們的名字直接寫著「收件端**自動加入**寄件人」。
    那正是行為改動為真的證據；已逐一改寫為新語意（而非放寬斷言）。

- 已知限制：
  - **訊息仍會抵達中繼並存在那裡（7 天 TTL）**。Nostr 的中繼會收下任何指名你的 kind 1059，
    客戶端無法阻止。本 ADR 是**客戶端的顯示層防禦**——真正的「不可被聯繫」在 Nostr 上做不到
    （除非改成付費中繼／relay ACL 白名單，那是另一個 ADR 的題目）。
  - **請求區本身可以被灌爆**（大量陌生 pubkey 各傳一則）。目前沒有數量上限、沒有速率限制、
    沒有「全部刪除」。**這是下一步該做的。**
  - 群組邀請不走請求流程：把你拉進群組的人若不是你的聯絡人，群組仍會直接出現。
  - 沒有「只接受聯絡人訊息」（白名單）的全域開關——留給使用者選項的 ADR。
