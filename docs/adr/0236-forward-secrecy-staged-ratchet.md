# 0236. 前向保密：分階段棘輪路徑

- 狀態：已接受（Stage 1 為可執行工作項；Stage 2/3 另立後續 ADR）
- 日期：2026-07-23
- 相關文件：`PRD.md §4／§7`、`ARCHITECTURE.md §1／§4／§5／§9`、ADR-0002（隱私元資料與協定基線）、
  ADR-0027（群組成對扇出、MLS 延後）、ADR-0107（多設備同步）、ADR-0071（加密雲端快照）、
  ADR-0070／0072（換機搬家）、ADR-0013（限時訊息）、ADR-0234（無痕收回）

## 背景與問題

Cinderous 主打「隱私優先」，但目前的密碼學模型**沒有前向保密（Forward Secrecy, FS）**，也沒有
後妥協安全（Post-Compromise Security, PCS）。這是本產品定位下最大的剩餘密碼學缺口，§9 已將它
列為開放問題；本 ADR 收斂方向並定出可執行的第一步。

### 現況精確描述

- **靜態金鑰。** 身分是一組固定的 secp256k1 金鑰對（`packages/core/src/keys.ts`）。NIP-44 加密的
  對話金鑰 `nip44.getConversationKey(sk, pk)` 是 `ECDH(靜態, 靜態)`——**對同一對收發者，這輩子都是
  同一把**（`packages/core/src/nip44.ts:16/26`）。
- **Gift Wrap 的臨時金鑰不是前向保密。** `sealAndWrap`（`nip59.ts`）外層 wrap（kind 1059）用一把
  一次性臨時金鑰對收件人**靜態**公鑰加密，內層 seal（kind 13）則是寄件人**靜態** → 收件人**靜態**。
  兩層的 ECDH 對手方都是**收件人的靜態金鑰**。臨時 wrap 金鑰的作用是**隱藏寄件人身分（元資料）**，
  ——它**完全不提供內容前向保密**：只要拿到收件人的 nsec，wrap 與 seal 都能解。
  > 這一點必須寫清楚，因為「我們已經用了臨時金鑰」是一個常見的錯覺。我們用的臨時金鑰保護的是
  > 「誰寄的」，不是「金鑰失竊後歷史密文是否還安全」。

### 威脅模型（FS 要防的是誰）

1. **被動記錄流量的網路對手**：長期側錄 kind 1059 密文。
2. **惡意／被入侵的中繼**：在 7 天 TTL 到期前記下 log。
3. **未來的 nsec 竊取**：裝置被扣押、惡意軟體、備份外流。

在目前模型下，上述任一方**日後取得任一端的 nsec，即可解開所有曾被側錄的歷史密文**。7 天 TTL 只
限制「中繼上的存量」，擋不住已經被複製走的密文。FS 的目標是：**金鑰一旦輪替並刪除，先前的密文
就再也解不開**——即使日後 nsec 失竊。

### 為什麼難（本專案特有的硬約束）

任何 FS 方案都得同時滿足這些，否則會撞壞既有設計：

- **非同步 / store-and-forward。** 收件人可能離線數天，一次拉回大量訊息。棘輪必須容忍亂序與
  「一步棘輪下有很多則訊息」，需要「跳過訊息金鑰」窗。
- **多設備共用同一把 nsec。** 身分金鑰**就是** nsec，所有裝置共用它（ADR-0107）。Signal 靠 Sesame
  給每台裝置獨立身分與 per-device session；Cinderous 沒有這層。若兩台裝置在同一條發送鏈上並行送訊，
  會**重用 counter → 金鑰重用**（對串流加密是災難）。這是 FS 在本專案最尖銳的技術點。
- **群組無共用金鑰、且刻意免 rekey（ADR-0027）。** 群組是成對 Gift Wrap 扇出，移除成員「扇出略過、
  免 rekey」。替群組加 FS 會把 MLS 專門要解的 rekey 問題重新引回來。
- **與標準 Nostr relay 互通（ADR-0034）。** 傳輸層必須維持相容（照樣是 kind 1059 事件）；內容變成
  棘輪狀態沒關係——別的 Nostr 客戶端本來就讀不了 Gift Wrap 內容，不算退化。
- **「換機還原完整歷史」與 FS 的張力。** FS 要求刪掉舊金鑰，但 ADR-0071/0072 讓使用者在新機還原
  全部歷史。關鍵釐清（見下）：**耐久歷史來自本機加密明文封存與加密雲端快照，不是靠重新解密中繼
  密文**，所以 FS 不會破壞歷史還原——兩者是不同的資料路徑。

## 考量的選項

- **選項 0：維持現狀（靜態金鑰），只把限制文件化。** 零成本，但缺口仍在，且與「隱私優先」定位
  長期矛盾。
- **選項 1：對話層對稱 KDF 棘輪（雜湊棘輪，無 DH）。** 由現有 NIP-44 靜態 ECDH 導出初始 root，
  之後以 KDF 鏈前進：每則訊息一把 message key、用完即刪；`CK_{i+1}=KDF(CK_i)`。
  - **給 FS，不給 PCS**（沒有新 DH 熵注入；當前鏈金鑰一旦洩漏，之後可推導）。
  - 非同步友善（收端可前進 N 步 + 保留跳過金鑰窗）。與 store-and-forward 相容度最高。
  - 成本中等：每對話狀態、刪除紀律、多設備需同步棘輪位置或分裝置子鏈。
- **選項 2：X3DH 式一次性 prekey 做 session 建立（＋選項 1 的對稱棘輪）。** 發佈簽章的一次性
  prekey（可尋址 Nostr 事件），首則訊息消耗一把 → 初始 root 由「臨時 × 一次性 prekey」導出，
  **強於靜態 × 靜態**，並為 PCS 鋪路。成本：prekey 發佈／補充基礎建設、耗盡時退回簽章 prekey（較弱）。
- **選項 3：完整 Double Ratchet（Signal）。** FS ＋ PCS。但 DH 棘輪要求有序、有 session；跨「多設備
  共用 nsec × 非同步」是出了名地難（需 Sesame 級的 per-device session 與扇出）。與 local-first／
  多設備模型正面衝突，工程量巨大。
- **選項 4：MLS（群組尺度）。** FS ＋ PCS ＋ 高效群組 rekey，是群組的正解，但需要有序 epoch 的
  Delivery Service。ADR-0027 已決定延後。

## 決策

採**分階段路徑**，本 ADR 定案「方向與 Stage 1」，Stage 2/3 為後續 ADR：

- **Stage 1（近期，本 ADR 的可執行第一步）：1:1 對話的對稱 KDF 棘輪（選項 1）。**
  - root 由現有 NIP-44 靜態對話金鑰經 HKDF 域分隔導出（`"cinder-ratchet-v1"`）；發送鏈與接收鏈
    依方向標籤分離；每則訊息用一把 message key、用完即刪；訊息在**加密內層 rumor**帶 counter 供
    收端前進／跳過；跳過金鑰保留有界窗（如上限 1000，尾端淘汰）以容忍非同步亂序。
  - **多設備**：以**分裝置子鏈**解 counter 碰撞——每台裝置由 root＋裝置識別導出自己的發送子鏈，
    收端各自追蹤「每（寄件裝置）一條鏈」，杜絕並行送訊的金鑰重用。棘輪位置隨 ADR-0107 自封同步
    事件、ADR-0071 快照、ADR-0072 配對克隆一併攜帶。
  - **明確界線**：Stage 1 給 **FS、不給 PCS**；只做 **1:1、不做群組**（群組維持 ADR-0027 免 rekey
    扇出，FS 待 MLS）。
- **Stage 2（中期）：X3DH 式一次性 prekey**強化 session 建立，為 PCS 鋪路（另立 ADR）。
- **Stage 3（長期）：評估 1:1 Double Ratchet（取得 PCS）與群組 MLS**，以「多設備 session 管理是否
  值得該複雜度」為決策閘（另立 ADR）。

**本階段明確不做**：群組 FS、PCS、身分金鑰輪替（nsec 仍是永久主金鑰，見「理由」）。

## 理由

- **為何分階段、且對稱棘輪優先。** 對稱 KDF 棘輪是唯一能在**不引入 prekey 基礎建設、不破壞多設備、
  不動群組語意**的前提下拿到前向保密的方案。它把「隱私優先」最欠缺的那塊（歷史密文在金鑰失竊後
  仍安全）補上，成本可控，且為 Stage 2/3 留好接口。
- **為何現在不上完整 Double Ratchet。** DH 棘輪在「多設備共用 nsec」下等於要維護 per-(寄件裝置,收件
  裝置) 的 N×M session，並需要 Sesame 級的裝置管理——與 local-first／非同步模型衝突。先用對稱棘輪
  拿 FS，把 PCS 留給有 prekey（Stage 2）之後再評估，是風險最低的排序。
- **為何群組等 MLS。** 替 ADR-0027 的免 rekey 成對扇出硬加 FS，會重新引入成員變動時的 rekey 難題，
  那正是 MLS 存在的理由。在 1:1 拿到 FS 的效益 ≫ 在群組勉強拼一個半成品。
- **為何 nsec 仍不輪替（與 PRD §4 一致）。** 棘輪金鑰是**session 金鑰**，與**身分金鑰**是兩回事。
  身分／備份／換機模型（ADR-0070/0071/0072）完全不受影響：nsec 仍是主金鑰、仍用於還原身分；
  棘輪只在 session 層輪替與刪除。
- **FS 不破壞歷史還原（重要釐清）。** 換機還原歷史走的是「本機加密明文封存 ＋ 加密雲端快照」
  （ADR-0071），**從來不是靠重新解密中繼上的密文**。所以「刪掉舊棘輪金鑰」不影響使用者在新機看到
  舊訊息——耐久歷史與棘輪金鑰是兩條資料路徑。棘輪**狀態**（非金鑰本身，是位置）需納入同步/快照/
  克隆，讓新裝置能接續解**新**訊息。
- **行銷誠信。** 在 Stage 1 上線前，官網與 FAQ **不得宣稱前向保密**。（ADR-0235 的 FAQ 文案已刻意
  只寫「端對端加密」而非「牢不可破」，與此一致。）

## 後果

- 正面：
  - 1:1 訊息取得前向保密：金鑰輪替並刪除後，先前側錄的密文在 nsec 日後失竊時**仍然解不開**。
  - 與既有 Gift Wrap／限時訊息（ADR-0013）／無痕收回（ADR-0234）互補：FS 保護「線上（wire）」，
    限時/收回保護「端點」，at-rest 加密（ADR-0054/0110）保護「落地」。
  - 為 Stage 2/3（PCS、群組 MLS）留好演進路徑，不需回頭重寫。
- 負面 / 已知殘餘風險：
  - **不給 PCS**：持續握有存取權的攻擊者仍能繼續解讀（待 Stage 3）。
  - **本機明文封存是另一條暴露面**，FS 管不到——由 at-rest 加密與 OS 金鑰庫負責；重視實務隱私者
    應併用限時訊息/封存修剪。
  - **群組無 FS**（待 MLS）。
  - **多設備並行送訊的正確性**是 Stage 1 的主要工程風險（分裝置子鏈解 counter 碰撞，但同步收斂、
    新裝置無棘輪狀態時的重建、跳過金鑰窗上限，都需審慎設計與測試）。
  - 每對話新增棘輪狀態的儲存與同步成本（位置＋有界跳過金鑰窗）。
- 後續行動 / 待辦：
  1. **接受本 ADR** → 開 Stage 1：`packages/core/src/ratchet.ts`（SSOT，純函式、可測），供
     `giftwrap`/`nip59`/`engine` 消費；TDD 測向量涵蓋「刪除後無法解先前」「跳過金鑰窗」
     「多裝置子鏈收斂、無 counter 碰撞」。
  2. 棘輪狀態接入 ADR-0107 同步、ADR-0071 快照、ADR-0072 配對克隆。
  3. 更新 `ARCHITECTURE.md §5` 事件契約：標明 1:1 訊息內層帶棘輪 counter。
  4. Stage 2（prekey）、Stage 3（Double Ratchet／MLS）各立後續 ADR。
  5. 產品文案守則：FS 未上線前不得宣稱前向保密。
