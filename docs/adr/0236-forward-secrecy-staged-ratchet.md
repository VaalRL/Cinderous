## 背景與問題

Cinderous 主打「隱私優先」，但目前的密碼學模型**沒有前向保密（Forward Secrecy, FS）**，也沒有
後妥協安全（Post-Compromise Security, PCS）。這是本產品定位下最大的剩餘密碼學缺口，§9 已將它
列為開放問題；本 ADR 收斂方向並定出可執行的第一步。

### 現況精確描述

- **靜態金鑰。** 身分是一組固定的 secp256k1 金鑰對（`packages/core/src/keys.ts`）。NIP-44 加密的
  對話金鑰 `nip44.getConversationKey(sk, pk)` 是 `ECDH(靜態, 靜態)`——**對同一對收發者，這輩子都是
  同一把**（`packages/core/src/nip44.ts:16/26`）。
- **Gift Wrap 的臨時金鑰不是前向保密。** `sealAndWrap`（`nip59.ts`）外層 wrap（kind 1059）用一把
  一次性臨時金鑰對收件人**靜態**公鑰加密，內層 seal（kind 13）則是寄件人**靜態** → 收件人**靜態**。
  兩層的 ECDH 對手方都是**收件人的靜態金鑰**。臨時 wrap 金鑰的作用是**隱藏寄件人身分（元資料）**，
  ——它**完全不提供內容前向保密**：只要拿到收件人的 nsec，wrap 與 seal 都能解。
  > 這一點必須寫清楚，因為「我們已經用了臨時金鑰」是一個常見的錯覺。我們用的臨時金鑰保護的是
  > 「誰寄的」，不是「金鑰失竊後歷史密文是否還安全」。

### 威脅模型（FS 要防的是誰）

1. **被動記錄流量的網路對手**：長期側錄 kind 1059 密文。
2. **惡意／被入侵的中繼**：在 7 天 TTL 到期前記下 log。
3. **未來的 nsec 竊取**：裝置被扣押、惡意軟體、備份外流。

在目前模型下，上述任一方**日後取得任一端的 nsec，即可解開所有曾被側錄的歷史密文**。7 天 TTL 只
限制「中繼上的存量」，擋不住已經被複製走的密文。FS 的目標是：**金鑰一旦輪替並刪除，先前的密文
就再也解不開**——即使日後 nsec 失竊。

### 為什麼難（本專案特有的硬約束）

任何 FS 方案都得同時滿足這些，否則會撞壞既有設計：

- **非同步 / store-and-forward。** 收件人可能離線數天，一次拉回大量訊息。棘輪必須容忍亂序與
  「一步棘輪下有很多則訊息」，需要「跳過訊息金鑰」窗。
- **多設備共用同一把 nsec。** 身分金鑰**就是** nsec，所有裝置共用它（ADR-0107）。Signal 靠 Sesame
  給每台裝置獨立身分與 per-device session；Cinderous 沒有這層。若兩台裝置在同一條發送鏈上並行送訊，
  會**重用 counter → 金鑰重用**（對串流加密是災難）。這是 FS 在本專案最尖銳的技術點。
- **群組無共用金鑰、且刻意免 rekey（ADR-0027）。** 群組是成對 Gift Wrap 扇出，移除成員「扇出略過、
  免 rekey」。替群組加 FS 會把 MLS 專門要解的 rekey 問題重新引回來。
- **與標準 Nostr relay 互通（ADR-0034）。** 傳輸層必須維持相容（照樣是 kind 1059 事件）；內容變成
  棘輪狀態沒關係——別的 Nostr 客戶端本來就讀不了 Gift Wrap 內容，不算退化。
- **「換機還原完整歷史」與 FS 的張力。** FS 要求刪掉舊金鑰，但 ADR-0071/0072 讓使用者在新機還原
  全部歷史。關鍵釐清（見下）：**耐久歷史來自本機加密明文封存與加密雲端快照，不是靠重新解密中繼
  密文**，所以 FS 不會破壞歷史還原——兩者是不同的資料路徑。

## 考量的選項

- **選項 0：維持現狀（靜態金鑰），只把限制文件化。** 零成本，但缺口仍在，且與「隱私優先」定位
  長期矛盾。

- **選項 1：對話層對稱 KDF 棘輪（雜湊棘輪，無 DH）。** 由 NIP-44 靜態 ECDH 導出 root，之後
  `CK_{i+1}=KDF(CK_i)`，每則訊息一把 message key、用完即刪。
  - ⚠️ **修正（2026-07-23）：單獨無效。** root 若由**靜態** ECDH deterministic 導出，日後取得 nsec
    者可 `nsec → K → CK₀ → 整條鏈`，把刪除過的舊金鑰全部重算——**對抗 nsec 失竊的 FS 根本沒達成**
    （刪除只擋得住「拿到當前鏈金鑰、但拿不到 nsec 且沒側錄密文」這種極窄情況）。純對稱棘輪不是
    能兌現主威脅的獨立方案。

- **選項 2：「定期更換金鑰」——關鍵在**換什麼**。**
  - ❌ **換「靜態 ECDH 導出的對話金鑰」**（每 epoch 一把 `CK_e=KDF(ECDH,e)`）：和選項 1 同一陷阱
    ——`CK_e` 從靜態秘密 deterministic 導出，nsec 一失即重算任一 epoch。**無 FS。**
  - ✅ **換「一組會被刪除的加密子鑰」**：見選項 3。
  - **通則：輪替金鑰只有在「被輪替的私鑰含新鮮熵、且事後真的刪掉、且無法從保留的長期秘密導出」
    時才有 FS。** 從 nsec 算得回來的東西，刪了也沒用。

- **選項 3（新增，推薦近期路徑）：輪替簽章加密子鑰——粗粒度 FS。**
  - 身分金鑰 IK（npub）**不動**（維持全網定址與簽章）；另生一把**加密子鑰** `EK_e`，每 epoch
    （如每週）一把，`pub(EK_e)` 由 IK 簽章後發佈為**可取代** Nostr 事件（收端信任這是你當前子鑰）。
  - 寄件人以**一次性臨時金鑰**（Gift Wrap 本就有）對收件人**當前子鑰** `pub(EK_recipient_e)` 加密
    ——只需把現有 seal/wrap 的 ECDH 對手方從「收件人靜態金鑰」換成「收件人當前子鑰」。
  - 收件人在 epoch 結束＋grace（對齊 7 天離線暫存）後，**在所有裝置與快照刪除 `priv(EK_e)`**。
    事後 nsec 失竊者拿不到已刪的 `EK_e` 私鑰、IK 也導不出它（隨機、只被 IK **簽章連結**而非導出）
    → **舊密文再也解不開**。
  - **給 FS（粒度＝epoch＋grace ≈ 1～2 週）、不給 PCS**。無 prekey 基礎建設、改動局部。
  - **關鍵優勢：避開棘輪的多設備地雷。** 沒有 per-message 序列鏈 → 沒有 counter 碰撞、沒有跳過
    金鑰窗 DoS；多設備只需「共享每 epoch 一把金鑰」（粗、低頻；落後裝置在 grace 內仍可解、送訊用
    一次性臨時金鑰不需 epoch 金鑰＝優雅降級）。

- **選項 4：X3DH 一次性 prekey ＋對稱棘輪。** 選項 3 是「中期 signed prekey」的粗粒度版；選項 4
  再加**一次性** prekey → 更細的 FS、並為 PCS 鋪路。成本：prekey 發佈／補充、耗盡退回 signed prekey。

- **選項 5：完整 Double Ratchet（Signal）。** FS ＋ PCS。但 DH 棘輪要求有序、有 session；跨「多設備
  共用 nsec × 非同步」極難（需 Sesame 級 per-device session 與扇出）。與 local-first／多設備模型
  正面衝突，工程量巨大。

- **選項 6：MLS（群組尺度）。** FS ＋ PCS ＋ 高效群組 rekey，是群組正解，但需有序 epoch 的
  Delivery Service（Nostr 天生沒有）。ADR-0027 已決定延後。

## 決策（2026-07-23 修訂）

**撤回初版的「Stage 1＝純對稱棘輪」（對主威脅無效）。改為兩條路徑：近期評估／原型「輪替加密
子鑰」（選項 3，粗粒度 FS）；完整 PCS 與群組 FS 走 MLS、延後、且不自捲。**

- **近期 FS 路徑＝選項 3（輪替簽章加密子鑰）。** 它以可接受成本兌現「對抗 nsec 失竊」的粗粒度 FS，
  且**避開棘輪的多設備殺手級問題**（無 per-message 鏈）。這是本 ADR 現在的推薦方向。**先做設計
  ADR＋原型**（子鑰 PKI／grace 窗與快照老化的刪除紀律／多設備 epoch 同步／retarget gift wrap／
  向後相容退回靜態金鑰／遷移），驗證可用性後再決定是否上線。
- **per-message 棘輪（選項 5）延後。** 其邊際收益（per-message 粒度——而對 nsec 威脅選項 3 已覆蓋）
  不抵它帶來的複雜度與新失敗模式（見〈理由〉風險 A–G）。
- **完整 PCS／群組 FS 若要做，以 MLS（選項 6，現成函式庫）為前提、不自捲棘輪。** 代價是給 Nostr
  補有序 epoch delivery。另立 ADR、排外部審計。
- **任何 FS 上線前的近期止血：** 鼓勵／預設**限時訊息**（ADR-0013）＋ **at-rest 加固**
  （ADR-0054/0110）＋ **誠實文件**——覆蓋大部分實務威脅，且無棘輪失敗模式。

**明確不做（現階段）**：per-message 棘輪、群組 FS、PCS、身分金鑰輪替（IK/npub 仍是永久主金鑰
——選項 3 輪替的是**加密子鑰**，不是身分）。

## 理由

### 純對稱棘輪為何不足（初版錯誤的更正）

任何「從靜態金鑰算得出、或用靜態金鑰加密傳輸」的金鑰材料，都會被「nsec ＋ 側錄密文」重建。
刪除舊金鑰只有在「被刪的東西無法從保留的秘密重算」時才有意義；nsec 是身分、必須保留，靜態
ECDH root 因此永遠可重算。故對抗 nsec 失竊的 FS **必須**含輪替／一次性、且**事後刪除**的私鑰
材料（公鑰可公開、私鑰不可從 IK 導出）。這正是選項 3/4 做的，也是初版把「純對稱棘輪」當獨立
Stage 1 的錯誤所在。**「定期換金鑰」若換的是靜態導出的金鑰，同樣無效——見選項 2。**

### 為何選項 3（輪替子鑰）優於 per-message 棘輪

per-message 棘輪假設「兩端點、有序、連續 session 狀態」，Cinderous 三者皆不成立，衍生風險 A–G。
選項 3 因為**沒有 per-message 序列鏈**，直接消掉其中最嚴重的幾條，同時仍拿到（粗粒度）FS：

- **A. 多設備（棘輪的殺手級）**：棘輪共用 nsec → 同步鏈狀態時並行送訊撞同一 counter＝**金鑰重用**
  （AEAD 災難），或 per-device N×M session＋裝置目錄。**選項 3 消掉它**：無序列鏈，多設備只需共享
  每 epoch 一把金鑰（粗、低頻），送訊用一次性臨時金鑰、不需共享送出狀態。
- **B. 非同步／無序 → 跳過金鑰窗**：棘輪窗小＝晚到訊息永久解不開（撞 7 天暫存）、窗大＝記憶體 DoS
  （棘輪經典 CVE 類）。**選項 3 消掉它**：無 per-message 鏈可跳。
- **C. 狀態脆弱 → 失步**：棘輪 counter 失步要重握手，且與快照時間點衝突。**選項 3 大幅減輕**：
  狀態只是「當前是哪把 epoch 金鑰」，粗且容錯。
- **D. FS 摧毀歷史後備**：**兩者皆有**——FS 刻意讓舊密文解不開，歷史只剩本機封存＋快照；沒開快照者
  「更安全」的代價是「更容易永久失去歷史」。選項 3 的刪除紀律要延伸到快照老化。
- **E. 群組**：per-message 成對棘輪給不了群組 PCS；選項 3 可把「輪替群組金鑰」套到群組拿到粗粒度
  群組 FS（但被踢成員的 PCS 仍需 rekey-on-removal，屬 MLS）。
- **F. 元資料**：兩者都要**發佈**金鑰材料（prekey／輪替子鑰）＝新的（可取代、極小）訊號；與
  ADR-0237 的元資料目標輕微拉扯。
- **G. 自捲協定風險**：per-message 棘輪的協定組裝（握手／session／header 加密／失步復原／重放）是
  bug 溫床、落在最安全關鍵的 `packages/core`，棘輪 bug（金鑰重用／狀態混淆／跳過金鑰 DoS／失步
  注入）可能**比沒棘輪更糟**。**選項 3 的攻擊面小得多**（輪替簽章子鑰＋刪除＋retarget gift wrap），
  自捲風險可控；但仍建議審計。

### meta 取捨

真實 FS 需要「新鮮、可刪、非靜態導出」的私鑰材料——這無法迴避。差別在**粒度與載體**：per-message
棘輪（細粒度、高複雜度、多設備地雷）vs 輪替子鑰（粗粒度、低複雜度、避開地雷）。對 Cinderous 的
規模與威脅（nsec 失竊是罕見事件，不需 per-message 粒度即可防），**輪替子鑰是更划算的兌現方式**。
而 FS 管不到的**本機明文封存**（靠 at-rest 加密）與已有的 7 天 TTL／限時訊息，仍是主要的實務緩解。

（保留：身分金鑰不輪替——選項 3 輪替的是加密子鑰，IK/npub 不動，ADR-0070/0071/0072 不受影響。
FS 不破壞歷史還原——歷史走明文封存/快照，非重解密文。行銷誠信——FS 未上線不得宣稱前向保密，
ADR-0235 FAQ 已守此線。）

## 後果

- 正面：
  - 把「前向保密」從長期被延後的模糊開放問題，收斂成**有明確、且更務實決策**的文件：輪替加密子鑰
    （粗粒度 FS）為近期路徑，per-message 棘輪延後、完整 PCS/群組走 MLS。
  - 更正並存證「純對稱棘輪對 nsec 威脅無效」「定期換靜態導出金鑰同樣無效」與「輪替金鑰只有在私鑰
    含新鮮熵且事後刪除時才有 FS」——擋掉未來重踩。
  - 選項 3 重用既有 Gift Wrap 一次性臨時金鑰模式（僅 retarget 到收件人當前子鑰），改動局部、避開
    棘輪多設備地雷。
- 負面 / 已知殘餘風險：
  - **FS 粒度為 epoch＋grace（≈1～2 週）**、且**無 PCS**——明確接受的取捨。
  - **刪除紀律**要延伸到多設備與快照老化，否則 FS 被「還沒刪的舊金鑰」削弱。
  - **子鑰 PKI**：輪替子鑰的發佈、簽章連結（IK→EK）、對方未發佈子鑰時退回靜態金鑰、遷移，都要設計。
  - **本機明文封存**仍是最大長期暴露面（FS 本就管不到）。
  - MLS 路線若啟動仍需自建有序 epoch delivery（Nostr 天生無序）。
- 後續行動 / 待辦：
  1. **撤回**初版的 `packages/core/src/ratchet.ts`（純對稱棘輪）Stage 1 工作項。
  2. **立「輪替加密子鑰（粗粒度 FS）」設計 ADR＋原型**：子鑰生成/簽章/發佈（可取代事件）、grace 窗
     與快照老化的刪除紀律、多設備 epoch 同步、retarget gift wrap、向後相容退回靜態金鑰、遷移。
     驗證後再定是否上線。
  3. 近期止血：評估「限時訊息預設化」與 at-rest 加固清單；文案守則維持「不得宣稱 FS」。
  4. 完整 PCS/群組若要做：立「MLS over Nostr」ADR（含有序 epoch delivery），排外部審計。
  5. 更新 §9 與 ADR 索引，反映修訂後的方向。
