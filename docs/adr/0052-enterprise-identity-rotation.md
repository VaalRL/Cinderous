# 0052. 企業工作身分輪替/重佈建（否決金鑰托管，無後門）

- 狀態：已接受
- 日期：2026-07-07
- 相關文件：PRD §4（不輪替/不備份/設備死亡）、§7（靜態金鑰取捨）、§13（企業模式與排除項）；docs/adr/0028（維持靜態金鑰、無 FS）、0044（封閉 allowlist）、0045（多身分）、0046（成員判定與邊界）、0047（簽章名冊）、0049（組織群組）、0009（多設備同步）；ROADMAP G4
- 取代：ROADMAP「未決策 ADR：G4 企業金鑰託管」

## 背景與問題

企業有實務硬需求：員工**換機/遺失設備後仍能繼續工作**。ROADMAP G4 原先把此需求描述為「金鑰託管（key escrow）」——把工作身分私鑰加密託管到公司還原金鑰，遺失時還原。

但 Cinder 的加密現實（ADR-0028）是 **NIP-44 靜態 ECDH，無前向保密**：**能還原一把私鑰＝能解密該身分全部歷史與未來訊息**。因此「托管靜態 nsec」不是一般備份，而是**公司持有一個可回溯、永久、全解密的後門**，涵蓋每位員工每則訊息。這等同 PRD §13.1 明確排除的「受監督版」能力（法遵歸檔/eDiscovery/DLP 需伺服器讀明文），與 E2E、零知識鐵則根本衝突。

需在「員工能繼續工作」與「公司不得成為解密後門」之間定案。經與使用者確認：**企業的真實需求是「換機續用」，而非「還原同一身分＋歷史」。**

## 考量的選項

- **選項 A（採用）：工作身分輪替/重佈建。** 遺失時作廢舊 npub、員工自產新 npub、經簽章名冊（ADR-0047）重新佈建；不托管任何金鑰。
- **選項 B（否決）：靜態金鑰托管。** 托管工作身分 nsec 至公司還原金鑰，遺失時還原同一身分。即使加 Shamir 門檻/HSM，公司仍具全解密後門，產品在企業模式下被迫定性為「受監督版」。
- **選項 C（否決）：只托管本機資料庫加密金鑰、不托管 nsec。** 技術上無效：DB 金鑰只保護靜態落地；要繼續收發同一 npub 的 Gift Wrap 仍需 nsec。無法達成「換機續用」，僅是半套。

## 決策

**採選項 A：以「企業工作身分輪替」滿足換機/遺失需求，明確否決金鑰托管。**

1. **員工自產、公司不碰私鑰：** 新設備上由**員工自行產生**新工作身分金鑰對（nsec 永不離開員工端，維持零知識）。管理者只更新名冊，**全程不接觸 nsec**。
2. **輪替＝名冊操作（複用 ADR-0047）：** 管理者在簽章名冊（kind 10038）中把舊 npub 標記 `supersededBy → 新 npub`、加入新 npub；relay `allowedAuthors` 以同一份名冊 diff 移除舊、加入新。輪替不新增協定機制。
3. **客戶端自動對帳＋remap：** 工作身分收到較新名冊 → 驗簽 → 依 `diffRoster` 更新通訊錄；遇 `supersededBy` 條目，把本機聯絡人/組織群成員從舊 npub 接到新 npub，UI 顯示「◯◯ 已更新金鑰」。組織群（ADR-0049）本就名冊對帳，自動換人。
4. **遺失防護＝多設備冗餘（複用 ADR-0009/M4），非托管：** 建議員工登記 ≥2 台設備（如桌機＋手機）。掉**單台**→ 另一台仍有金鑰與完整 DB，身分與歷史皆不丟、免輪替。僅**全部設備皆毀**才走輪替（此時歷史依「設備死亡」承諾消失）。部署文件據此建議雙設備登記。
5. **範圍：** 僅適用 enterprise 工作身分（名冊權威所轄）。**個人身分完全不受影響**（仍設備死亡、無輪替權威），呼應 ADR-0045/0046。

## 理由

- **消除後門、保住定性：** 公司永遠沒有能解員工訊息的金鑰；被駭/被傳喚也拿不出明文（不存在）。產品不會滑入「可監控員工」的受監督版。
- **滿足真實需求：** 「換機續用」是企業真正要的；輪替五分鐘回到線上，同事端自動換人。
- **最大複用、此環境可做：** 輪替 ≈ 名冊「撤銷舊＋加入新＋supersede 指向」，重用 G1（ADR-0047）與 G3（ADR-0049）既有對帳；為純協定/客戶端邏輯＋UI（🌐），不需 Tauri/Cloudflare/RN，與托管方案（需 HSM/KMS）非同量級。
- **與承諾一致：** 不違反 PRD §4「不提供金鑰輪替/撤銷」——§4 指**開放模式協定層**（Nostr 無全域撤銷，不變）；企業輪替是**公司範圍**的 allowlist/名冊動作（ADR-0044/0046），非宣稱全網撤銷。
- **冗餘取代托管：** 「不想丟歷史」的正解是多設備備援（安全），而非公司留鑰（危險）。

## 後果

- 正面：企業遺失/換機有解且**零後門**；同事端自動換人；大量複用既有名冊；可在此環境實作。
- 負面 / 已知限制：
  - **全設備皆毀時，舊歷史不可復原**——刻意為之，與「設備死亡」承諾一致；以「建議雙設備」緩解。
  - 舊 npub 尚未取件的離線 Gift Wrap 隨輪替**永久無法解密**（無金鑰），符合設計。
  - 輪替成員的聯絡人 remap **依賴組織通訊錄**：以名冊/組織群認得的同事自動換人；同事**手動加的**該員 npub（非名冊來源）不會自動 remap（沿用 ADR-0047「手動加的不被名冊移除」界線），需靠 UI 提示手動更新。
  - **單一管理者金鑰**仍為信任根（ADR-0047 既有限制）；能發名冊者即能輪替成員→需搭配佈建流程治理，多管理者/金鑰輪替仍為後續。
  - 若某客戶合約硬性要求「還原同一身分＋歷史」（選項 B），須**另立獨立「受監督版」ADR**，明列後門與定性改變，不進預設版。
- 後續行動：
  1. core `org-roster`：`OrgRosterMember` 加 `supersededBy?`，`diffRoster` 產出 remap 對映；純函式 TDD。
  2. 客戶端工作身分：收名冊 remap 本機聯絡人/組織群成員（舊→新 npub），保留舊訊息歸屬、UI「已更新金鑰」提示。
  3. 管理者佈建工具（G1 UI）新增「輪替成員」動作：撤銷舊、貼入員工提供的新 npub、發名冊、匯出 allowlist。
  4. 文件：企業部署指南加「建議雙設備登記」為遺失防護；PRD §13.6／ROADMAP G4 標記為此方案完成。
