# 0090. 官方網站、官方捐款與資金透明度（靜態站＋簽章 JSON runway）

- 狀態：已接受
- 日期：2026-07-12
- 相關文件：PRD.md §12（界線澄清）、docs/adr/0039（維護者簽章金鑰為信任根）、
  0089（relay 營運者贊助入口）、0075（自架 relay/節點營運）；apps/website（新增）

## 背景與問題

專案為**開源、永久免費**。官方本身需要一個對外門面與**自己的捐款管道**，收到的款項投入
**官方節點營運**與**部分人員獎金**；為維繫社群信任，分配方式須**定期公開**，並**即時顯示
「目前資金可營運官方節點多久」（runway）**。

兩個張力必須先解：

1. **零伺服器狀態的精神轉折**：本專案身分是「零伺服器狀態／serverless／本地優先」，連 relay
   都只轉發密文。而「管理後台＋即時 runway」天然傾向一台**有狀態的官方伺服器**——與此精神相斥。
2. **治理轉折（志願 → 有給）**：「部分人員獎金」使專案從純志願轉為有給貢獻者；這正是需要
   徹底透明的原因。此為**組織治理決策**，本 ADR 如實記錄並確保透明機制到位。

## 考量的選項（後台形態）

- 靜態站＋**維護者簽章 JSON**（採用）：無常駐伺服器；管理者定期更新已簽章資料檔，前端算 runway。
- 無伺服器函式讀付款方 API：Cloudflare Worker 定時抓餘額；較即時但受各平台 API 限制。
- 輕量後台（DB＋管理面板）：秒級即時，但引入有狀態官方伺服器與其安全/維運/被傳喚責任，最偏離 serverless。

## 決策

### 1. 硬隔離邊界（鐵則）

- 官網／透明度服務是**獨立的官方營運工具**，與 **E2E 通訊平面完全分離**：**獨立網域**、
  **永不接觸**使用者資料／金鑰／relay 流量／npub。它壞掉、被入侵或被傳喚，**都不得**危及通訊隱私保證。
- 官網**零使用者追蹤／分析／cookie**（與隱私品牌一致）。

### 2. 形態：靜態站＋維護者簽章 JSON（無常駐後台）

- **`apps/website`**（monorepo 新模組，比照 `apps/desktop`/`apps/mobile`）：純靜態站，
  訴求「開源・永久免費・隱私」、下載連結、官方捐款、透明度頁。以靜態託管（如 Pages）發佈。
- **官方捐款**：導流至 **GitHub Sponsors／Buy Me a Coffee／Liberapay／Lightning 位址**（同 ADR-0089
  四管道，`lightning` 為選填 LN 位址/LNURL，交外部錢包）；純外部連結，**無站內托管/錢包/抽成**。
  **不採 NIP-57 Zap**（見 ADR-0089「Zap 評估」）。
- **透明度資料 `funds.json`（維護者簽章）**：
  ```json
  {
    "balance": 1234.5, "currency": "USD",
    "monthlyBurn": 210.0,           // 官方節點成本 + 已承諾獎金
    "updatedAt": "2026-07-12T00:00:00Z",
    "allocations": [
      { "period": "2026-06", "nodeOps": 180, "bonuses": 120, "other": 0, "note": "…" }
    ],
    "sig": "<schnorr>", "pubkey": "<hex>"
  }
  ```
  - **簽章信任根**：沿用 ADR-0039「維護者簽章金鑰」模式（Fix First）——前端**驗簽通過才渲染**，
    使 CDN/GitHub/託管被入侵也**無法竄改數字**。建議用**專屬透明度金鑰**（與 relay-list 維護者金鑰
    分離，限縮爆炸半徑）；此金鑰**與任何使用者身分金鑰無關**。
  - **runway**：前端計算 `balance / monthlyBurn`，顯示「官方節點可續營運約 N（月/日）」，
    **並顯示 `updatedAt`**——誠實標示為「上次更新時的估算、非秒級即時」。
  - **定期分配公開**：`allocations` 陣列渲染為歷史分配報告（含節點營運與人員獎金分項）。
- **「後台」＝流程而非伺服器**：管理者離線編輯 `funds.json` → 以透明度金鑰簽章（repo 內附簽章
  腳本）→ 提交/發佈。無 DB、無 API、無登入面板。

## 理由

- **保住零伺服器狀態精神**：透明度不靠常駐後台，而靠「靜態檔＋簽章＋前端計算」，維運與攻擊面最小。
- **可驗證的透明**：簽章讓公開數字**防竄改**，比一台「請相信我的後台」更符合本專案的信任模型
  （信任根＝可驗證金鑰，非託管方善意）。
- **複用既有信任根模式**（ADR-0039），非另造機制。

## 後果

- 正面：對外門面到位；官方財務**可驗證透明**（含志願→有給的治理揭露）；serverless 精神不破。
- 負面 / 已知殘餘風險：
  - **runway 非秒級即時**：僅於 `funds.json` 更新時刷新；以 `updatedAt` 誠實標示（可接受）。
  - **餘額為自報**：簽章防第三方竄改，但**不證明銀行/錢包真實餘額**；透明度靠社群信任＋可稽核的
    分配歷史，非鏈上證明。**未來強化（選配、唯讀）**：官方 LN 收款位址的**公開 zap 收據（kind 9735）**
    可當第三方可查的捐款流水佐證——注意此為**唯讀消費公開資料**，非站內發起 zap、不碰付款（見 ADR-0089）。
  - **治理責任**：「部分人員獎金」的分配準則須由組織定義並落在 `allocations`；本 ADR 不代為決定分配比例。
  - **官方節點**與 relay 零狀態不衝突：官方節點仍是一般 relay（只轉發密文）；「營運多久」談的是**金流永續**，非資料留存。
- 後續行動 / 待辦：
  1. 立 `apps/website` 靜態站骨架與透明度頁（前端驗簽 + runway 計算）。
  2. 定義 `funds.json` schema、透明度金鑰產生與簽章腳本、發佈流程文件。
  3. 官方捐款導流連結與 §12 界線措辭；與 ADR-0089 共用捐款管道常數。
  4. 由組織定義獎金分配準則並首次公開；上線後更新 ARCHITECTURE.md 標記官網模組。
