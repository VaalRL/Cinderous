# 0112. 網頁／行動端的靜態加密——並修好「nsec 明文躺在 localStorage」

- 狀態：已接受（已實作）
- 日期：2026-07-14
- 相關文件：ADR-0004（core 以 @noble 為加密原語）、0053（OS 金鑰庫）、0054（加密儲存基質）、
  **0067（本地密碼——本 ADR 推翻其「瀏覽器＝假安全感」的推論）**、0111（訊息封存）

## 背景與問題

ADR-0111 落地時記了一條已知限制：「OPFS 是明文（與該平台的 localStorage 現況一致）」。
往下查，發現問題比那嚴重得多。

### 1. 桌面早就加密，web/mobile 全是明文

- **桌面**：`store_save` → `encstore::encrypt`（AES-256-GCM，金鑰在 OS 金鑰庫）✅
- **web/mobile**：localStorage 與 OPFS **全部明文**——訊息、聯絡人、群組直接可讀 ❌

### 2. 更嚴重的：**nsec 明文躺在 localStorage**

`browserKeyVault.setKey()` 直接 `localStorage.setItem(LS_PREFIX + pubkey, nsec)`。
**私鑰明文落盤。** 這與專案自己的紅線直接矛盾——程式碼裡甚至有一段遷移註解寫著
「抹掉 localStorage 內的明文私鑰……達成『私鑰不明文落地』」，但瀏覽器路徑本身就在製造它。

### 3. ADR-0067 的推論是錯的

`passlock.ts` 的註解寫著：

> 瀏覽器環境不提供此功能：瀏覽器路徑本就無 OS 金鑰庫，假裝有密碼保護會是 ADR-0067 否決的
> 「假安全感」。

**但「不提供」的結果不是誠實，而是上面第 2 點——nsec 明文落盤。**

Argon2id 包裹在瀏覽器提供的是**與桌面相同**的靜態保護：KEK 由密碼導出、**從不落盤**，
磁碟上的 blob 沒有密碼就打不開。它擋不住**頁面內的惡意 JS**——但桌面的 webview
**同樣擋不住**。兩者真正的差別只在「OS 帳號」那一道邊界，而那正是瀏覽器**沒有基線保護**、
**更需要**密碼的理由。

## 決策

### 1. 資料靜態加密：DEK = HKDF-SHA256(nsec)

localStorage 的值與 OPFS 的封存塊一律以 **AES-256-GCM** 加密（`nonce || ciphertext`，
`c1:` 前綴 ＋ base64）。金鑰由 **nsec** 導出——**不引入新的金鑰材料**，nsec 本來就是根秘密。

**為什麼是同步的 AES-GCM，不是 WebCrypto**：`localStorage.setItem` 是**同步**的，
`crypto.subtle` 是**非同步**的——直接打架。專案已有 `@noble/ciphers` 的**同步** AES-256-GCM
（`encryptBundle`/`decryptBundle`，配對搬家在用），直接複用，不引入新機制
（符合 ADR-0004：core 以 @noble 為加密原語）。

### 2. **關鍵推論**：nsec 不能明文落盤，否則第 1 點是演戲

DEK 由 nsec 導出。若 nsec 就躺在同一個 localStorage 裡，**金鑰等於附在鎖旁邊**。

故：**web/mobile 一律不明文儲存 nsec。**

- **行動端**：本來就**不持久化 nsec**（每次輸入）→ **金鑰不在磁碟上** →
  localStorage/OPFS 上的訊息**真的**解不開。加密立即為真。
- **瀏覽器**：改為只接受 **Argon2id 密碼包裹的 blob**。`browserKeyVault.setKey()` 收到明文
  **拒絕落盤**（並清除殘留的舊明文條目）——這是**強制點**，不是建議：呼叫端若忘了包裹，
  資料不會被寫出去，而不是靜默地明文寫出去。
  不設密碼的使用者 → **不持久化**，每次重新輸入 nsec（＝行動端現況，安全）。

### 3. 推翻 ADR-0067 的「瀏覽器＝假安全感」

`passwordLockAvailable()` 在瀏覽器改回 `true`。Argon2id 走 `@noble/hashes`，
**參數與 Rust 版刻意一致**（m=19456 KiB、t=2、p=1）——同一套威脅模型，不該因平台而漂移。

blob 格式（`{ v:1, kdf:"argon2id", m, t, p, salt, data }`）**與桌面完全相容**：既有的
`isWrappedValue()` 與 `UnlockScreen` 流程直接就能用，不必另建一套。

### 4. 遷移：舊的明文資料**必須**仍讀得出來

`openValue()` 對**無 `c1:` 前綴**的值**原樣回傳**（視為舊明文），並在下次寫入時自動轉成密文。

沒有這條，升級等於把所有既有使用者的資料變成亂碼。已有測試釘住。

### 5. 竄改的密文回 `null`，**不可**當明文回傳

否則竄改就變成**靜默的資料污染**。GCM 驗證失敗 → 該值視為不存在。

### 6. 拒收荒謬的 KDF 參數

包裹 blob 自帶 `m/t/p`。毀損或惡意的 blob 宣告 `m = 95 GiB` 時，解鎖不該真的去配置那麼多記憶體。
上限與 Rust 版一致（m ≤ 1 GiB、t ≤ 16、p ≤ 16）。

## 理由

- 「訊息加密」與「私鑰明文」放在一起是**自相矛盾**的。本 ADR 必須同時解，只解一半沒有意義。
- 密碼保護在瀏覽器**不是假安全感**——它是**唯一**能讓 nsec 不明文落盤又保有「記住我」的方法。
  ADR-0067 的推論把「不如桌面」誤當成「沒有價值」。

## 後果

- 正面：
  - **行動端**：訊息、聯絡人、群組、封存**全部密文落地**，且金鑰不在磁碟上 → **真的解不開**。
  - **瀏覽器**：nsec **不再明文落盤**（要嘛密碼包裹、要嘛不持久化）。
  - 桌面（Tauri）**不受影響**（本來就走 `encstore` ＋ OS 金鑰庫）。
  - blob 與桌面相容 → `UnlockScreen` 直接重用，不留平行實作。
  - 測試 +19（含「磁碟上是密文」「換 nsec 解不開」「舊明文仍讀得出來」「竄改回 null」
    「拒收荒謬 KDF 參數」「**金鑰庫拒絕明文 nsec**」）。全 repo **884 通過**。

- 負面 / 已知限制：
  - **瀏覽器不設密碼 → 每次重新輸入 nsec**。這是誠實的取捨：要「記住我」就得有密碼。
  - **擋不住頁面內的惡意 JS**（XSS、惡意擴充套件）——它能讀到記憶體中的 nsec。
    桌面的 webview 同樣擋不住；這是 webview 架構的固有邊界，不是本 ADR 引入的。
  - 瀏覽器的 `passChange` / `passDisable` / `passRescue`（ADR-0073 救援）**尚未實作**
    ——目前只有 enable / unlock。救援需要第二道包裹，另議。
  - **DEK 綁 nsec**：使用者輪替身分（ADR-0052）後，舊 nsec 加密的資料需以舊金鑰讀出再重寫。
    目前未處理——已知不一致。
