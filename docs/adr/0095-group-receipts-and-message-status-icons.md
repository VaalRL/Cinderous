# 0095. 群組送達/已讀分級（依成員數）＋訊息狀態擴充（failed）與眼睛圖示語言＋行動端狀態同步

- 狀態：已接受（已實作）
- 日期：2026-07-13
- 相關文件：ADR-0058（送達/已讀回條）、0027（群組扇出）、0088（在線元數據最小化）、0085/0087（行動端）；
  `packages/core/src/{group,receipt}.ts`、`packages/engine/src/backend/relay-backend.ts`、`packages/theme/src/icons.ts`

## 背景與問題

1. **群組沒有任何送達/已讀**：`receiveGroup` 不回任何回條、`sendGroupMessage` 不設 status——ADR-0058
   整套回條是**純 1:1**。使用者想知道群組「誰已讀」。
2. **前置缺陷（本 ADR 順帶修正）**：群訊在各成員端的識別**互不相同**——`wrapGroupMessage` 對每位成員各包
   一個 wrap，而收件端把**外層 wrap 的 event.id** 當訊息 id，發訊者則用 `evts[0].id`。同一則群訊在每台裝置
   上是不同的 id → **回條（以及回應/引用）根本對不回發訊者**。這是既有的潛在 bug，且是群組回條的前置條件。
3. **沒有「傳送失敗」狀態**：外送匣重試耗盡（`onGiveUp`）目前只 `console.warn`，UI 無從得知。
4. **行動端完全不顯示狀態**：`ConversationScreen` 沒有狀態渲染、`MobileApp` 沒掛 `onMessageStatus`。

## 考量的選項

**群組回條的成本**（決定為何要分級）：一則群訊每位成員各回一則**儲存型**（kind 1059、7 天 TTL）回條給
發訊者——約等於**每則訊息的中繼事件數翻倍**；回條積在發訊者收件箱，而 `dm` 訂閱**沒有 since/limit、
每次重連抓全部** → 直接拖慢重連。活躍度隨人數成長時，全網回條 ≈ **r·N²**、每裝置收件 ≈ r·N。
名單在大群也失去可讀性（>15 人的已讀名單是噪音）。

- **A. 全群一律追蹤**：最完整，但 N² 成本與收件箱膨脹在大群失控，且與 ADR-0088 元數據最小化相悖。**不採**。
- **B. 一律不做**：最省，但使用者需求未解。**不採**。
- **C. 依成員數分級（採用）**：把「有用」與「可負擔」的區間框出來——三條 knee（名單可讀性、收件箱膨脹、
  N² 全網量）都落在 ~10–15，取保守值分兩段。

## 決策

### 1. 群組回條依成員數分級（`groupReceiptMode(memberCount)`，含自己）

| 成員數 | 模式 | 行為 |
|---|---|---|
| **≤5** | `list` 名單制 | 記錄每位成員的送達/已讀；UI 顯示**誰已讀** |
| **6–10** | `count` 計數制 | 同樣收回條，但 UI 只顯示**已讀 M/N**（不列名，降噪與觀感暴露） |
| **>10** | `off` 完全不記 | **不送送達、也不送已讀回條**；零額外流量，收到別人塞的回條也丟棄 |

- 計數制**不省流量**（要算 M/N 就得收齊每位成員的回條）；它省的是**畫面噪音與列名**。真正省成本的是 `off`。
- 已讀仍受 ADR-0058 的 **opt-in＋互惠**約束（自己關閉已讀＝不送、也不顯示他人已讀）；送達則自動。
- 群組已讀為**水位語意**，且**分別送給每位發訊者**（群訊來自多人，各發一則指向「他最新那則」的回條）。

### 2. 前置修正：群訊識別改用**內層 `rumor.id`**

rumor（kind/時間/tags/內容/作者）對每位收件人**完全相同** → 其雜湊也相同，且 `openWrap` 會核對雜湊。
故 `rumor.id` 是**跨成員一致且可驗證**的群訊識別。`wrapGroupMessage` 改為回傳 `{ id, events }`；
收發兩端一律以此為訊息 id（外層 wrap id 僅用於 wrap 去重）。**回條/回應/引用自此對得起來。**

群組回條在 rumor 內層帶 `["g", groupId]`（中繼看不到），發訊者據此知道要更新哪個對話。
注意：`receiveDm` 的群組路由必須**只吃 CHAT/GROUP_CONTROL**，否則帶 `g` 的回條會被誤路由而吞掉。

### 3. 訊息狀態擴充：新增 `failed`

`MessageStatus = "sending" | "failed" | "sent" | "delivered" | "read"`，rank 依序 0–4——
`failed` 排在 `sending` 之後、`sent` 之前：可從傳送中推進到失敗；失敗後若延遲成功仍能推進到 `sent`；
較晚到的 `failed` 不會覆蓋已成功的 `sent`。外送匣 `onGiveUp` → 標記 `failed`。
群訊扇出：**任一** wrap 被中繼接受＝`sent`；**全部**放棄才算 `failed`。

### 4. 狀態圖示語言：眼睛的張合＝「對方看到多少」

| 狀態 | 圖示 | 色 |
|---|---|---|
| `sending` 傳送中 | 沙漏 | 灰 |
| `failed` 傳送失敗 | 循環箭頭（重試） | **紅** |
| `sent` 已送出 | **閉眼**（單一下弧、無瞳孔） | 灰 |
| `delivered` 已送達 | **半開眼**（壓低上眼瞼＋小瞳孔） | 灰 |
| `read` 已讀 | **張開眼**（完整杏形＋大瞳孔） | **主色＋較粗線** |

以 SVG 實作（`currentColor`）而非 emoji——emoji 無法依主題/主色重新著色，做不到「主色＋粗體」。
幾何放 `@cinder/theme`（跨前端 SSOT，ADR-0080），桌面與行動端共用同一套。

### 5. 行動端狀態同步

`MobileApp` 掛 `onMessageStatus`／`onMessageReceipts`，`ConversationScreen` 在自己送出的氣泡旁渲染同一套圖示。
並補上**已讀回條開關**與開對話時的 `markRead`——否則因互惠機制，行動端的 tick 永遠到不了「已讀」。

## 後果

- 正面：小群看得到「誰已讀」、中群看得到「已讀 M/N」、大群零成本；傳送失敗看得見；群訊 id 一致性缺陷修掉；
  桌面/行動端狀態語言一致。
- 負面 / 已知殘餘風險：
  - 6–10 人群仍有 N−1 回條/則的流量（計數制不省流量，只降噪）——這是取捨後接受的成本。
  - 發訊者**裝置**在計數制下仍逐一收到回條（技術上知道是誰），計數制只是**不顯示**名字，非密碼學隱藏。
  - 群訊 id 語意變更：**舊有**群組歷史仍是舊的 wrap id（不遷移、ids 為不透明值）；新訊息起才一致。
  - 行動端圖示目前用內嵌 `<svg>`（RN-web 可行）；移植真正的 React Native 時需換成 `react-native-svg`
    （幾何來自共享 SSOT，只換元素名）。
  - 行動端尚未渲染群組「誰已讀／M/N」（資料已接進 `onMessageReceipts`，只差 UI）。
- 測試：core（群訊 id 跨成員一致、分級邊界 5/6/10/11）、engine（小群記錄每成員送達、群訊 id 一致、
  大群完全不記）、desktop（眼睛圖示語言：閉眼無瞳孔／半開有瞳孔／張開主色粗線／失敗紅）。
