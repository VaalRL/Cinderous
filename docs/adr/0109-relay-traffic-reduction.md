# 0109. 中繼流量削減：自適應心跳、心跳自報節奏、合併訂閱、增量收件箱

- 狀態：已接受（已實作）
- 日期：2026-07-14
- 相關文件：ADR-0059（30 秒心跳）、0088（在線元數據最小化）、0034/0036/0039（多中繼池）、
  0056（離線留言/7 天 TTL）、0107（自封副本——讓收件箱翻倍）；NIP-01（REQ 多 filter）、NIP-59

## 背景與問題

盤點 Cloudflare 免費額度的消耗來源。中繼是 **SQLite-backed Durable Object ＋ WebSocket
Hibernation**，計費關鍵：

- **每一則進來的 WS 訊息 ＝ 1 個 DO request**（EVENT / REQ / CLOSE / AUTH 都算）
- **中繼送出去給客戶端的事件不計 request**（只計 duration）← 本 ADR 的省法建立在這一點上
- **只有非 ephemeral 事件寫 row**（心跳是 kind 20000，ephemeral → 不寫 row）

### 量測：心跳佔 92%

每位「同時在線」使用者每天打到中繼的 request：

| 來源 | requests/日 | 佔比 |
|---|---:|---:|
| **心跳**（`HEARTBEAT_MS = 30_000`） | **2,880** | **~92%** |
| 輸入中（App 層已有 1 秒節流；P2P 通道開通時卸載） | ~200 | 6% |
| 訊息（每則 3 顆事件：給對方 ＋ 自封副本 ＋ 對方回的送達回條） | ~150 | 5% |
| 訂閱（每次連線 **9 個** REQ） | ~30 | 1% |

**免費層 10 萬 requests/日 → 約 30 位同時在線使用者就見底。**
而 ADR-0107 讓多裝置真的能用了——**一個人開手機＋電腦就是兩份心跳**。

（row writes 每人約 150/日，上限 10 萬/日 → 非綁束條件。儲存 5GB、rows read 5M/日 亦非綁束。
**requests 才是天花板，而心跳就是它的 92%。**）

### 核心觀察：閒置時的心跳是在對空氣廣播

心跳每 30 秒發一次，**不管有沒有任何聯絡人在線**。而大多數時間，使用者的聯絡人一個都不在線。

## 決策

### 1. 心跳基礎間隔 30s → 60s（S2）

`OFFLINE_TIMEOUT_MS = 60_000` 原本只有 2 倍餘裕；改 60 秒心跳需一併放寬容忍窗（見決策 3）。

### 2. 自適應心跳：沒人在線就放慢到 5 分鐘（S1）

- 任一聯絡人在線 → **ACTIVE（60 秒）**
- 無人在線 → **IDLE（5 分鐘）**

**關鍵：這不會讓「顯示上線」變慢。** 因為**收**心跳是免費的（中繼→客戶端不計 request），
機制如下：

1. 每個客戶端**連線時一律立刻發一次心跳**（`start()` 既有行為，本機制的基礎）。
2. Bob 上線 → 立刻發心跳 → 我**免費**收到。
3. 我偵測到「從沒人在線變成有人在線」→ **立刻補發一次心跳** ＋ 切換到 ACTIVE。
4. Bob 收到 → 看見我在線。**總延遲約一個 RTT。**

**防風暴**：只在 **IDLE→ACTIVE 的轉換**時立刻補發，不是每收到一則心跳就發
——否則兩端會互相觸發成心跳風暴。

實測（見測試）確認這是一個**兩步握手且在一輪後收斂**：
A 發（IDLE）→ B 收到、轉 ACTIVE、補發 → A 收到、轉 ACTIVE、補發 → B 收到時
**已經是 ACTIVE**（`wasIdle === false`）→ **不再回發**。收斂。

在 N 個互為聯絡人的節點上，每個節點在轉換時**至多多發一顆** → 總量 **O(N)，不是 O(N²)**。

若沒有第 1 點（連線即發），兩台都閒置時會互相看不見達 5 分鐘。那是這個設計唯一的死穴，
故連線即發是**不變式**，不是最佳化。

### 3. 心跳自報節奏——否則閒置者會被誤判離線

這是自適應心跳最容易被忽略的坑：

`PresenceTracker.statusOf()` 用**固定**的 `OFFLINE_TIMEOUT_MS` 判定。若我處於 IDLE
（5 分鐘一次），而有人正在看我，他會用 ~150 秒的窗把我判成**離線**——但我其實在線。

這在**單向聯絡人**時真的會發生：Bob 加了我、我還沒加他 → 我的 presence 訂閱不含 Bob
→ 我看不到他的心跳 → 我保持 IDLE → 但 Bob 一直在看我。

**解法**：心跳帶 `["hb", "<間隔秒>"]` tag；觀察端據此計算容忍窗
（`tolerance = 2.5 × 宣告間隔`，無宣告則退回預設值）。

**這不洩漏任何新東西**：心跳節奏本來就能從事件時戳直接觀察出來，明寫只是把**已可推得**的
資訊講清楚，不與 ADR-0088（在線元數據最小化）衝突。

（自癒：Bob 一旦傳訊息給我，`ensureContact(bob)` 會把他加進我的聯絡人 → 我的 presence 訂閱
納入他 → 我看見他在線 → 轉 ACTIVE。單向窗口只存在於「加了但從未互動」。）

#### 3b. **P2P 卸載路徑必須同樣自報節奏與觸發喚醒**（初版遺漏，事後補上）

ADR-0088 (e) 會在 P2P 通道開通時把在線信標**卸載到資料通道**；而 `allP2P`（所有聯絡人都有
通道）時，`beat()` **完全不發 relay 心跳**——**P2P 是唯一的在線信號**。

初版只改了 relay 心跳，漏了這條路，造成一個**由本 ADR 引入的真 bug**：

- P2P 在線狀態是由 `beat()` 送出的 → **節奏＝心跳節奏** → 閒置時每 5 分鐘才一則；
- 但收端 `observe(peer, nowSec())` **不帶節奏** → 退回預設的 150 秒容忍窗；
- → **P2P 對端在 150 秒後把在線的我判成離線**，而且因為 `allP2P` 不發 relay 心跳，
  沒有任何備援信號。

且 `onPresence` 也沒有 IDLE→ACTIVE 的喚醒補發 → 「唯一在線的聯絡人只走 P2P」時我永遠不會加速。

修正：`DataMessage.presence` 增加 `hb`（自報節奏，選填以相容舊版），並讓 `onPresence`
與 relay 心跳走**完全相同**的兩步：帶節奏 `observe()` ＋ 轉換時喚醒補發。

**教訓**：ADR-0088 說 P2P 在線狀態與 relay 心跳「**同源處理**」——同源就必須**同樣完整**。
任何為心跳新增的語意（節奏、喚醒），都必須同時在兩條路徑上實作，否則卸載路徑會靜默退化。

### 4. 9 個 REQ 合併成 1 個（S3）

已驗證可行：

- **客戶端**：`onEvent: (_sub, event) => this.onEvent(event)` **完全忽略 subId**，一律依
  `event.kind` 分派；從不註冊 `onEose`、從不 `unsubscribe`。
- **中繼**：`buildEntry()` 已**跨所有 filter** 收集 kinds 建索引、`handleReq()` 已**逐 filter**
  重播並以 event id 去重、即時扇出是 `filters.some(...)`（OR 語意）。
  **多 filter REQ 早就完全支援**，這是純客戶端改動。

每次（重）連線的訂閱成本：**9 個 request → 1 個**。`resubscribe()` 有 11 個呼叫點
（含 `ensureContact`——每加一個聯絡人就重掛一次）。

### 5. 收件箱增量抓取（`since`），但**必須退讓 2 天**（S4）

`dm` 訂閱沒有 `since` → **每次重連都重抓整個 7 天收件箱**（ADR-0107 之後翻倍）。

**陷阱**：NIP-59 的 `TIMESTAMP_JITTER_SECONDS = 2 * 86_400`——剛發出的 Gift Wrap，其
**外層** `created_at` 可能落在**兩天前**（隱私設計）。若天真地用 `since = 上次連線時間`，
**會漏訊息**。

故：`since = 該中繼上收過的最大外層 created_at − 2 天`（jitter 窗）。
水位**逐中繼**追蹤（不同中繼的事件集合不同，共用一個全域水位會漏事件）。

**這省的不是 request**——一個 REQ 就是一個 request，回傳幾筆不影響。它省的是
**rows read／DO duration／頻寬／客戶端啟動延遲**（ADR-0107 讓收件箱翻倍後更有感）。

水位只存在記憶體（本 session）：**session 內的重連**走增量，**App 重啟**仍抓全量一次。
重連遠比重啟頻繁，這已取得大部分效益，且不需動儲存層。

## 理由

- 省 request 的唯一有意義的槓桿就是心跳（92%）。其餘（回條、群訊扇出）動了只省個位數百分比，
  卻要犧牲設計完整性——不值得。
- 自適應心跳之所以「免費」，完全建立在「**收**事件不計費」這個計費事實上；離開這個前提，
  整個設計不成立，故明確記於此。
- 合併訂閱與增量收件箱是**純清理**：不改變任何語意，中繼端早已支援。

## 後果

- 正面：
  - 心跳：閒置 2,880 → **288**/日；ACTIVE 亦由 2,880 → **1,440**/日。
    典型使用者（多數時間無人在線）**總量降約 90%**。
  - 天花板由約 **30 位**同時在線推到 **數百位**。
  - 訂閱成本每次連線 9 → 1 個 request。
  - 重連時收件箱抓取量由 7 天降至約 2 天（rows read／頻寬／啟動延遲）。
  - 順帶收斂：`HEARTBEAT_INTERVAL_MS` 在 core 宣告卻**無人使用**（引擎自己另定了一份
    `HEARTBEAT_MS`）——已刪除重複；`TIMESTAMP_JITTER_SECONDS` 改由 core 匯出（SSOT），
    避免引擎重寫一份而與 NIP-59 的實際抖動窗失聯（一旦失聯就會**靜默漏訊**）。
  - 測試 +14（core 8 ＋ engine 6：IDLE 節奏、喚醒握手、**防風暴收斂**、單一 REQ、
    `since` 退讓 2 天、**P2P 卸載路徑亦自報節奏**）。全 repo **833 通過**。

- 負面 / 取捨：
  - **「離線」的顯示變慢**：ACTIVE 對象約 150 秒（原 60 秒）；IDLE 對象約 12.5 分鐘
    （`2.5 × 5 分鐘`）。**「上線」仍是即時的**（約一個 RTT）——這是刻意的不對稱：
    上線需要即時（要立刻能聊天），離線不需要。
  - 心跳多一個 `hb` tag（節奏）。如上所述不構成新的元數據洩漏。
  - 增量收件箱的水位不持久化 → App 重啟後仍全量抓一次（可接受；避免動儲存層）。

- 已知限制：
  - 若聯絡人的心跳因中繼問題未抵達，我會停在 IDLE（降級但不失效——對方仍會在 5 分鐘內
    看到我的 IDLE 心跳並轉 ACTIVE，隨後我也會看到他）。
  - 心跳仍會發往**連線池中的每一座**中繼。若日後新增第二座**自家** Cloudflare 錨點，
    心跳成本會翻倍——那應該在該 ADR 中一併評估。
