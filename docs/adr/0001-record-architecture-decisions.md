# 0001. 採用 ADR 記錄架構決策

- 狀態：已接受
- 日期：2026-06-29
- 相關文件：CLAUDE.md〈全域硬規則〉、docs/adr/README.md

## 背景與問題

Nostr Buddy 的隱私與加密取捨牽涉大量「為何不這麼做」的脈絡（例如為何捨棄 Web 版、為何不提供金鑰備份、為何接受上線時序洩漏）。這些理由若只散落在 PRD 與聊天紀錄，日後容易被重新爭論或無意間推翻。

## 考量的選項

- 選項 A：只在 PRD.md / ARCHITECTURE.md 內以散文記錄決策。
- 選項 B：建立獨立的 ADR 目錄，一決策一檔，不可竄改、可被取代。

## 決策

採用選項 B。於 `docs/adr/` 建立 ADR 機制，自此所有架構/設計層級決策都需新增 ADR。

## 理由

ADR 將「規格現況（PRD/ARCHITECTURE）」與「決策理由與取捨」分離，提供可追溯、不可竄改的決策歷史，避免重複爭論並讓新成員快速理解脈絡。

## 後果

- 正面：決策可追溯；PRD/ARCHITECTURE 維持精簡。
- 負面：每次重大決策多一份文件維護成本。
- 後續行動：CLAUDE.md 全域硬規則加入「決策需寫入 ADR」條目。
