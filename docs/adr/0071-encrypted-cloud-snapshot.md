# 0071. 加密雲端快照（三檔模式、可尋址取代、opt-in）

- 狀態：已接受
- 日期：2026-07-09
- 相關文件：docs/adr/0002（隱私基線）、0009（多設備收斂語意）、0028（無前向保密的既知取捨）、
  0042（容量上限雙端驗證模式）、0045（多身分）、0048（企業政策開關）、0056/0065（relay 儲存與 TTL）、
  0058（opt-in 慣例）、0067（nsec 主金鑰定位）、0068（群組快照廣播）、0069（容量治理）、
  0070（加密備份碼）、packages/core/src/sync.ts、relay/

## 背景與問題

1. nsec 落地新裝置只能拿回「身分＋7 天離線窗」；聯絡人靠開機廣播自癒（0061/0066/0068），
   但歷史、單向好友、封鎖清單、設定、以及**自己發出的訊息**都不會跟著走。
2. P2P 配對同步（ADR-0009/D4）能搬全量，但要求**兩機同時在線**——手機在外、桌機關機
   的常態下體驗不成立；使用者的參照系是 LINE 級「新機登入即還原」。
3. 既有的「自封 Gift Wrap」構想（增量 delta 寄給自己）已具備正確的隱私性質——
   把它推廣到「狀態快照」即可讓還原不依賴舊機在線。
4. 代價要誠實面對：快照密文長駐 relay 擴大回溯解密暴露面（靜態 ECDH 無 FS，
   ADR-0028）；免費層儲存空間是唯一受壓的計量維度（請求/時長/寫入列增量皆 <1%，
   心跳仍是先撞頂者）。

## 考量的選項

- 選項 A：僅 P2P 配對（D4）——隱私最小暴露，但兩機同時在線的要求讓還原體驗不成立。
- 選項 B：**加密雲端快照，三檔模式 opt-in**（採用）——見決策。
- 選項 C：快照預設開啟——否決：隱私優先是預設值的鐵則；暴露面擴大必須知情同意
  （同 0058 已讀回條的 opt-in 慣例）。
- 選項 D：每次備份堆新 blob——否決：儲存無限增長；採可尋址取代讓儲存與時間無關。

## 決策

1. **快照事件**：內容以 NIP-44 加密給**自己的 pubkey**（nsec 持有者可解、relay 只見
   密文，隱私等級同離線私訊）；以 **NIP-33 可尋址事件**發佈（`d` tag＝裝置 id），
   relay 對每台裝置**只保留最新一顆**、新的取代舊的。
2. **三檔模式（設定切換，預設關）**：
   - **關閉**（預設）：不發佈；切換到關時**主動刪除 relay 上既有快照**（purge on
     disable——「已關閉」必須立即為真），並彈確認說明新裝置將無法自動還原。
   - **基本**：聯絡人、群組、封鎖清單、設定、已讀位置（KB 級；無訊息內容）。
   - **完整**：基本＋近期訊息（壓縮後受單顆上限約束；百 KB 級）。
3. **範圍與治理**：逐身分獨立設定（0045）；模式本身屬可變狀態、經同步通道以 LWW
   傳播（各裝置自動跟隨）；企業名冊政策增 `disableCloudBackup`（0048 模式），
   組織可硬性禁止工作身分狀態上雲。
4. **保存與容量**（relay 端強制，0042 雙端驗證模式）：快照專屬 kind 採
   「**30 天過期＋每次備份刷新**」（活躍者永不過期、棄用帳號自動回收；不適用
   0065 的 7 天上限——該上限治理的是收件匣 gift wrap）；單顆壓縮後上限 256KB；
   每 pubkey 裝置（`d` tag）數上限 5；發佈節奏＝變更觸發防抖＋每日至多一次。
5. **新裝置還原**：nsec（或 0070 備份碼）匯入 → 訂閱自己的快照事件 → 以 0009 收斂
   語意（id 去重＋LWW，交換律）合併**所有裝置**的快照 → 上線。多裝置各自備份
   無需協調，合併結果保證一致。
6. **與既有機制的分工**：日常同步走小顆自封 delta（D4b，未來）；快照管「新裝置
   引導」與「7 天窗外的補洞」；P2P 配對（D4）降級為完整歷史搬運＋雙機在線時的
   即時通道；關閉模式下優雅降級回「7 天窗＋0068＋P2P」。

## 理由

- 體驗到位：新機「掃備份碼＋密碼→拉快照→秒級還原」——LINE 級體驗的八成，
  差距（帳密登入、全滅永存）是 E2E 與去中心化的本質代價且已有緩解（0070、30 天刷新）。
- 容量溫和：請求（20:1 折算）/時長（休眠化後）/寫入列三條帳增量 <1%；儲存為
  `用戶×裝置×上限` 的**平坦公式**（萬人級才逼近 5GB），撞頂順序仍是心跳先到，
  撞頂治理沿用 0069（accepting/draining）。
- 原則自洽：opt-in＋預設關＋purge-on-disable＋分層範圍，把「隱私 vs 體驗」的
  選擇權還給使用者；意外收穫——自己的快照含自己的群組，解掉 0068 的殘留缺口
  （管理員換機自救）。

## 後果

- 正面：新裝置還原不再依賴舊機在線；「拿不回清單」（歷史/單向好友/封鎖/設定/
  自己的訊息）在開啟模式下全數閉合；儲存有界、可治理。
- 負面／已知殘餘風險：
  - 開啟「完整」模式＝近期對話密文長駐 relay；無 FS（0028）下若 nsec 日後外洩，
    側錄型攻擊者可回溯解密——UI 文案須誠實（「快照由身分金鑰保護；本地密碼
    只保護這台裝置」），且過期刷新無法對抗持續存檔的攻擊者。
  - relay 需新增可尋址取代語意與快照配額（kind、size、d-tag 數）——事件契約變更。
  - 快照不含媒體附件（P2P 傳輸的檔案不回溯）。
- 後續行動 / 待辦：依 ROADMAP Phase J 施工（J1 relay 端 → J2 發佈端 → J3 還原合併
  → J4 設定 UI＋政策）；完成後更新 ARCHITECTURE.md 與 docs/SECURITY.md 威脅模型。
