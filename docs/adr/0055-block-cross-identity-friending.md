# 0055. 禁止跨身分互加好友（保護分身不可連結）

- 狀態：已接受
- 日期：2026-07-08
- 相關文件：ADR-0045（多身分並存）、0002（隱私元資料基線）；PRD §6/§7（威脅模型：中繼站不可見社交圖譜）；ARCHITECTURE §4/§11（多設備自封同步）

## 背景與問題

多身分（ADR-0045）的**唯一目的是區隔**——讓同一人的身分 A 與 B **對外不可連結**（真名 vs 匿名等）。但目前 `addContact` 只擋「已封鎖 / 已是聯絡人」，**沒擋加自己的身分**：

- **加自己作用中身分**（self-contact）：會訂閱到自己的 presence/DM，收到自己送的訊息、看到自己上線——明確 bug。
- **加自己另一個身分（A↔B）並傳訊**：把分身連結**主動送給最不該信任的中繼站**（PRD §6 對手 #1）。連結機制：
  1. 同裝置/同 IP 連同一座 relay；
  2. **連線關聯**——A 的連線訂閱「A 的收件匣」（`#p:[A]`）又在同連線送 Gift Wrap 給 B（p-tag=B），relay 可推出「A 正發訊給 B」（即使 wrap 藏了寄件人）；
  3. **切換時序**——一次一身分（reload 切換）造成「A 離線→B 上線回訊」的可辨識 pattern。

評估過「只在企業模式擋」，但方向是反的：企業是**封閉自架節點**（allowlist 已認得每個成員、且工作/個人身分鎖不同座天然隔離），連結風險低；**個人/公開 relay 才是不可連結最重要、風險最高之處**。

## 決策

**全模式禁止把自己的任一身分加為好友，並從介面明確阻擋。**

1. **App 層（主）：** 加好友前解出 npub，若命中 `profilesState.profiles` 任一 pubkey（作用中或其他自身身分）→ 丟 `"self-identity"`，`AddContact` 顯示明確訊息（i18n `contact_addSelf`）。此為唯一知道「使用者所有身分」的層。
2. **後端（防禦）：** `RelayChatBackend.addContact` 加 `pubkey === this.self.pubkey` 防呆，擋作用中身分（App 被繞過時的底線）。
3. **範圍：** 個人與企業模式皆擋（個人/公開 relay 才最需要）。
4. **正解替代：** 「跨裝置給自己留言」用既有**多設備自封同步**（NIP-17 自己加密寄給自己，同一身分，ARCHITECTURE §4/§11）——不會製造跨身分連結。

## 理由

- **保住多身分的核心價值**：不讓區隔被自我連結抵消。
- **self-contact 是 bug**：訂閱迴圈、自我上線，本就該擋。
- **企業天然隔離**：工作身分鎖公司座、個人走公開座，跨身分傳訊本就被 relay 隔離擋掉，無需特別放行。

## 後果

- 正面：分身對中繼站維持不可連結；介面有清楚阻擋訊息；防呆同時修掉 self-contact bug。
- 負面 / 已知限制：
  - **無法對自己的分身傳訊**（設計如此）；需要跨裝置同一身分同步時走多設備自封。
  - 阻擋為**客戶端層**：改版客戶端的使用者仍可繞過——那是其自身的隱私取捨，非本產品預設路徑（與 PRD §13.5「管得到身分＋節點、管不到人」一致）。
- 後續行動：i18n `contact_addSelf`/`contact_addInvalid`；App `addContactGuarded`；後端 self-guard；AddContact 顯示阻擋訊息（皆本批）。
