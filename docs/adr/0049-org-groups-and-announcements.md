# 0049. 組織群組與公告（管理者佈建）

- 狀態：已接受
- 日期：2026-07-05
- 相關文件：docs/adr/0027（群組成對扇出）、0047（組織名冊）、0044（allowlist）；PRD §13、ROADMAP G3

## 背景與問題

企業需要**管理者建立的部門群組**（成員自動入群）與**公告頻道**（單向、成員不能回發）。應延伸既有群組模型（ADR-0027）與名冊機制（ADR-0047），不另造。

## 決策

1. **組織群組經名冊分發：** `OrgRosterDoc` 加可選 `groups: OrgGroup[]`，`OrgGroup = { id, name, members: pubkey[], announce? }`。客戶端採用名冊時**對帳本地群組**——加入名冊新群、移除名冊外的組織群（權威管理，比照聯絡人對帳）。群組 `admin` = 名冊管理者。
2. **公告 = announce 群組：** `Group` 加 `announce?` 旗標。**發文授權**以純函式 `canPostToGroup(group, sender)` 判定：announce 群僅**管理者**可發、一般群任何成員可發（皆須為成員）。
   - **接收端強制：** 群訊接收時以 `canPostToGroup` 取代原本的「sender ∈ members」檢查——announce 群非管理者的訊息一律拒收。
   - **UI：** announce 群且自己非管理者時**隱藏輸入框**（唯讀）。
3. **訊息機制不變：** 沿用 ADR-0027 Gift-Wrap 成對扇出；只新增「分發（名冊）」與「發文授權（announce）」兩件事。

## 理由

- **最大複用：** 群組分發搭名冊採用機制（ADR-0047）、訊息搭成對扇出（ADR-0027）；只加一個旗標與一個純函式。
- **授權集中且可測：** `canPostToGroup` 純函式，接收端與 UI 共用同一判準；管理者為信任根（名冊簽章）。
- **對帳一致：** 組織群組與組織聯絡人皆由名冊權威管理，離職/改組即反映。

## 後果

- 正面：管理者集中建部門群與公告頻道、成員自動入群；公告單向唯讀。
- 負面 / 已知限制：
  - **announce 強制為客戶端層**（誠實客戶端拒收非管理者訊息；惡意客戶端可自行顯示，但無法讓他人接受）。真正硬牆仍是「非成員無法發訊」與 relay allowlist。
  - 成員本地可離開組織群，但下次名冊採用會**重新加入**（組織權威管理）；如需「允許退出」為後續政策。
  - 群成員清單以名冊為準；名冊外的手動群不受影響。
- 後續行動：core `announce`/`canPostToGroup` + 名冊 `groups`（本批）；後端對帳與接收強制；UI 唯讀閘門；管理者佈建工具的群組/公告編輯。
