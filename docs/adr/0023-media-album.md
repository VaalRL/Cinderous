# 0023. 媒體相簿（Album，M7）

- 狀態：已接受
- 日期：2026-07-01
- 相關文件：docs/ROADMAP.md（Phase E / M7）；docs/adr/0017（WebRTC 檔案傳輸）、docs/adr/0022（語音訊息）

## 背景與問題

M7 相簿：把對話中收發的圖片聚合為可瀏覽的媒體庫。希望**複用**既有 A4 WebRTC P2P
檔案傳輸，不新增協定或後端狀態。

## 決策

- **來源複用**：圖片即 `mime` 為 `image/*` 的檔案訊息（走 A4 P2P 檔案通道）。相簿
  由對話訊息即時衍生——`ConversationWindow` 從 `messages` 篩出帶 `url` 的 `image/*`
  檔案，無需額外後端狀態或持久化欄位。
- **三層檢視**：
  1. **內嵌縮圖**：`FileLine` 對 `image/*` 檔渲染 `<img>` 縮圖（取代一般檔案卡）。
  2. **相簿格狀**：工具列 🖼️（帶張數徽章）開啟本對話所有圖片的格狀模態框。
  3. **燈箱**：點縮圖以全螢幕覆蓋層放大檢視，點擊關閉。
- **送出端一致**：延續 ADR-0022，送出端檔案亦保留本機 `blob:` URL，故自己送出的
  圖片也進相簿並可放大。

## 後果

- 正面：純前端、零協定/後端改動；站在 A4 之上（分塊、進度、P2P 加密沿用）。經
  真實 relay + 真實 WebRTC 兩 context E2E 驗證（內嵌縮圖 → 相簿聚合 → 燈箱放大）。
- 負面 / 未來：v1 相簿範圍為「單一對話、目前 session 收發過且仍持有 blob 的圖片」；
  跨 session 需搭配原生持久化（B4 SQLite）落地媒體位元組後重建（尚未接線）。無相簿
  分頁/日期分組/影片縮圖；大量圖片的記憶體（blob URL）回收策略待最佳化。
