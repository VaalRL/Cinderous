# 0099. 中繼站實作 NIP-01 可取代事件（收斂 ADR-0035）＋修正 kind 10038 撞號

- 狀態：已接受（已實作）
- 日期：2026-07-13
- 相關文件：ADR-0035（relay hint／可取代事件，當時標注「自家 MessageStore 未實作」）、
  0039（維護者簽章 relay 清單）、0047（組織名冊）、0071（可尋址快照）、0092（節點自報）；
  `relay/src/{message-store,sql-message-store,relay-core}.ts`

## 背景與問題

### 1. 可取代事件從未實作 → 正在囤積（線上問題）

ADR-0035 就記過「可取代事件（自家 MessageStore 未實作）」。後來 ADR-0071 補了**可尋址**
（NIP-33，30000–39999，快照），但 **NIP-01 可取代（10000–19999）始終沒做**——這些事件走一般
`put()`，**只會累積、不會取代**。

Cinder 用到的可取代 kind：`RELAY_LIST_KIND`(10037)、`ORG_ROSTER_KIND`(10038)、節點自報。

實際後果：**health-check cron 每小時**把簽章 relay 清單發佈到每座健康 relay。清單事件沒有
`expiration` 標籤 → 吃預設 7 天 TTL → **7 天內囤積約 168 份重複清單**。客戶端每次連線訂閱
`{kinds:[10037], authors:[maintainer]}` 就得**把這 168 份全部下載一遍**。功能上沒壞（客戶端有
`shouldAdoptList` 去重），但是純粹的浪費，且會隨時間持續惡化。

### 2. kind 10038 撞號（潛伏的正確性錯誤）

`ORG_ROSTER_KIND = 10038`（ADR-0047）與 `NODE_ATTEST_KIND = 10038`（ADR-0092）**同一個 kind**。

在沒有取代語意時尚屬良性（訂閱都帶 `authors` 過濾）。但**一旦實作可取代語意，它就變成主動的錯誤**：
可取代是以 `(kind, pubkey)` 為鍵——同一把金鑰的**節點自報會直接覆蓋掉組織名冊**。

## 考量的選項

- **A. 新增獨立的 `putReplaceable` 與資料表**：直觀，但記憶體版與 SQL 版都要各加一套（含索引、
  prune、query），純屬重複。
- **B（採用）. 認清「可取代 ≡ `d` 為空字串的可尋址」**：NIP-01 的可取代事件就是位址元件為空的
  可尋址事件。既有 `putAddressable`（記憶體版與 SQL 版都已實作、已測、含配額/TTL/purge）
  的鍵是 `(kind, pubkey, d)`，而 `dTagOf` 對沒有 `d` 標籤的事件**正好回空字串**——
  於是 `(kind, pubkey, "")` 天然就是「每 (kind, pubkey) 只留一顆」。**只需把路由接上去。**

## 決策

1. **`isReplaceableKind(kind)`**：`kind === 0 || kind === 3 || (10000 ≤ kind < 20000)`（NIP-01）。
   `relay-core` 改以 `isReplaceableOrAddressable()` 決定是否走取代路徑——兩者共用 `putAddressable`。
2. **NIP-01 決勝規則**（新增 `shouldReplace`）：較新的 `created_at` 勝；**相同時保留 id 字典序較小者**。
   這確保所有中繼站對同一組事件**收斂到同一顆**（過去 `created_at` 相同時是「後到的贏」，各站可能分歧）。
   記憶體版與 SQL 版同步套用。
3. **`NODE_ATTEST_KIND` 10038 → 10039**，解除與 `ORG_ROSTER_KIND` 的撞號。
   此刻**尚無第三方節點**（生產 `relays.json` 為空），**無資料遷移問題**。
4. **可取代事件維持公開扇出**：ADR-0071 對**可尋址**（快照＝私人密文）有「只回作者本人」的閘門，
   **不可**套用到可取代事件——relay 清單/組織名冊本來就該人人收得到。已加測試釘住這個區別。

## 理由

- 選項 B 讓改動落在**路由與決勝規則**，而非新增一整套儲存路徑：記憶體版與 SQL 版都不必新增資料表，
  既有的配額、TTL、purge、query 全部原地複用。改動小、風險低、兩個實作自動一致。
- kind 改號必須與可取代語意**同一次**做完——先做語意、後改號會有一段時間名冊真的被覆蓋。

## 後果

- 正面：relay 清單/組織名冊/節點自報**不再累積**；客戶端每次連線只收到**一份**最新清單。
  各中繼站在 `created_at` 相同時收斂到同一顆。撞號的正確性地雷拆除。
- 負面 / 已知殘餘風險：
  - 可取代事件現在吃 `ADDRESSABLE_*` 的配額（256KB/顆、每作者每 kind 5 個位址、30 天 TTL）。
    對清單/名冊都綽綽有餘；但**組織名冊若成長到 256KB** 會被拒——目前無此規模，記於此備查。
  - `content === ""` 在此路徑代表 **purge**（ADR-0071 的語意）。可取代事件不應發空內容
    （簽章清單/名冊都有 JSON 內容），故無實際衝突，但這個共用語意值得記住。
  - 生產中繼站上**既有的累積事件不會被回溯清除**——它們會隨 7 天 TTL 自然消失。
- 測試：relay 新增 6 項——cron 連發 24 次只留 1 顆（正是修的東西）、舊事件不覆蓋新的、
  NIP-01 決勝（送入順序無關、收斂到同一顆）、不同作者各留一顆、REQ 只回一份、
  可取代事件公開扇出（不被快照的作者閘門誤擋）。全 771 測試通過。
