# 0046. 企業成員判定與對外通訊邊界（威脅模型）

- 狀態：已接受
- 日期：2026-07-05
- 相關文件：docs/adr/0044（封閉 allowlist 中繼）、docs/adr/0045（多身分）；PRD §6 威脅模型、§13 企業模式

## 背景與問題

企業模式（ADR-0044/0045）落地後，需明確回答兩個治理問題並記錄其邊界：

1. **如何判定一個使用者是「內部成員」？**
2. **內部成員是否仍保有以「非企業身分」對外通訊的能力？企業能否禁止？**

這些是威脅模型／政策層決策，先前於討論中確立但尚未成文。

## 決策

### 成員判定＝發布 allowlist

- **內部成員的唯一判準：** 其 hex pubkey 是否在企業中繼的 `allowedAuthors` allowlist 內。中繼只認得「是否為名單成員」，**無中央目錄、無帳號**。
- **佈建是企業身分與 npub 的唯一綁定點：** 「公司身分（HR／AD／SSO）→ npub」發生在管理者把 pubkey 加入名單這一步。系統本身不驗證「他是誰」，只驗證「這把 key 是否在名單」。
- **離職＝移出名單**，即時生效（下次不放行）。

### 對外通訊邊界＝企業控制「身分＋節點」，不控制「人」

- **使用者永遠保有以個人身分對外的能力。** 金鑰自主、軟體 AGPL 開源；一個人可持多把金鑰，企業 allowlist 只約束「工作身分」在「公司節點」。
- **企業可控制：** 自家 App 建置鎖定（工作身分只連公司座、不漫遊，見 ADR-0045）、把工作 npub 移出名單。
- **企業不可控制：** 員工另用個人金鑰＋一般版 App／任何 Nostr 客戶端連公開 relay 對外。
- **真正禁止對外只能靠系統外的裝置管理（MDM／端點）。** 明確界定此為裝置治理、不在本系統範圍。

## 理由

- **與自主金鑰身分模型一致：** 身分即金鑰對，天生可多重且不可連結；把「成員」定義為「名單成員」而非「帳號」，最貼合協定現實、且不引入中央目錄。
- **邊界誠實：** 明講「控制身分＋節點、不控制人」可避免對企業做出系統做不到的承諾（如「完全禁止員工對外」）。
- **讀取層取捨承接 ADR-0044：** allowlist 只擋發布，讀取層靠私網部署；本 ADR 不重複，只補「成員與人的區別」這一治理面。

## 後果

- 正面：企業對「工作身分」有完整生命週期控制（佈建／撤銷）；使用者保有個人自主；邊界對客戶透明。
- 負面 / 已知限制：
  - 「員工用個人身分對外」無法在本系統阻止（需 MDM）——須於企業導入文件明確告知。
  - 佈建流程（企業身分 → npub 對應、名冊分發）為企業責任，本系統只提供 allowlist 機制；佈建工具為後續（見 PRD §13.6）。
  - 金鑰還原（換機／遺失）與「離職即停用」的還原面，仍受制於 E2E 的金鑰託管取捨（若需還原須另立 ADR）。
- 後續行動：成員名冊佈建工具；企業導入文件載明對外通訊邊界與 MDM 建議。
