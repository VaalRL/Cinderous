# 0151. 標題列 ⚙ 上移外框＋按鈕滑鼠拖曳排位＋平時隱藏

- 狀態：已接受（已實作）
- 日期：2026-07-16
- 相關文件：**ADR-0150（自繪視窗外框——本 ADR 升級其設定模型）**、0142（設定入口位置）、
  0079（provider 模式）

## 背景與問題

ADR-0150 的自繪外框上線後，使用者要求三件事：

1. **⚙ 設定鈕上移到外框層**——與 ─ □ ✕ 同一條標題列（原本在三欄 nav bar／聯絡人視窗內）。
2. **按鈕排位改滑鼠拖曳**——取代 0150 的左右 radio＋←→ 箭頭。
3. **按鈕可設「平時隱藏」**——滑鼠碰到標題列才顯示。

0150 的 `{side, order}` 單帶模型裝不下「⚙ 在左、視窗控制在右」這種常見配置——拖曳排位
需要**每顆按鈕**都能落在任一側的任意位置。

## 決策

### 1. 設定模型 v2：左右雙帶（`nb.titlebarControls`，同鍵無痛遷移）

`TitlebarControls = { left: ControlId[], right: ControlId[], autoHide: boolean }`，
`ControlId` 增加 `"settings"`。預設 `⚙｜靠左、─ □ ✕｜靠右`（Telegram 風＋Windows 慣例）。
`parseTitlebarControls`：v1（`{side, order}`）自動遷移——order 落在原側、⚙ 補到**對側**；
v2 正規化——未知剔除、跨帶去重（左優先）、缺漏補回各自預設帶；`autoHide` 只認 `true`。
拖放核心 `placeControl(c, id, side, beforeId)`——放到某顆之前或帶尾，純函式可測。

### 2. ⚙ 上移外框：context 註冊開啟器

標題列在 App 樹之外（`WindowChrome`，ADR-0150），設定面板的開關在 App 內——以
`TitlebarProvider` 當管道：App 掛載時 `useRegisterSettingsOpener()` 註冊「開啟設定」，
標題列**收到開啟器才畫 ⚙**（沒註冊就略過該控制項，不畫死鈕）。Tauri 下三欄 nav bar 的
⚙ 不再重複畫（classic 佈局聯絡人視窗內的入口保留——那是視窗隱喻的一部分，且瀏覽器版
沒有標題列，仍需 app 內入口）。

### 3. 拖曳排位：設定頁「編輯器即預覽」

設定 → 外觀 → 視窗外框改為一條**假標題列編輯器**（HTML5 drag & drop）：四顆 piece
（⚙ ─ □ ✕）可拖進左右兩帶——拖到某顆上＝插到它前面、拖到帶空白處＝放帶尾；拖曳中帶
顯示虛線目標框。編輯器與真外框共用同一個 Provider 狀態，**拖完立即生效**。
刻意**不**在真標題列上拖（誤拖 ✕ 的風險 > 便利；業界慣例也是在設定裡排）。

### 4. 平時隱藏：`autoHide`

勾選後按鈕帶掛 `--autohide`（opacity 0），`.titlebar:hover / :focus-within` 才顯示——
標題列本體仍在（拖曳/雙擊照常），只是按鈕不佔視覺。設定頁編輯器不受影響（永遠可見）。

## 後果

- 正面：
  - ⚙ 與視窗控制同層，四顆按鈕位置/順序全由滑鼠拖曳自訂；可選極簡「無按鈕」外觀（hover 才現身）。
  - v1 設定無痛遷移（同 localStorage 鍵、parse 層轉換），防呆不變——壞設定永遠退回有效預設。
  - 測試 354（desktop）全綠；controls 純函式（遷移/正規化/placeControl）與 SSR 渲染皆有覆蓋。
- 已知限制／取捨：
  - HTML5 拖放無鍵盤替代（0150 的 ←→ 箭頭已移除）——無障礙排位留待需求出現。
  - autoHide 是 hover 觸發：觸控裝置上等同一直隱藏（桌面版無此情境，僅記錄）。
  - ⚙ 在瀏覽器版仍走 app 內入口（無自繪標題列可放）。
