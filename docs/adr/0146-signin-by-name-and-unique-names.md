# 0146. 登入以顯示名稱解析既有身分＋本機名稱唯一

- 狀態：已接受（已實作）
- 日期：2026-07-15
- 相關文件：**ADR-0140（修復切換身分→掉登入頁→建重複身分）**、0045/0138（多身分登錄）、
  0067（隱藏身分／本地密碼旗標）、0112/0122（nsec 不明文落盤、瀏覽器本地密碼必填）、
  0144（更改顯示名稱）、0061（加密 profile 廣播名稱）

## 背景與問題

登入畫面過去**每次都產生新金鑰＝新身分**（`generateSecretKey()`），顯示名稱只是標籤、從不參與
身分解析。於是「在登入畫面打一個既有身分的名字」會**又建一個同名的新身分**（不同 pubkey）。這正是
ADR-0140 那個坑的來源：切換到舊身分若金鑰取不到會掉回登入頁，使用者以為「打同名字就能回去」，結果
建出重複同名身分。ADR-0140 從「金鑰取不到」那側止血（改走 nsec 救援、命名空間隔離），但**登入畫面
本身**仍然：打既有名字＝建重複。

使用者要的是符合直覺的登入：**打名字時，若本機已有這個身分就登入它；沒有才建新；若該身分有本地密碼
則驗密碼後才進**。

### 一個必須釐清的邊界：名稱不是加密金鑰

`Profile` 主鍵是 **pubkey**，`name` 是純標籤（可重複、可改，ADR-0144）。nsec 在瀏覽器是 Argon2id
密文、在原生是 OS 金鑰庫密文——**密碼解開前無法取得明文金鑰**。所以「以名稱找金鑰」實際上是：
**用名稱在明文登錄（`nb.profiles`）選出對應 profile（含 pubkey）→ 再用密碼解那個 pubkey 的 nsec
密文**。名稱只是「選哪個身分」的查找鍵，等價於在切換器點選；**密碼仍是解密的唯一途徑，安全模型不變**。

## 決策

### 1. 純函式 `resolveSignIn` / `nameTaken`（engine，兩端共用）

於 `packages/engine/src/storage/profiles.ts`：

- `resolveSignIn(state, name)` → `enter`（恰一個**可見**同名 → 進既有）／`ambiguous`（多個同名，僅可能
  來自本規則之前的舊資料 → 擋下不靜默進入）／`create`（無命中 → 建新）。
- `nameTaken(state, name, exceptPubkey?)` → 本機是否已有**可見**同名身分（新增/改名時擋重名）。

兩者都**排除 hidden**：隱藏身分（ADR-0067）永不被名稱命中、也不因「名稱被佔用」而洩漏存在——因此
可有一個可見與一個隱藏同名並存，登入只命中可見那個。比對走 `trim()`、大小寫敏感。

### 2. 桌面登入：命中既有＝`setActive`＋`reload`（Fix First）

桌面 `signIn` 開頭以 `resolveSignIn` 分流：`enter` → `saveProfiles(setActive(...))` + `location.reload()`
——**重用開機既有的解鎖/救援路徑**（鎖定身分出解鎖畫面驗密碼、無鎖 Tauri 自金鑰庫載入、金鑰取不到走
ADR-0140 的 `needNsec` 救援），不另寫一條解鎖邏輯、不新增安全面。`ambiguous` → 自訂對話框提示改用
「用 nsec 登入」。`SignIn` 元件加 `lookupName` prop：輸入命中既有名稱時顯示「將登入既有身分」、按鈕改
「登入既有身分」、**收起建新用的中繼站/密碼欄**（既有身分自帶中繼站；密碼於重載後的解鎖畫面驗證）。

### 3. 本機名稱唯一（兩端）：新增／改名擋重名

- **桌面**：`AddIdentityModal` 收 `nameTaken` predicate，命中即停用建立鈕並顯示提示；`renameSelf` 改名前
  以 `nameTaken(state, name, ownPubkey)` 擋，`SettingsPanel` 顯示重名錯誤。
- **行動**：行動端登入是 **nsec 導向**（pubkey 由金鑰決定，不由名字），故「以名稱找金鑰登入」不適用；
  套用的是名稱唯一——`NsecSignInScreen` 收 `nameTaken(name, pubkey)`（排除同一把金鑰的重複匯入），
  新增身分撞名即擋；`renameSelf` 同樣以 `nameTaken` 擋，`SettingsScreen` 顯示重名錯誤。

`renameSelf`（兩端）改回傳 `boolean`：`false`＝撞本機同名，UI 顯示 `settings_nameTaken` 且不視為成功。

## 後果

- 正面：
  - 登入畫面打既有名字＝**登入那個身分**（有鎖則於解鎖畫面驗密碼），不再建重複同名身分——把 ADR-0140
    的坑從登入畫面這一側也補上。
  - 名稱在本機唯一 → `resolveSignIn` 命中無歧義；切換器/登入不再有兩個難以分辨的同名身分。
  - 桌面登入解析**完全重用**既有 `setActive`+reload 與解鎖/救援路徑（Fix First），零新增解密邏輯、
    安全模型不變（密碼仍是解 nsec 的唯一途徑）。
  - 測試 +10（engine 純函式：resolveSignIn 4／nameTaken 3；desktop SignIn SSR：enter/ambiguous/預設建新 3）。
    全綠 engine 241／desktop 329／mobile 143／i18n 8。

- 已知限制：
  - **行動端不做「以名稱找金鑰登入」**：其登入是 nsec/配對導向，身分由金鑰決定；行動端回到既有身分走
    切換器解鎖。名稱唯一與改名擋重名兩端一致。
  - `ambiguous`（多個可見同名）只可能來自本規則之前的舊資料；登入畫面此時無切換器（尚未登入），故導向
    「用 nsec 登入」或稍後在設定改名，不自動挑選。
  - 名稱比對大小寫敏感、僅 `trim()`；「Work/work」視為不同名（可接受，多為中文顯示名）。
  - 桌面命中既有身分採「切換＋重載」兩畫面（名字畫面→解鎖畫面），非單畫面直接驗密碼——換取零重複解鎖
    邏輯與完整重用救援路徑；單畫面 inline 驗證留作後續 UX。
  - 匯入既有身分時若本機另有同名（不同金鑰）會被擋，需先改名；同一把金鑰的重複匯入不受影響（排除自己）。
