# 0147. 自架網頁版部署——app 與官網分屬不同子網域（origin）

- 狀態：已接受
- 日期：2026-07-16
- 相關文件：**ADR-0090（官網與 E2E 通訊平面硬隔離）**、0112/0117/0122（web 本地密碼包裹 nsec、
  瀏覽器本地密碼必填）、0053/0054（桌面 OS 金鑰庫與加密 blob，對照組）、0069（自動選座/relay 錨點）、
  0075（自架 relay）；`docs/self-hosting-web-app.md`（新增）

## 背景與問題

`apps/desktop` 的 `vite build` 產出的**本就是一份純靜態 web app**（`index.html` → `main.tsx` → 完整
`App`）；Tauri 桌面版只是把同一份 web 建置包進原生殼。當這份建置以純網頁靜態託管時，`isTauri()` 為
`false` → 走**瀏覽器模式**：金鑰以 Argon2id 本地密碼包裹存 localStorage（ADR-0112）、本地密碼**必填**
（ADR-0122）、封存走 OPFS，連線一樣走 `wss://` 到 relay。

自架者想要的部署是：**官網一個入口是登入 → 接瀏覽器版 → 以瀏覽器為介面運行、金鑰身分存本機**。這與
ADR-0090 為**官方站**立下的「硬隔離鐵則」（獨立網域、永不接觸金鑰）在同一條線上，但需要一份針對
**自架者**的部署拓撲與信任模型的明確決策——因為對 client-side E2E app 而言，**送出 JS 的伺服器等於
握有金鑰**，部署拓撲即是安全邊界。

## 決策

### 1. app 與官網分屬不同 origin（子網域）——沿用 ADR-0090 隔離精神

自架部署**必須**把「產品（瀏覽器版 app）」與「行銷/官網內容」放在**不同 origin**（建議不同子網域，
如 `app.example.com` vs `www.example.com`）。理由同 ADR-0090：行銷內容的攻擊面（CDN、供應鏈、第三方
腳本）**不得**與握有金鑰的 app origin 共用——否則行銷站被入侵即可替換 app JS 竊取記憶體中的 nsec。
官網入口的「登入」只是**一條跨 origin 連結**指向 app 子網域，不在官網 origin 內嵌 app。

### 2. 產品即 `apps/desktop` 的 web build；金鑰本機、體驗與桌面同源

自架的「瀏覽器版」就是 `pnpm --filter @cinder/desktop build` 的靜態產物（`index.html` 為產品入口）。
與桌面版共用**同一份** React＋`@cinder/engine`，核心通訊體驗一致；差異限於 `isTauri()` 分流的原生能力
（本地密碼必填、金鑰在瀏覽器加密儲存而非 OS 金鑰庫、原生檔案拖放/塊檔封存改用瀏覽器對應）。

### 3. 交付信任模型是部署層責任

自架者承擔「每次載入送出誠實 JS」的信任，故部署**必須**：**全程 HTTPS**、對 app origin 上**嚴格 CSP**
（`connect-src` 僅允許 relay 的 `wss://`、`default-src 'self'`、禁止第三方腳本）、與官網分 origin 縮小
爆炸半徑。建議搭配不可變（immutable）發佈與資源指紋。**relay 只轉發密文**，不在此信任邊界內。

### 4. 指向自架 relay 的三條路（不改共用程式碼即可自架）

預設站來自 `@cinder/engine` 的 `ANCHOR_RELAYS`（編譯期）。自架者可：(a) 官網登入連結帶
`?relay=wss://你的relay`（`initialRelayUrl` 已支援、優先於本地記憶）；(b) 建置前改 `ANCHOR_RELAYS`；
(c) 讓使用者以「使用其他中繼站」手動輸入。三者皆不需分叉核心邏輯。

### 5. 官方站（`apps/website`）維持 ADR-0090 不變

本 ADR 只規範**自架者**的部署拓撲；官方 `apps/website` 仍是與通訊平面硬隔離、零追蹤的純靜態站，
**不**在其中內嵌登入/ app。兩者不相牴觸：ADR-0090 管官方營運工具，本 ADR 管自架網頁交付。

## 後果

- 正面：
  - 明確授權並規範「自架網頁版 + 分子網域官網」的部署，沿用既有瀏覽器模式（ADR-0112/0122），
    零核心程式碼改動；附 `docs/self-hosting-web-app.md` 逐步指南（build → 子網域託管 → CSP → 指定 relay）。
  - 以「不同 origin」把行銷內容攻擊面擋在 app 金鑰邊界外，延續 ADR-0090 的隔離價值到自架情境。
- 已知限制／取捨：
  - **web 交付的固有信任問題**：使用者每次載入都信任 app origin 送出誠實 JS——這是所有網頁交付 E2E
    app 的共性，非 Cinder 特有。要更強保證請用桌面原生版（簽章二進位 ＋ OS 金鑰庫，ADR-0053）。
  - **持久性**：瀏覽器「清除網站資料」或儲存驅逐會清掉本機（加密）身分 → 必須引導使用者**備份 nsec/
    備份碼**方能救回。
  - CSP/SRI/HTTPS 屬部署層責任，程式碼不強制；指南提供建議設定，實際由自架者落實。
  - 本 ADR 不改動任何模組邊界或事件契約，屬部署拓撲決策。
