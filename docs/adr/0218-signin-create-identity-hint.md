# 0218. 登入畫面：新名稱明示「將建立新身分」

- 狀態：已接受
- 日期：2026-07-20
- 相關文件：ADR-0122（瀏覽器本地密碼必填）、ADR-0146（登入畫面以顯示名稱解析身分：create/enter/ambiguous）、`apps/desktop/src/ui/SignIn.tsx`

## 背景與問題

登入畫面（`SignIn`）以顯示名稱統一「建立＋登入」（ADR-0146）：打**既有**名稱＝登入該身分、打**沒用過**的名稱＝建立新身分。`lookupName` 已能分辨兩者，但呈現**不對稱**：

- 命中既有 → 有動態提示（`signin-enter-existing`）＋按鈕改「登入既有身分」。
- 名稱是新的（create）→ **無建新專屬提示**；按鈕仍寫「登入」（`signIn_button`），其實是在建立。

結果：使用者可能「以為在登入、其實在建了一個新帳號」——尤其首次使用時。

## 決策

補上與「登入既有」對稱的 create 訊號：

- **動態提示**：名稱是新的且**已輸入**時，顯示 `signin-create-new`：「這個名稱是新的，將建立新身分（本機生成金鑰）。」（空白名稱不顯示，避免還沒打字就跳。）
- **按鈕文字**：create 分支由「登入」改為「**建立新身分**」（`signIn_createButton`），與「登入既有身分」對比。

純顯示層變更；不動 `submit`/`lookupName` 邏輯與建新流程。

## 理由

- **消除誤解**：把「建立 vs 登入」兩種結果都給明確、動態的訊號，符合最小驚訝原則。
- **對稱、低風險**：與既有 `signin-enter-existing` 同一處、同機制；不改資料流。

## 後果

- 正面：首次/建新時清楚知道「這會建立新身分」；按鈕語意與實際行為一致。
- 負面 / 已知殘餘風險：空白名稱時按鈕即顯示「建立新身分」（尚未打字）——語意正確（將建立），可接受。`signIn_button`（「登入」）此處不再使用，保留於 i18n 供未來復用。
- 後續行動 / 待辦：無。
