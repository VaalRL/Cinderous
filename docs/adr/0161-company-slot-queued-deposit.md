# 0161. 公司儲存槽——員工選擇性存放、離線排隊、企業主上線背景 P2P 傳輸

- 狀態：已接受（已實作）
- 日期：2026-07-16
- 相關文件：ADR-0093/0102/0103（檔案 P2P 與 savedPath）、0047（名冊）、0128（檔名消毒
  與讀檔白名單）、0155（企業主身分）

## 背景與問題

企業要「對話中的檔案能存到企業主電腦」。全自動歸檔＝監控性質（倫理負擔重）、
走 relay＝儲存成本與配額工程（另立 ADR-0162 處理小檔案）。使用者裁示的形態：
**員工主動選擇**哪些檔案存入公司儲存槽 → 本地排隊 → 企業主上線後**背景 P2P 傳輸**。
企業主端**不顯示通知**，靜默落盤。

## 決策

1. **語意：存放（deposit），不是歸檔**。員工對自己送出或收到的檔案（有 `savedPath`
   者）按「存入公司儲存槽」——這是自主動作，非政策強制，無須揭露機制。
2. **員工端佇列**：`nb.slotQueue.<pubkey>`（localStorage）存「路徑＋metadata＋來源
   對話」，**不存位元組**（傳輸時以 ADR-0103 的 `savedPath` 重讀；檔案被搬走＝失敗，
   可重試）。管理者在聯絡人中上線（心跳現成）→ 逐一背景傳輸。Tauri 桌面限定
   （瀏覽器拿不到路徑）。
3. **傳輸＝現有 P2P ＋ `slot` 標記**：`wrapFileMessage` 的 rumor 加 `["slot", 來源]`
   標籤。兩端據此**不建立聊天訊息**（不污染 1:1 對話）；企業主端位元組到齊 →
   `onSlotDeposit` 交 App 落盤。metadata 不發自封副本（自己的其他裝置無需知道）。
4. **企業主端落盤（無通知）**：只收**名冊成員**的存放；寫入
   `<槽目錄>/<員工顯示名>/<YYYY-MM-DD>-<原檔名>`（重名自動加尾碼）＋根目錄
   `index.jsonl` 一行索引（誰/何時/來源/原檔名/相對路徑）。槽目錄可於設定選擇
   （原生資料夾對話框）；未設＝`<appData>/CinderSlot` 預設槽。寫入走新的 Rust 指令
   （基底目錄之外一律拒寫；檔名/員工名逐段消毒——ADR-0128 的路徑穿越防禦延伸）。
5. **狀態可見**：員工端設定內小佇列面板（排隊中/傳輸中/已存放/失敗＋重試/移除）。

## 理由

- 員工自選＝隱私負擔消失；本地排隊＝企業主離線問題以零協定成本解掉；重用 P2P 管線
  ＝密文不落中繼、實作面最小（Fix First）。

## 後果

- 正面：組織檔案集中到企業主電腦、員工完全自主、不動任何協定與中繼站。
- 已知限制：Tauri 桌面限定（兩端）；排隊期間原檔被搬走＝該筆失敗（可重新指定後再試）；
  傳輸需雙方同時在線（排隊已把「同時」變成「企業主上線的任何時刻」）；企業主多台
  裝置只有收到那台落盤。
- 測試：core（slot 標籤 round-trip）＋engine（存放不建訊息、企業主收 onSlotDeposit、
  非名冊成員拒收）＋desktop（佇列純函式、UI 分流）。
