# 0042. 自製貼圖容量與規格限制

- 狀態：已接受
- 日期：2026-07-04
- 相關文件：docs/adr/0032（自製貼圖：內容隨訊息）、docs/adr/0027（群組成對扇出）、docs/adr/0041（外送匣）；apps/desktop UI

## 背景與問題

自製貼圖（ADR-0032）採「內容隨訊息」——SVG 直接內嵌在訊息（`nb-sticker:v2:{label,svg}`）。在群組中，這份內容會**對每位成員各扇出一份**（ADR-0027），因此貼圖的容量直接乘上群組人數放大頻寬，也影響 relay 接受度。需檢視「自製貼圖是否應有容量與規格限制」。

## 現況盤點（已存在的限制）

- **SVG 位元組上限 32 KiB**（`STICKER_SVG_MAX_BYTES`），安全落在 NIP-44 明文上限 65535 內；於匯入 / fork / 收端渲染 / 點擊收藏皆驗。
- **禁用特性拒收制**：`<script>`、事件處理器、`javascript:`、`<foreignObject>`、`<iframe>/<embed>/<object>`、外部參照、非 image 的 `data:`。
- **本地貼圖庫上限 32 張**（`LIBRARY_MAX`），以 `contentHash(svg)` 去重。

## 缺口與決策

盤點發現一個真實缺口與若干取捨：

1. **標籤（label）無長度限制（本 ADR 修正）**：`label` 與 SVG 一同隨訊息扇出，但先前完全未受檢，等於可繞過 SVG 上限膨脹每份負載，收端渲染亦可能被超長 alt/title 影響。
   - **決策**：新增 `STICKER_LABEL_MAX = 48` 字。建立端（`addSticker`）夾住；**收端（`parseCustomSticker`）亦防禦性夾住**，不信任對端手工訊息。
2. **維持 32 KiB SVG 上限**：對 1:1 合適；群組扇出雖 N 倍放大，但由 ADR-0041 外送匣節流投遞、且大貼圖的根本解（內容定址去重、只送一次 blob）屬更大的協定變更，列為後續，不在此收緊硬上限以免傷及正常使用。
3. **暫不新增的限制（記錄取捨）**：
   - 元素數量 / path 複雜度上限（防渲染 CPU）——`<img>` 沙箱已擋 JS，複雜度屬效能而非安全，優先度低。
   - 動畫（ADR-0031 CSS keyframes）限制——自製貼圖沿用內建動畫能力，暫不特別限制。

## 理由

- **對症且最小**：標籤上限關掉唯一未受控的膨脹來源，建立端＋收端雙夾，純函式可測；不動加密與扇出路徑。
- **一致的容量心智模型**：貼圖的線上負載 = `svg (≤32KiB) + label (≤48 字) + JSON 包裝`，全部有界。
- **群組放大另案處理**：真正省頻寬要靠去重扇出（送一次 SVG blob + N 份引用），與訊息格式相關，範圍大、風險高，不與本限制混做。

## 後果

- 正面：關閉標籤膨脹缺口；容量規格明確、可測；渲染更耐惡意輸入。既有貼圖測試全綠，新增 2 項標籤上限測試。
- 負面 / 後續：
  - 群組送大貼圖仍是 O(N) 頻寬；根治需「內容定址去重扇出」——另立 ADR。
  - 未加元素/動畫複雜度上限；若日後出現渲染效能濫用再議。
  - 可考慮於「送出大貼圖到大群」時給 UI 提示（人數 × 大小估算），屬體驗層後續。
