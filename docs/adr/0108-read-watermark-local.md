# 0108. 已讀水位的本機持久化——並修正訊息時間戳（改用 `rumor.created_at`）

- 狀態：已接受（已實作）
- 日期：2026-07-14
- 相關文件：ADR-0009（多設備收斂：可變狀態採 LWW）、0058（送達/已讀回條）、
  0088（在線元數據最小化）、0095（群組回條水位）、0107（自封副本、rumor.id）

## 背景與問題

ADR-0107 收尾時盤點多設備缺口，發現「已讀狀態不同步」底下還有一層更基本的問題：

> **已讀狀態連本機都沒存。**

`unread` 只是 React 的 `useState<Record<string, number>>`——一個記憶體計數器：收訊時 +1、
開對話時歸零。`StoredMessage` 沒有任何已讀欄位。

**後果：未讀數在同一台機器上、重新載入就歸零。** 關掉 App 再打開，所有紅點消失，
而訊息其實沒讀過。談「跨裝置同步已讀」之前，得先讓它在**一台**機器上活下來。

### 但要做水位，先撞上一個真的 bug：訊息時間戳是「下載時間」

水位需要一個**排序鍵**。現有的 `StoredMessage.at` 卻是：

```ts
const message = { id: rumor.id, ..., at: Date.now(), ... };  // 接收端「處理到」的時間
```

**`Date.now()` 是接收端的下載時間，不是訊息的送出時間。** 這帶來兩個真實破損：

1. **離線補收後，所有訊息被蓋上「現在」。** 離線一天再上線，中繼把一整天的留言一次回放，
   每一則都拿到 `at ≈ 現在` → 時間顯示全錯，且彼此的先後被壓平。
2. **順序是隨機的。** NIP-59 的 `jitteredPast()` 把**外層** wrap 的 `created_at` 隨機往前推
   （隱私設計），所以中繼回放的順序本來就不是送出順序。`loadMessages()` 不排序、UI 也不排序
   ——顯示順序＝**到達順序**＝隨機。
3. ADR-0107 之後更糟：**兩台裝置的 `at` 必然不同**（各自的下載時間），連帶順序也可能不同
   ——同一個對話在手機和電腦上長得不一樣。

而水位若以 `at` 比較，這個鍵在兩台裝置間**根本不可比**——跨裝置同步已讀從一開始就不可能。

**幸好正確的時間一直都在**：`jitteredPast()` 只抖動 **seal 與 wrap** 的時戳；
**`rumor.created_at` 是發送者宣告的真實送出時間**，且 `openWrap()` 會核對 rumor 雜湊
——**中繼改不了它**。

## 考量的選項

- **A. 水位以「本機遞增序號」為鍵**：對本機是精確的（晚到的舊訊息必得較大序號，不會被吞），
  但序號是**裝置本地**的，永遠無法跨裝置比較 → 之後要同步已讀時得再換一套模型。**不採**。
- **B. 逐訊息 `read` 旗標**：最精確，但每則多一個欄位，且與 ADR-0058 已在用的**水位**語意
  分裂成兩套模型；跨裝置同步時也比「一個數字」吵得多。**不採**。
- **C（採用）. 以送出時間（`rumor.created_at`）為鍵的每對話水位**：一個數字、單調遞增；
  與 ADR-0058 的已讀**回條**共用同一套水位語意（不引入第二種模型）；且日後跨裝置同步時，
  ADR-0009 的 **LWW（比 `updatedAt`）** 直接適用——單一數字天生收斂。
  代價見「後果」的已知缺陷。

## 決策

### 1. 訊息時間 `at` 改用 `rumor.created_at`，不用 `Date.now()`

收訊端與送訊端一律以**發送者宣告的送出時間**為訊息時間：

```ts
function msgTime(rumor: Rumor): number {
  return Math.min(rumor.created_at * 1000, Date.now()); // 箝制未來時戳
}
```

- **箝制未來時戳**：壞掉或惡意的時鐘不得把訊息永遠釘在對話頂端。
  代價：只有在「送出時間在未來」這種異常情況下，兩台裝置算出的 `at` 才會不同——可接受。
- **送出端也用同一個值**：`sendMessage` 先取 `now = nowSec()`，傳給 `wrapMessage({ now })`，
  並以 `now * 1000` 作為本機 `at`。否則發送裝置存 `Date.now()`（毫秒）、其他裝置存
  `created_at * 1000`（秒截斷），同一則訊息會差最多 999ms —— 水位比較就不精確了。

### 2. 已讀水位持久化：`AppStorage.setReadAt(convo, at)` / `loadReadAt()`

- 水位＝「這個對話中，我已讀到的最新訊息時間」。**單調遞增**（不可倒退）。
- **未讀數 = 該對話中 `!outgoing && at > readAt` 的訊息數**——由儲存**推導**，不是計數器。
- 1:1 與群組共用（鍵為 `convo`：對方 pubkey 或 groupId）。

### 3. **本機已讀水位** 與 **已讀回條** 解耦

這是本 ADR 最重要的一條。現行 `markRead()` 第一行就是：

```ts
markRead(convo) { if (!this.readReceipts) return; ... }
```

已讀回條**預設關閉**（ADR-0058 的互惠設計）。若把本機水位掛在 `markRead()` 上，
**大多數使用者的未讀狀態永遠不會被保存**。

因為這是**兩件不同的事**：

| | 是什麼 | 誰決定 |
|---|---|---|
| **已讀回條** | 告訴**對方**我讀了 | **隱私選擇**（opt-in，預設關） |
| **已讀水位** | 記得**我自己**讀到哪 | **UX**（應永遠有效） |

故拆成兩個方法：

- `clearUnread(convo)`：推進本機水位。**一律持久化**，與回條設定無關。
- `markRead(convo)`：`clearUnread(convo)` **＋** 送已讀回條（僅在 `readReceipts` 開啟時）。

桌面 App 其實**早就**把這兩件事分開了——視窗隱藏時它清徽章卻不送回條
（`if (!document.hidden) markRead(pk)`）。本決策只是把這個既有分野落實到引擎與儲存層。

### 4. 未讀數由後端推導、以 `onUnread(counts)` 推給 UI

UI 不再自行 +1 / 歸零。後端在**開機載入歷史後**、**收到新訊息時**、**水位推進時**重算並發出。
於是重新載入後未讀仍在——這正是本 ADR 的目的。

## 理由

- 先修時間戳，才有可用的排序鍵；而且它本身就是個真 bug（離線補收後時間全錯、順序隨機）。
- 水位語意與 ADR-0058 的已讀回條一致，不引入第二套模型。
- 單一數字的水位，讓日後的跨裝置同步直接套用 ADR-0009 的 LWW——**這一步已經為 D4b 鋪好路**，
  但**本 ADR 不送任何東西上中繼**（見「不做什麼」）。

## 不做什麼（明確劃線）

- **不跨裝置同步已讀。** 那需要把每次「開啟對話」變成一則中繼事件，是**最吵、最洩漏活動樣態**
  的一項，與 ADR-0088（在線元數據最小化）方向相衝。要做應另立 ADR，並處理批次/節流。
- **不改已讀回條的預設**（仍為 opt-in、關閉）。

## 後果

- 正面：
  - 未讀徽章**重新載入後仍在**（今天會歸零）。
  - 訊息時間與順序**正確且跨裝置一致**——離線補收不再把整批訊息壓成「現在」。
  - 水位是單一數字 → 日後跨裝置同步可直接套 LWW。

- **已知缺陷（水位模型的固有代價，誠實記下）**：
  **「晚到的舊訊息」會被靜默視為已讀。** 若某則訊息的送出時間早於我的水位，卻在我讀過之後
  才抵達（例如多中繼路徑中某條特別慢），它的 `at` 小於水位 → **不計入未讀，不亮紅點**。
  訊息仍在對話裡（不會遺失），但可能被忽略。
  觸發條件很窄（須恰好在我讀取的當下、有一則更早送出的訊息仍在途中），
  但這是選項 C 換取「可跨裝置比較」所付的代價；選項 A（本機序號）不會有此問題卻永遠無法同步。
  若日後證實會咬到人，再以「本機序號輔助鍵」補強。

- 遷移：既有訊息的 `at` 仍是舊的下載時間；不回填（無從得知原始送出時間）。
  新訊息起採用正確時間。水位初始為 0 → 首次啟動時既有收訊全部視為未讀（保守，寧可多提醒）。
